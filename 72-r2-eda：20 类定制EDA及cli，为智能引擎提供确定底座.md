# r2-eda：20 类定制 EDA 及 CLI，为智能引擎提供确定底座

---

寄存器读回挂了：仿真里 `CTRL.ENABLE` 是 1，软件头文件写在 bit[0]，UVM RAL 写在 bit[1]。三份各说各话——地址冲突类 bug，迁移到单一配置源之前，每个季度都能摊上两三个。

Agent 接到失败，没有确定性工具约束。LLM 不碰规格源，直接打开三个产物分别「对齐」：改 RTL 位域、改 C header、改 RAL。本地 case 全绿，Formal 也没意见。问题是产品规格文档里 ENABLE 仍在 bit[0]；驱动按文档写，集成时三套地址语义对不上——季度级冲突又回来了，只是这次穿着「AI 修好了」的皮。

这不是 AI 不会写代码。是系统没有把「只能改一份真理源、其余由工具生成」立住。**AI 的能力越强，修改半径越大，触碰的边界越多——没有工具底座，能力本身就是风险。**

## 工具不是给人用的了，是给 AI 用的

你大概率已经买了 AI，团队里也推了半年。模型更强了，单个工程师写 RTL 更快了。但有一个问题，很少有人愿意正面回答：**交付周期变短了吗？**

到现场看，画面是这样的：AI 跑完一步，停下来等人审批。人刚进入心流，被审批请求打断。一整个下午，AI 实际工作不到 10 分钟，其余时间在等人。

为什么？**因为 EDA 工具还是给人用的。**

人用仿真器：打开 GUI，配参数，点 Run，盯波形。人用 Verdi 追信号：点 10 多次，一层层展开。人对比通过和失败：肉眼扫，手动标差异。人判断嫌疑：脑里综合推理，排优先级。这些工具的交互方式——GUI、拖拽、点击、肉眼——全是给人设计的。AI 用不了。AI 要的是：一条命令，拿结构化结果，毫秒级决策下一步。

**工具不给人用了，给 AI 用。** 这就是 r2-eda。极大丰富 · 极度垂直 · 极限低成本。确定性 syscall，覆盖 Spec → RTL → 验证 → 调试。

一个芯片团队的 AI 落地 = Agent + Skills + Data + EDA。r2-eda 就是这公式里的 EDA。

## AI 用 EDA，跟人有三大根本区别

**第一，榫卯结构。** 人用工具是串行手动，一个一个调。AI 把智能组件和自动组件像榫卯一样拼成复杂但稳定的工作流，调用频率比人高 1000 倍。不是"AI 调用工具"——是同一个神经回路的呼吸。LLM 是中枢神经，CLI 是外周神经，两者回路内一体，不可分割。

**第二，极限并行化。** 人脑是 CPU——串行，一天有效专注 4 小时。AI 是 GPU——在预算范围内可以拉起成千上万个智能体并行。人用工具时 AI 在空转，这不是 AI 的问题，是工具没给 AI 准备好接口。

**第三，AI 现场给自己造工具。** 人用固定工具集，等供应商给。AI 从相似案例和成功失败路径中快速打磨出可用脚本。`cbbrtl` 就是实例——CBB 库匹配 + AI 补残差 + FEC 等价兜底。左脚踩右脚，快速提升自身能力。

## 20 个工具，五层分层

r2-eda 做两件事：**用 AI 造工具**，和**让 AI 用工具**。

**用 AI 造工具——设计自动化的定制。** Spec 变成数据，数据变成设计，设计变成验证环境。这条链上，凡是能确定的，整个移出概率世界。

寄存器生成最容易手抄出错——地址对不齐、位域写错、文档和 RTL 版本不同步。`reggen` 一份 regmap.json 进去，RTL + C Header + UVM RAL + SVA 四个产物同源出来。`c2rtl` 拿 C 算法模型，分钟级试架构、摸 PPA。`cbbrtl` 在流片验证过的 CBB 上，AI 只补有界残差，FEC 等价兜底才入库——不从零造轮。`topgen` 连 SoC 顶层，`buildsoc` 管 IP 供应链。`uvmgen` 从 yaml 生成完整 UVM 环境。

**让 AI 用工具——数据加工和上下文服务。** 工具跑完，结果回来了。如果是裸 log——10000 行 UVM 文本混在一起——AI 要从里面猜"这个测试过了没""哪里错了"。效率低，还猜错。

第二类工具，把 EDA 引擎的输出加工成 AI 能直接用的结构化上下文：`r2-scan` 3000 条失败日志 AI 30 秒，`r2-trace` 一条命令拿因果链，`r2-wave` 一条命令拿波形数据，`r2-diff` 自动定位差异，`r2-suspect` 自动排名候选。

---

五层分层全景：

| 层 | 工具 | 一句话 |
|---|---|---|
| Spec → SSoT | pdf2md, specx | PDF 变成结构化 JSON，12 类元信息 |
| SSoT → RTL | reggen, c2rtl, cbbrtl, topgen, buildsoc | 确定性生成，零幻觉，可编译 |
| SSoT → 验证 | uvmgen, tbgen, scb_gen, seq_gen, cov_gen, sva_gen | 从 SSoT 同源生成验证环境 |
| AI-Native 调试 | xtrace, xwave, xloc, verdi-cli | 结构化因果链，JSON API，AI 可消费 |
| 质量+进化 | lintx, formalx, autoresearch | 门禁 + 形式验证 + 经验自动蒸馏 |

每个工具的共同特征：输入是 SSoT（机器可读 JSON），不是自然语言；输出是可验证的工程产物，有质量门禁；AI 可消费的接口（JSON API，不是 GUI）；配置驱动生成——输入确定则输出确定，不需要概率判断。

## 精选五个工具，看它们怎么工作

**reggen：寄存器生成器。** regmap.json → RTL 寄存器模块 + C header + UVM RAL + SVA 断言 + 文档。商业工具（Synopsys RALGen）绑定 VIP 生态。自研 reggen 以 SSoT 为唯一真源，结构上不可能 RTL 和 UVM RAL 不一致。OpenTitan ralgen 只出 UVM RAL，reggen 是全栈：从一份 regmap.json 同时出 RTL、C Header、UVM RAL、SVA、文档。五个产物同源——改 regmap.json 自动重生全部五个。

**c2rtl：C 算法到 RTL 编译器。** C++/Python 算法 → SystemVerilog RTL + Verilator 验证闭环。LLM 先写 Python 金模，再写 RTL，逐拍比对。传统 HLS（Vivado HLS、Catapult）黑箱优化，结果不可解释。c2rtl 输出可读 RTL + Python 金模逐拍比对——AI 友好，人类可审。论文支撑：Samsung DVCon 2025 Python Golden Model、cocotb 2.0、pyuvm 系列。

**uvmgen：UVM 验证环境全栈生成器。** SSoT feature_points.json → 完整 UVM DV 环境（Agent/Monitor/Driver/Env/Test）。手写 UVM 环境不产生差异化价值。uvmgen 从 SSoT 同源生成——改 Spec 自动重生验证环境，结构上不可能 Spec 改了验证没改。论文支撑：DVCon "Generating UVM Testbenches"、"How Far Can You Take UVM Code Generation"、OpenTitan uvmdvgen。

**xtrace：RTL 信号因果链追溯。** 基于 NPI 查询设计数据库语法树，输出信号的 driver/load/control_dependency 结构化 JSON。AI 拿到的不是 grep 文本片段，是带因果标注的电路连接图。grep 没有因果方向。AI 用 grep 查信号时会把 driver 和 load 搞反、跨模块断链。xtrace 结构化因果链——AI debug 的信息维度从「平文本」升级到「因果图」。

**lintx：多级 Linter + 静态检查。** HJSON 三级 Linter（语法→Schema→语义）+ RTL 静态规则检查。五层质量门禁：未过 lintx 严禁写入 SSoT。商业 Lint 工具（SpyGlass、LEC）报告泛滥，噪音 90%。lintx 只报「过不了这一关就不能往前走」的硬错误——Agent 行为的确定性靠门禁，不靠提示词。

## 双引擎：左脚确定性，右脚创造力

工具产出分两条路径，构成双引擎：

**左引擎（脚本）**——确定性生成。Hjson → UVM 骨架，零幻觉，可编译。能确定的部分，整个移出概率世界。

**右引擎（AI）**——生成式注入。VPlan → 业务逻辑，补全算法和 Sequence，智能代码推理。需要创造的部分，由 AI 生成。

左引擎造榫——有确定形状的骨架。右引擎填卯——弹性注入业务逻辑。两者咬合成完整的产物。

## 工具是管道，不是裁判

r2-eda 有一个铁律：**工具不替代引擎，工具调用引擎。** 仿真器、形式验证器、lint 工具——这些引擎已有 20 年积累。r2-eda 不重造它们。r2-eda 做的是给已有引擎加一层 CLI/MCP 接口，让智能体能调用。引擎是采购的，接口是自己做的。

更深一层：**工具是管道，可换后端。** 规则不来自工具——先定 Vicky 如何判 PASS，再定调用哪些工具。工具只管执行，不管立法。换了后端引擎，管道不变，规则不变，智能体的工作流不变。你不会被任何一个 EDA 供应商锁死。

## 护城河不在快，在慢和脏

15 个工具，AI 用它们比人快 1000 倍。但快和干净，不是护城河。

"供应商提供一半能力——工具层可以采购。"一个智能体跑得快，换一个也能跑这么快。工具本身买得到。

**护城河在慢和脏。**

慢——不是代码跑得慢，是方向校准慢。每一次看到成品才发现想错了，每一次改了 Spec 再重做，每一次把新的理解重新注入系统。

脏——不是代码写得脏，是经验积累的过程脏。每一次踩坑，补一条规则；每一次失败，留一条决策路径。不断犯错，一点点攒下来。

张三修一个 bug 花 4 小时，试了 5 条错路。这些错路和最终成功路径全部被记录、蒸馏成 Skill v1。李四遇到同类问题，20 分钟解决——因为 Memory 里有张三的经验。三个月，200 次 CI loop，Skill 库从 3 个长到 47 个，准确率从 61% 到 92%。

**"这个能力是无法通过采购来达成的，或者至少一半一半，供应商一半，团队自己做好另一半。"**

对手买同样的模型，起点一样。你每交付一颗芯片，踩过的坑攒成经验库，起点就往前挪一次。他抄得走一个工具，抄不走一个从 200 次失败中长出来的经验库。

## 百万美金花在哪

不是花在买更强的模型——模型人人都能买，每月在贬值。

花在买一层让模型敢签流片的工具底座：

- **产生率降下来**——10 个设计自动化工具，能确定的不交给概率，Spec 不一致在结构上不可能
- **逃逸率压到零**——工具返回结构化上下文，不是裸 log；机械裁判全绿才停，写 RTL 的不能同时当唯一裁判
- **爆炸半径控住**——工具是管道可换后端，改一处只坏一处，不触发全片回归

一次流片失败的成本是 500 万到 2000 万美金。r2-eda 把流片失败概率从"人肉 review 漏率"压到"机械裁判零逃逸"。**避免一次流片失败，回本 5 到 20 倍。**

而经验库的复利——第 N+1 颗芯片的起点是第 N 颗的终点——这个价值不在第一次交付，在后面每一次。

# 第一幕

## 第 1 节：你的回归跑了几小时？——先做个体检

---


如果你的 CI 回归从 20 分钟变成了 4 小时，你第一反应是买机器、砍测试、还是干脆不跑了？

我见过一个团队的回答是："每周跑一次，周五下班前提交，周一早上看结果。"这个回答的潜台词是：我们已经接受了 4 小时的回归是不可改变的物理定律。

 ARM 在 DVCon US 2024 论文 mL: Shrinking the Verification volume using Machine Learning 中披露：在约束随机测试（CRT）流程里，ML 在仿真前对随机生成的测试做 pass/fail 预筛，实际进入仿真的测试量约为 CRT 的 16%。在该口径下，单位 CPU 小时的 unique fail signature
  发现效率约为 CRT 的 1.35 倍，给定 1 CPU hour 的 bug 找回率约为 1.6 倍。论文同时说明，ML 与 CRT 需并行运行——既用于持续重训，也用于探索尚未覆盖的设计区域。

  Siemens EDA 在 DVCon Taiwan 2025 论文 Sanity Test Case Selection by Machine Learning Approach 中，在另一套「代码变更 → 测试映射」的 ML 方案下报告了同类结果：用 11–51 个 sanity test 替代 274 个全量回归测试，速度约为全量的 6.6
  倍；在变更模块上，覆盖率与全量回归的差异为 ±0–2%，且选中子集捕获了全量回归的全部失败。

  两篇论文共同指向同一判断：回归的主要浪费，来自大量与当前变更无关、或反复扫过已验证区域的测试——问题往往不是测得不够，而是跑得太多。

回归要验的不是「测完了」——完成不等于优化完成、测试完成，而是这次小变更能否作为可交付特性继续往下走，冒烟测试正是短迭代里的质量门禁；可约束随机测试会在已测过的健康设计区域反复搜索、白白浪费算力，所以应在仿真前用AI筛掉低效测试，筛后仿真量约为原来的
  16%，缺陷发现效率 1.35 倍，缺陷找回率 1.6 倍。

---

### 悬念钩子

好，你说"精准验证"。那怎么知道哪些测试"精准"？凭经验？一个验证工程师能凭记忆判断几十个测试的相关性，但你的回归有 1000 个。凭覆盖率文件？覆盖率告诉你一个测试覆盖了哪些代码行，但不告诉你这次变更影响了哪些测试。凭直觉？周一和周五的直觉不一样，人离职了直觉也带走了。

有一个方法，每次 CI 自动对 1000 个测试逐个打分，只跑分数最高的 50 个，跑完把结果喂回去，下次重算——50 个名单每次都不一样。这个方法 ARM 在用，Siemens 在用，而且两家公司独立验证了有效性。

**下一节，我们讲这个方法的真正内核——它不是"选出 50 个最重要的"，而是"每次重算 50 个"。** 这两个说法的区别，决定了你是做一次性的手工优化，还是搭一个持续进化的系统。


如果你听完这节只带走一个判断，就是这句：**50 不是测试名单的长度，是预算的上限。**

大多数人听到"把 1000 个测试砍到 50 个"，第一反应是恐慌："那 950 个不测了？bug 漏了怎么办？" 这个问题的前提是——50 个是固定的。但如果 50 个是每次重算的呢？今天这 50 个和明天那 50 个完全不同，覆盖的测试总量在一个月内接近 1000 个——只是每次只跑最相关的 50 个。

**预算不变，内容随情况变。**

---

### 案例反转法

想象两个团队。

**团队 A**：手工挑了 50 个"最重要的测试"，固定跑。三个月后，新加了一个 low power 模块，50 个测试里没有 low power 相关的——因为挑的时候这个模块还不存在。一个 low power bug 漏到硅后，返工成本 200 万。

**团队 B**：每次 CI 自动对 1000 个测试打分，取 Top-50。low power 模块加入后，相关测试自然在打分中浮现——因为 low power 测试的覆盖率文件里包含了新模块的文件名，而打分模型把"覆盖率重叠"作为一个维度。low power bug 在第三次 CI 就被抓到。

团队 A 的问题是：50 个是静态名单，无法响应变化。团队 B 的秘密是：**50 是动态预算，名单随变更重算。**

---

### 内核拆解

ARM 的做法用一张图就能说清楚：

代码变更 → CRT 生成候选测试 → ML 打分 → 高分 Top-50 入选 → 仿真执行  
                                                          ↓  
                                                结果回写（pass/fail）  
                                                          ↓  
                                                模型更新（每周重训）  
                                                          ↓  
                                                下次变更 → 重新打分

注意闭环的方向：结果回写模型，模型下次更准。这不是一次性优化，是持续进化系统。

Siemens 的做法略有不同，但函数签名完全一致：

代码变更 → Learned Mapping（代码文件→测试关联）→ 相关测试子集 → 仿真执行 → 结果回写

两家公司的差异只在"怎么打分"：ARM 用 Gradient boosted trees 做 pass/fail 预测，Siemens 用 Learned Mapping 做代码-测试关联。底层结构完全相同：**变更 → 排序 → 子集 → 执行 → 反馈 → 进化。**

**50 是预算，不是名单。** 名单是静态的，预算是动态的。预算是系统进化的前提——约束制造选择，选择产生反馈，反馈驱动进化。

## 第 3 节：仿真前打分：预测+筛选的内核


---

### ARM 论文的三句话

第一句："Using Machine Learning, we classify randomly generated tests on their likelihood of failing/passing pre-simulation."

翻译：在仿真启动之前，对随机生成的测试做分类——判断每个测试通过或失败的概率。

第二句："The objective is to use an ML based predictive filter, before simulating tests on the design."

翻译：在仿真启动之前，用 ML 做一个预测型过滤器。

第三句："ML based filter prunes inefficient tests that waste compute."

翻译：剪掉那些低效测试——跑了也白跑，纯浪费算力。

三句话的关键词都是"before"——在仿真前。不是在仿真后分析结果，是还没跑就决定跑不跑。

---

### 技术细节

ARM 的实现细节：

- **算法**：Gradient boosted trees。选择原因是 pass/fail 比例 95:5，极度不平衡，神经网络在这种数据上容易过拟合到 majority class。
  
- **输入特征**：test knobs（测试配置参数）+ 变更特征（改了哪些文件、哪些模块）。
  
- **输出**：pass/fail 概率。不是"这个测试覆盖了多少行代码"，是"这个测试有多大概率挂"。
  
- **训练数据**：过去两周的回归结果。每周重训一次，保持模型对最新代码状态的敏感度。
  
- **打分维度**：可切换——wall-clock（运行时间）、coverage（覆盖率增量）、pass-fail（历史失败率）。根据当前验证目标选择维度。
  

Siemens 的实现细节：

- **算法**：Learned Mapping——学习代码文件和测试之间的映射关系。
  
- **机制**：每次变更来了，看改了哪些文件，查映射表，找出关联测试。
  
- **效果**：11-51 个测试替代 274 个，快 6.6 倍，覆盖率差异 ±0-2%，捕获全部失败。
  

---

### 一课一钉

**预测+筛选的内核不是模型多复杂，是决策在仿真前。** 仿真前决策省的是仿真时间，仿真后分析省的是分析时间——前者贵几百倍。

## 第 4 节：都江堰效应：为什么约束越紧系统越准

---

### 反常识结论先行

**50 这个约束不是系统的缺陷，是系统进化的动力源。**

大多数人听到"只跑 50 个测试"，第一反应是"不够安全"。真实的逻辑恰恰相反：如果你跑 500 个，失败信号淹没在 490 个通过的噪音里，模型学不到东西。只跑 50 个，每一次失败都是珍贵的反馈，模型被迫在窄约束下进化。

---

### 都江堰的物理课

2200 年前，李冰在岷江上修了都江堰。三道结构：

**鱼嘴**：把岷江劈成内江和外江。表层清水走凹岸进内江，底层浑水走凸岸进外江。没有闸门，没有电力，水自己在弯道里做出选择。

**飞沙堰**：一道低矮的溢流堰，高出河底仅 2 米。水从旁边流过，泥沙被旋流横向排出。它不挡水，它坐水。

**宝瓶口**：宽仅 20 米。不管上游来多少水，进成都平原的上限是 700m³/s。

宝瓶口窄 → 上游壅水 → 壅水遇到离堆 → 产生旋流 → 旋流就是飞沙堰。如果宝瓶口是敞开的宽渠，水全冲过去，没有壅水，没有旋流，飞沙堰不过是一堆泡在水里的石头。

**宝瓶口的窄不是缺陷——是驱动飞沙堰生长的动力源。**

洪水越大，旋流越强，排沙越狠。没有闸门，没有电力，没有人在洪水来了才跑去拧开关。持续运转。最终宝瓶口不扩，但通过的水量变大——不是口子大了，是水变干净了。

---

### 翻译到验证

| 都江堰         | 验证系统               |
| ----------- | ------------------ |
| 宝瓶口（20 米窄口） | 50 个测试的预算上限        |
| 壅水          | 只有 50 个名额，测试必须竞争入选 |
| 飞沙堰旋流排沙     | 智能体持续进化，越筛越准       |
| 没有闸门/电力     | 全自动闭环，每次 CI 自动重算   |
| 水变干净        | 系统越来越准，漏的越来越少      |

每次 CI 只跑 50 个 → 必然有测试被拦在外面 → 被拦的测试如果其实应该跑（漏了 bug）→ 失败信号回写 ML → ML 下次更准。

如果你跑 500 个：

- 失败的测试仍然失败
  
- 但失败信号淹没在 490 个通过的噪音里
  
- 模型分辨不出"这次失败是因为测试相关"还是"巧合"
  
- 系统不进化
  

50 这个数字的意义：它不是随意的，不是拍脑袋定的，不是"我们只能跑 50 个"。**它是宝瓶口——是那个窄口约束，让壅水够猛、旋流够强、排沙够狠。**

---

### 一课一钉

**约束制造选择，选择产生反馈，反馈驱动进化。** 系统产出增加不是因为约束放松了，是因为输入被持续净化了。宝瓶口不扩，通过量变大——不是口子大了，是水变干净了。

## 第 5 节：两个权重源：数据 vs 人

---

### 反常识结论先行

**ML 和 PM 不是二选一，是同一个函数的不同权重源。**

ARM 用 Gradient boosted trees 打分，权重来自历史 pass/fail 数据。PM 用业务判断排序，权重来自客户风险、市场窗口、架构影响面。输入一样（变更了什么），输出一样（排序后的测试列表），反馈一样（跑完结果回写），区别只在中间的权重来源。

---

### 函数签名对比

|维度|ARM（ML）|PM（人）|
|---|---|---|
|输入|代码变更 + 测试历史|需求变更 + 业务影响|
|处理|Gradient boosted 打分|PM 优先级判断|
|输出|Top-50 测试列表|优先级排序的 case 列表|
|反馈|跑完结果回写 ML|跑完结果调整优先级|
|权重源|历史数据统计关联|业务上下文判断|

**函数签名完全一致：变更 → 排序 → 子集 → 执行 → 反馈 → 进化。**

---

### ML 的优势与盲区

**优势**：

- 样本量：两周回归数据，几千次 pass/fail 记录，人记不住
  
- 一致性：周一和周五的判断是同一个模型，人的直觉会变
  
- 速度：对 1000 个测试逐个打分，毫秒级；人做不到
  

**盲区**：

- 上下文：训练数据里没有的维度永远学不到。客户下周要 demo 这个功能——这个信息不在 test knobs 里，不在历史 pass/fail 里
  
- 可解释性：ARM 论文自己承认"很难解释 ML 为什么选择某个测试"
  

---

### PM 的优势与盲区

**优势**：

- 上下文：知道哪个客户在等哪个功能，知道 tape-out 风险预算还剩多少
  
- 可解释性：能说出"为什么这个测试优先"——因为客户下周要 demo
  

**盲区**：

- 样本量：人凭记忆判断几十个测试，对 1000 个测试无法逐一评估
  
- 一致性：周一和周五的直觉不一样，人离职了直觉带走
  

---

### 叠加方案

**权重源的选择不是原则站队，是工程决策。** 你可以在同一个框架里：

- ML 提供基准分（数据驱动，一致性好）
  
- PM 在关键节点加权重覆盖（上下文信息，客户风险）
  

比如：

- 基准分：ML 给每个测试 0-100 分
  
- PM 覆盖："客户下周 demo low power → low power 相关测试无条件 +50 分"
  
- 最终分 = 基准分 + PM 覆盖分
  

---

### 一课一钉

**ML 提供基准，PM 提供覆盖。** 不是二选一，是叠加。ML 解决"大规模一致打分"，PM 解决"上下文信息补位"。

---

两条效率路径：运行效率 × 搭建效率

---

### 反常识结论先行

**ARM 的 ML 筛选解决的是"跑哪些测试"（运行效率），但还有另一条同样重要的路径："怎么快速搭建测试"（搭建效率）。两条路径叠加，效果相乘。**

---

### 路径一：运行效率（ARM 方法）

已经讲了五节。核心：每次 CI 只跑最相关的 N 个测试，用预测模型动态筛选。

效果：流量砍 80-84%，速度提升 1.35-6.6 倍，质量不降（覆盖率差异 ±0-2%，捕获全部失败）。

---

### 路径二：搭建效率（标准接口方法）

算法类模块有个特征：以算法实现为核心，大多是 mem 进 mem 出。这意味着：

- 算法模型可以千变万化
  
- 但基于同样标准形式的验证环境可以是同一套
  
- 同一套 env，只需改动接口 DUT 连接以及 C 模型调用，就能适配任意模型
  

**标准接口让所有模块共享同一套 env → 测试之间结构高度同构 → 可比性更强 → ML 筛选冗余更容易。**

具体来说：

- 设计人员吭哧吭哧半个月搞个新模块
  
- 验证人员半天完成环境搭建适配
  
- 因为 env 骨架不变，只改 DUT 连接和 C 模型调用
  

**这条路径解决的是"搭建时间"，ARM 路径解决的是"运行时间"。两者叠加：搭建从两周变半天，运行从 4 小时变 20 分钟。**

---

### 推广路径

"把标准接口形式推广成为公司标准"——这意味着：

- 新模块接入验证环境的时间从"两周"变成"半天"
  
- 所有模块的测试结构同构，ML 模型更容易学习"哪些测试相关"
  
- 验证知识可以沉淀在标准接口文档里，不因人员流动而丢失
  

---

### 一课一钉

**筛选解决运行效率，标准接口解决搭建效率。双轮驱动，效果相乘。**

### 智能体每周蒸馏什么

不是"测试 A 分数 0.83"这种具体数字——数字会变，规则更稳定。

是**可执行的规则**：

- "改了 axi_wr_channel 的，rd_channel 的 scoreboard 相关测试历史上挂过 4 次——优先跑"
  
- "test_apb_reg_reset 连续六周全绿——不再作为 smoke 候选"
  
- "本周新增 low power 测试，PM 标注为客户 demo 路径——无条件进 smoke"
  

这些规则的特点是：**人看得懂，脚本能执行，每周更新。**

---

### 智能体蜂群并行查证

每次 CI 变更来了，300 个 agent 并行工作，每个 agent 查一个测试：

- **查证任务 1**："这个测试的覆盖率文件里，是否包含本次 diff 修改的文件名？"
  
    - 答案只有两种：「查到」或「查不到」
    
- **查证任务 2**："这个测试最近 30 天的 pass/fail 序列中，是否有 fail？"
  
    - 答案只有两种：「有」或「无」
    
- **查证任务 3**："这个测试的 knobs 配置，是否与本次变更的模块有历史关联？"
  
    - 答案只有两种：「关联」或「不关联」
      

**关键：agent 只做查证，不做判断。** 查证是信息检索，确定性高。判断是价值决策，交给脚本。

为什么 agent 不能做判断？因为 LLM 有非确定性——同一个 prompt 问两次，偶尔会在某个判定项上给出矛盾的 yes/no。这对 CI gate 是致命的。但查证（查文件、查历史记录）是确定性的——文件要么包含这个字符串，要么不包含。

### 脚本毫秒级排序

def score(test, change, evidence):
      s = 0

      # Task 1: 覆盖率关联
      if evidence.covers_change(test, change):
          s += 30
    
      # Task 2: 历史失败率
      rate = evidence.recent_fail_rate(test)
      if rate > 0:
          s += 20 * rate
    
      return s


  def select_smoke(pool, change, rules, evidence, budget=50):
      # PM override: 无条件入选，不走排序
      must_run = rules.pm_override_list(change)
      remaining_budget = budget - len(must_run)

      # 剩余 pool 算分排序
      rest = [t for t in pool if t not in must_run]
      rest_sorted = sorted(rest, key=lambda t: score(t, change, evidence), reverse=True)
    
      return must_run + rest_sorted[:remaining_budget]

# 排序，取 Top-N  
tests_sorted = sorted(tests, key=lambda t: score(t, change, rules, evidence), reverse=True)  
smoke_tests = tests_sorted[:50]

**确定性**：同样的输入，永远产出同样的输出。没有 LLM 的随机性。毫秒级执行。

Agent 只能查证，不能判断

---


**把 agent 的任务从"判断"降级为"查证"，是这个架构能用在 CI gate 里的前提。**

如果你让 agent 做判断——"这个测试应不应该进 smoke？"——你会遇到一个问题：同一个 diff、同一批测试、同一套规则，两次跑 CI 出来的 top-50 列表可能不一样。LLM 的非确定性会让你的 CI 变得不可预测。

但查证是不同的——"这个覆盖率文件里有没有这个文件名？"——答案是确定的。

### 查证的具体实现

**查证 1：覆盖率关联**

输入：本次 diff 修改的文件列表 + 测试的覆盖率文件  
输出：这个测试覆盖了哪些变更文件？

**查证 2：历史失败模式**

输入：测试最近 N 次的 pass/fail 结果  
输出：失败次数、最近失败时间、失败模式

**查证 3：模块关联**

输入：本次变更的模块 + 测试的配置 knobs  
输出：测试是否针对变更模块

**查证 4：代码变更影响**

输入：diff 的代码行 + 测试的触发条件  
输出：代码变更是否可能触发测试的敏感路径

四个查证任务，每个都是信息检索，每个答案都是确定的。300 个 agent 并行，分钟级收表。

### 判断留给脚本

脚本用确定性规则做最终排序：

- 覆盖率关联 +30 分
  
- 历史失败 >0 +20×失败率
  
- PM 覆盖 = 无条件 100 分
  
- 连续全绿 4 周 = 排除出 smoke

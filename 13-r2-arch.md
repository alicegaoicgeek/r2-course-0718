# r2-arch：架构探索九步收敛，编译持续交付的芯片产线

SoC 很少从零写起。多数是把已验过的 IP 或子系统拼起来；到了芯片级，验证的主战场往往不是再证一遍每个核，而是查清这些部件之间的集成是否站得住。时钟、复位、电源、安全一类公共能力，在整片 UVM 工作量里常常能占到四成以上——漏一项，后面整条产线都要补。

更糟的是：传统覆盖多半只能等仿真跑完才量得出来，等于把关口推到验证极晚期。拖得越晚，未检出的逻辑与功能缺陷风险越高——这正是 ASIC 反复流片的头号来源之一。若规划阶段无意漏掉某一公共任务，问题拖到逻辑后期才爆，补救就要砸掉大量资源。

性能与架构探索也在抢时间窗。行为级性能模型可以在设计极早期就跑；RTL 仿真进得稍晚、速度又慢；等固件就绪再上加速平台，已经偏晚。项目早期就要能不依赖生产固件、把真实用例压上去：早碰撞，才能在同一项目周期里把反馈送回架构，再综合、再裁剪。

系统侧用一套算法语言定模型，设计侧用另一套语言落 RTL。验证若按规格书重写预测器，规格一误解就返工；系统侧算法一改，预测器还得手工跟着改——耗时、易错、跟不上。库里明明有带全套检查器的验证资产，设计侧却常因不懂用法、日程太紧而用不起来：延期、漏缺陷，契约进不了产线。规格本身还在变，验证产物必须跟着改——晚半拍，集成阶段就要全额买单。

所以架构探索真正难的，不是写 RTL。

在信息极度匮乏的研发早期，就要精准划定软硬边界与接口契约。用研发与返工代价极高的刚性硬件，承接充满未知与变化的动态需求。什么固化在硬件？什么留给软件？什么必须参数化？什么必须成为接口契约？什么必须提前验证？什么必须进入受控变更？

接口不是拍脑袋设计出来的。接口需求通过真实场景碰撞暴露，接口契约通过架构设计收敛。需求由碰撞暴露，契约由架构收敛，错误留在左边，确定性交给机器。

先把结构钉死——交付边界、接口契约、空壳占位、工具门禁——后面的复用和智能体填空才站得住。结构是因，复用是果；别反过来赌。无论 IP 来源多么复杂，最终都必须在顶层收敛为顶层 RTL、软件、接口契约和验证闭合。

九步流程图把收敛收成产线：上泳道决策，中泳道机器可读契约，下泳道虚原型、高层综合估算、加速器碰撞回灌。签收看工具真实输出——综合、仿真、静态检查、形式证明、协议检查、性能监控——不看会上点头。九步不是瀑布，是基于 loop 的螺旋。每一步问清：试错成本、机器能否消费、左移了什么错误。不能消费，不算交付。没左移任何东西，这一步是白做的。

## 第一层 · 设计数据进场

架构师的核心决策：哪些路径适合本项目？哪些接口必须统一？哪些可以动态适配？哪些 HSI / ICD 必须 Frozen？哪些可以迭代？哪些能力固化在硬件？哪些能力留给软件？哪些风险必须在 VP / HLS / Emulator 阶段提前暴露？

不同来源的 IP（复用、手写、HLS、ASIP、Fixed CPU + SW……）有截然不同的研发节奏、风险点和契约形态。先确定路径，再谈接口。

How（操作动作）：

1. 列出系统功能清单，逐项标注变化频度（高=软件、低=硬件）；
2. 对每个功能填路径决策表：来源、风险、SW 形态、可复用度；
3. 与系统、软件、验证团队三方对齐路径假设。

Exit Criteria：每个功能块都有明确的集成路径标签；风险高的路径已标注「需 VP 提前验证」。

左端设计数据与上表绑定：系统需求回答「验什么值」；变化假设 / 变化频度表驱动软硬切片；复用 IP 资产决定哪些走复用路径；工具链 / CI 基础设施决定 VP、HLS 估算、Emulator、自动回归能否跑起来。

## 第二层 · 上泳道决策：路径 → 边界 → 接口三分

### 路径选型：决定 IP 怎么进来

把一颗 IP 集成进 SoC，先问三个问题：RTL 从哪儿来、软件栈要不要新建、含不含处理器。三个问题一组合，就是六种集成路径。验证策略和风险分布各不相同，但核心区别只有一个：有没有明确的交付方。

| # | 名称 | 定义 |
|---|---|---|
| ① | 复用硬件、无软件栈 | 成熟硬核 / 硅验证 RTL 直接集成（Hard IP） |
| ② | 新建硬件、无软件栈 | RTL 从头手写或按规范全新设计 |
| ③ | 自动生成硬件、无软件栈 | 从高层描述（C/算法/DSL）综合出 RTL；LLM for Chip 最集中落点 |
| ④ | 新建处理器核、新建软件栈 | 自研 CPU/DSP/NPU 核心 + 完整工具链与软件生态 |
| ⑤ | 复用处理器核、新建软件栈 | 集成现有核（Arm/RISC-V 等），自研上层软件与驱动 |
| ⑥ | 软硬全复用 | 子系统连同固件 / 驱动 / 中间件整体打包复用 |

任意一颗 IP 怎么进 SoC，都能对上其中一条。六种都要 ICD，只是 ICD 从哪儿来不同——不是「有了 IP-XACT 就覆盖六种」。

打法按开关分岔：

- 有交付（①⑥，⑤的核）：走供应商包 / Day-1 评估，先问交付物齐不齐。
- 无交付（②④，⑤的软件）：没有外部交付包可开箱，集成要左移——空壳主要打这条。
- ③ 标「模糊」：有工具产出，但没有「供应商」这个角色；别拿供应商 QIP 原样套，要改成「工具链可复现」那套问法。空壳策略仍然适用。

「有没有交付边界」不是贴标签，是总开关。后面怎么评估、能不能先挂空壳、风险从哪爆，全看这一列。

所有集成路径最终都必须收敛到接口契约；ICD 覆盖所有受控接口边界，其中软件可见部分形成 HSI。软件可能是复用、生成、适配或重新开发。

### 边界划分：软硬切片 + 风险定级

架构师真正难的不是写 RTL，而是回答：什么固化在硬件？什么留给软件？什么必须参数化？

对每个模块给出五个标签：HW-only / SW-only / HW+SW 协同；是否需要参数化（generic / parameter）；接口风险等级（关键 / 普通 / 可裁剪）。

How：

1. 按「变化频度 × 性能要求」画四象限：高变高性能 → 参数化硬件 + 软件配置；
2. 标注每个接口的风险等级——决定后续四维基准的覆盖深度；
3. 输出「软硬切片表」+「接口风险表」。

Exit Criteria：每个接口都有风险等级；关键接口已锁定为后续 ICD Frozen 候选。

### 接口三分：管道 / 阀门 / 翻译

接口设计的本质不是 pin list，而是信息怎么走、异常怎么挡、差异怎么藏。

| 层级 | 核心问题 | RTL 存在形式 | VP/TLM 存在形式 |
|---|---|---|---|
| 管道 Pipe | 信息走哪几条通道？带宽够不够？会不会堵？ | AR / AW / W / R / B、FIFO、NoC path、DMA path | TLM socket、事务队列、性能模型 |
| 阀门 Valve | 什么允许跨边界？什么必须挡住？何时反压？ | valid / ready、credit、timeout、CDC、isolation | 断言、契约检查、异常注入 |
| 翻译 Translator | 下层差异如何对上层不可见？ | Wrapper、protocol bridge、ID mapping、ROB | TLM transaction、地址映射、一致性模型 |

接口不是「你支持什么功能」，而是「信息走哪几条管道、经过哪些阀门、如何被翻译，并且怎么保证不堵、不乱、不死」。接口设计的本质不是 pin list，而是：数据如何流动；请求如何排队；响应如何返回；异常如何收敛；软件如何感知；工具如何检查。

对每个接口，分别填三层模板：

| 层 | 必答问题 |
|---|---|
| 管道 | 通道数？带宽？Outstanding？反压路径？ |
| 阀门 | 什么不许跨边界？timeout？error response？CDC？isolation？ |
| 翻译 | 协议转换？ID mapping？ordering？一致性？中断合并？ |

How：用三层接口模板逐项填空，不允许留白；留白处即风险点，进入 VP 场景碰撞重点碰撞；输出「接口三层卡片」（每接口一张）。

Exit Criteria：每个关键接口都有完整的三层卡片；留白条目已转化为待碰撞的「假设清单」。

操作信条：先碰撞，再冻结；先行为，再信号；先契约，再实现；先机器，再人工。把错误留在左边，把确定性交给机器，把决策留给人。

## 第三层 · 中泳道 ICD：行为契约先行

关键原则：先碰撞，再冻结——不在真实 workload 前过早冻结接口。先行为，再信号——先定义事务语义，再落到 RTL pin / timing。先契约，再实现——RTL、driver、UVM、formal 消费同一份 ICD。先自动检查，再人工评审——人负责 trade-off，机器负责一致性。收敛 ≠ 冻结——收敛是需求裁剪完成，冻结是进入受控变更流程。Frozen 不是不能改——Frozen 意味着变更必须有影响分析、版本管理和自动回归。

ICD，Interface Control Document，不应只是接口说明书，而应是：架构、RTL、验证、软件、系统、Flow 共同消费的机器可读契约。

| 研发阶段 | VP / 架构探索层 | RTL / SW 开发层 | SoC 集成验证层 |
|---|---|---|---|
| ICD 状态 | ICD v0.5，草案 | ICD v1.0，Frozen，变更需影响分析 + 回归 | ICD v1.x，基准 + 受控变更 |
| 核心动作 | 行为契约；C / TLM 模型；真实场景探索；性能评估 | 信号级契约；Wrapper / driver / UVM / SVA 生成；评审 | 全系统一致性检查；变更影响分析；自动回归 |
| 试错成本 | 最低，但依赖模型保真度 | 中等，可控 | 极高，需严格变更管理 |
| 关键输出 | 行为模型、性能模型、接口草案 | RTL 接口、HSI、checker、driver | 集成基线、回归结果、签核证据 |

ICD 的正确理解：ICD 不是文档，是可执行契约。ICD 不是 pin list，包含事务、时序、异常、性能、一致性、软件可见行为。ICD 冻结后不是不能改，而是进入受控变更。ICD 不靠人阅读保证一致，靠工具自动检查保证一致。机器可读 ICD 不是一种文件格式，是一组可被工具消费的契约资产。

契约写在机器可读 ICD 里（需要时再投影成 IP-XACT 给工具用）——ICD 管「怎么接」，不管「RTL 大半自动长出来」。IP-XACT 方向对，但必要不充分，标准到工程还有 usability gap；包装格式不是契约，也别拿包装格式当银弹。

接口一定，空壳就不能改。空壳就是契约；后面真 IP 必须按它接。破了这个规矩，空壳阶段跑通的 case，真 RTL 一接全废。

无外部交付包的那几条路（②③④ 为主），痛点一样：IP 写得久，集成撞得晚。等 RTL 齐了再塞进 SoC，接口不对、时钟复位配错、总线协议有坑——返工极贵。解药不是「把 IP 写快点」，而是先挂一个最小可用版本占位，把两件事拆开验：这颗 IP 本身对不对；它能不能接进这颗 SoC。

### ICD v0.5 草案

先行为，再信号；先契约，再实现。早期不写 pin，写事务语义。

ICD v0.5 必须机器可读，包含：事务定义、流量与反压、异常路径、性能包络、软件可见行为（HSI）。

操作动作：

1. 选用机器可读载体：SystemRDL / IP-XACT / YAML+schema；
2. 行为契约用 SystemC / TLM / C++ 模型表达，可执行；
3. 生成第一版骨架：C header、UVM register model、wrapper skeleton；
4. 标注每个字段状态：Draft / Under Review / Candidate-Frozen。

出口准则：ICD v0.5 可被工具消费；行为模型可在 VP 中跑通最小事务。

中泳道不会一次写完。下泳道碰撞会把需求撞出来，回填成 ICD v0.6 / v0.7，直到性能数字、HSI 与实现可行性一致。

## 第四层 · 下泳道碰撞：让需求自己暴露

接口需求不是拍脑袋设计出来的，是被真实场景碰出来的。VP 阶段试错成本最低。

### VP 场景碰撞

在 Virtual Platform 上跑真实 workload：IP + 软件 + 数据流 + QoS + 异常路径一起碰撞。

操作动作：

1. 准备至少 3 类场景：典型负载 / 峰值压力 / 异常注入；
2. 监控接口行为：反压频率、Outstanding 占用、Timeout、死锁、QoS 饥饿、CDC 毛刺；
3. 把每条暴露的问题登记为「接口需求条目」，反馈到 ICD；
4. 关键发现：哪些假设被推翻？哪些参数需要调整？

出口准则：至少跑通典型 + 峰值 + 异常三类场景；所有暴露的需求已回填 ICD v0.6 / v0.7。

成熟团队的一种落地：RTL 还没冻之前，先挂 Virtual Platform 当参考模型，塞进 UVM Scoreboard 当 predictor，四方都拿同一个 VP 在 EDA 里打分，别各养一份手写 SV 参考。对齐不是开会对齐说法，是同一套 case 跑 EDA。

### HLS / 快速实现估算

行为对了，不代表能实现。要尽早拿到面积、时序、功耗的量级反馈。

对关键模块跑 HLS 或快速综合估算，验证 ICD 中的性能包络是否可实现。

操作动作：

1. 对计算密集 / 高带宽接口跑 HLS，得 area / timing / latency 量级；
2. 对 RTL 路径跑 quick synthesis（粗约束），看 timing 是否可收敛；
3. 不可实现的指标 → 回到 ICD 调整或回到边界划分重切。

出口准则：关键路径都有实现量级数据；ICD 中的性能数字与实现可行性一致。

### Emulator / FPGA：系统级真实反馈

VP 速度快但保真度有限，Emulator 提供系统级真实交互——多 IP、多时钟域、真实软件栈一起跑。

在 Emulator / FPGA 上做系统级 bring-up 预演，暴露 VP 看不到的问题：CDC、复位顺序、性能拐点、软硬协同 bug。

操作动作：

1. 选 2~3 个端到端场景（如：boot / 典型业务 / 异常恢复）；
2. 让真实软件栈对接 RTL 接口，验证 HSI 契约；
3. 性能监控点采集真实带宽 / 延迟 / Outstanding 分布；
4. 发现的接口问题再次回填 ICD。

出口准则：关键场景在 Emulator 上跑通；ICD 中的 HSI / 性能契约通过真实软件验证。

图上的蓝虚线就是这条回路：碰撞暴露需求 → 回填 ICD → 必要时回到上泳道重切边界。九步不是瀑布，是基于 loop 的螺旋。

## 第五层 · 四维基准闭环：让机器签收契约

人评审保证不了一致性，机器才能。每个关键接口必须穿过四道闸门。

四层基准按接口风险等级裁剪；关键接口必须全过，普通接口满足对应必选基准后，才能标为 Frozen。

| 基准层级 | 验证核心目标 | 工具 / 手段 |
|---|---|---|
| Golden 层 | 接口行为与参考模型在定义的抽象层级和场景范围内等价 | C model、SystemC / TLM、Spike、DiffTest |
| 协议层 | 接口时序不违反标准协议和项目约束 | UVM VIP、Protocol Checker、AXI / CHI checker |
| 形式化层 | 在明确假设 assumption 下，对所有合法输入，契约属性成立 | JasperGold、VC Formal、SVA、PSL |
| 性能层 | 带宽、吞吐、延迟、Outstanding、反压、QoS 达标 | Performance Monitor、NoC simulator、VP perf model |

注意：Golden 不一定要求逐周期完全一致，而是要求在定义好的抽象层级上等价。Formal 不是证明「任何输入都正确」，而是证明「在明确假设下，对所有合法输入成立」。性能层不是锦上添花，很多接口 bug 本质就是性能 bug。简单接口可按风险裁剪；高带宽、高一致性、高安全、高实时接口必须严格闭环。

操作动作：

1. 对每个关键接口建立四维基准矩阵；
2. 工具输出作为唯一签收依据（拒绝「我看了一下没问题」）；
3. 自动生成 checker / assertion / agent 资产，避免人工二次实现。

出口准则：关键接口四维全绿；普通接口至少通过协议 + 必选基准。

确定性验证闭环：以综合 / 仿真 / lint / formal / 协议检查 / 性能监控等工具的真实输出为依据。

## 第六层 · Frozen + 受控变更：ICD v1.0 上线

收敛 ≠ 冻结。收敛是需求裁剪完成，冻结是进入受控变更流程。Frozen 不是不能改，是改必须有代价。

ICD v1.0 进入版本管理，所有变更走「影响分析 + 自动回归」流程。

操作动作：

1. ICD v1.0 入库，打 baseline tag；
2. 变更模板必填：影响范围（哪些 RTL / driver / UVM / formal）、回归矩阵、风险评估；
3. 自动重生成：wrapper、register model、checker、C header、driver skeleton；
4. CI 流水线触发自动回归，红则阻塞合入。

出口准则：ICD v1.0 Frozen 入库；任意接口变更可在 X 小时内出影响报告 + 回归结果。

右端三件产出同时成立：Top-Level RTL + Software、Frozen ICD v1.0、验证资产。无论 IP 来源多么复杂，最终都必须在顶层收敛为：Top-Level RTL + Software + Interface Contract + Verification Closure。

## 收束

每一步都要回答三个问题：

| 问题 | 落脚点 |
|---|---|
| 这一步的试错成本是多少？ | 影响范围 × 验证代价 × 对齐损耗 |

| 步骤 | 主导 | 必须参与 | 交付物消费方 |
|---|---|---|---|
| 1 路径选型 | 架构师 | 系统、软件、IP | 项目计划 |

九步不是瀑布，是基于 loop 的螺旋。每一轮 VP / HLS / Emulator 的反馈，都让 ICD 更收敛一点；每一次机器化的检查，都让 Frozen 更可信一点；直到多源异构 IP 安全收敛到 Top-Level RTL + SW + Frozen ICD + Verification Closure。

先碰撞，再冻结；先行为，再信号；先契约，再实现；先机器，再人工。把错误留在左边，把确定性交给机器，把决策留给人。

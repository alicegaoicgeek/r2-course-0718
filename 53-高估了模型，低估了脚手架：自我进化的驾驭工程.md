# 高估了模型，低估了脚手架：自我进化的驾驭工程

---

你的 Agent 能跑。它能写代码、能改文件、能跑测试、能在 Slack 里通知你「修好了」。你觉得 Loop 搭起来了。

然后你发现：昨天修好的 bug 今天又出现了。Agent 跑了 30 轮不知道自己跑偏了。两个工程师碰到同一个问题，第二个从零开始。每次排查结果——试过哪五条错路、最后到底是边界条件没对齐还是时钟域 crossing——全在聊天窗口里蒸发了。

**能跑 ≠ 能收敛。** 你说「模型还不够强」。模型更强也没用。

Scaling Laws 决定天花板，Harness 决定走多稳多远。这篇文章只回答一件事：在权重级「硬递归」——模型真正能改自己权重——仍遥远时，**真正可持续的自我进化，第一步不是改模型，是改模型外面那层脚手架。**

读完，你应该能对着自己的 Loop 系统——不管是 `/goal` 命令、自建编排流水线、还是团队的 CI 修复 bot——逐一对号入座，知道它现在站在哪里，哪里在漏，以及下一步该修什么。

---

## 你先得知道自己站在哪里

你第一次听到 Loop Engineering，很可能以为自己要学一种新东西——比 Prompt 更高级、比 Context 更自动化。于是你开始研究怎么让 AI「自己跑起来」。

停。四范式嵌套告诉你：**不是替代，是包含。**

AI 开发不是一条从低到高的进化阶梯，而是一层套一层的嵌套结构：**Prompt ⊂ Context ⊂ Harness ⊂ Loop。**

最内层是 Prompt——决定模型对单次指令的理解。Context 包在外面：给模型资料、文档、代码库，让「理解」建立在正确的信息上。Harness 继续向外：为模型搭建完整运行时——工具调用、权限管理、反馈回路、记忆持久化。Loop 在最外层：动态调度 Harness，让系统自己发现任务、分配任务、检查任务、决定下一步。

关键认知只有一句：**Loop 里面每一轮仍然需要好的 Prompt、好的 Context、好的 Harness。** 外圈跑得再好，内圈推理错误，整个系统就跑偏。学了 Loop 不代表可以扔掉 Prompt——就像有了操作系统不代表可以不要 CPU。

你的角色也随着范式外移。Prompt 时代你写提示词。Context 时代你给资料。Harness 时代你搭办公室。Loop 时代你设计自动化循环——你从执行者变成了设计者。

这不是一个抽象的分类游戏。一旦你接受嵌套模型，你就能回答两个此前模糊的问题。第一，「我现在在干什么？」——你主要在 Prompt 层写提示词，还是在 Harness 层搭工具管道？第二，「下一步该学什么？」——不是学更多 Prompt 技巧，而是把外面那一层建好。从 Prompt 到 Context 到 Harness 到 Loop，每一步外移都是一次能力升维：不是在同一个层次做得更好，是换了一个层次工作。

如果你卡住了，大概率是因为跳级了——从 Prompt 直接跳到 Loop，中间没建 Context 和 Harness。等于没修路就想跑车。

## 能跑 ≠ 能收敛

你的 Agent 能跑。但回路闭合只是第一道防线。能闭合不代表你走到终点了。从「能跑」到「能收敛」，中间有四道伤。这四道伤不是模型的问题。模型更强也没用。它们的解药全部在工程架构。

**第一道：经验孤岛。** 张三用 Agent 调通了一个偶发测试失败，花了 4 小时试了 5 条错路，最后发现是时钟域 crossing 的边界条件没处理好。李四第二天遇到几乎一样的问题——从零开始。张三的经验留在他的私人聊天记录里，对李四没有任何帮助。

「如果一次排查只在一个人身上产生价值，组织的智力总量没有增加。」

根因不在模型不够聪明。AI Agent 每次会话结束即丢失全部上下文，不存在任何会话内的替代方案。跨 session 的状态必须落在磁盘上——GOAL.md、PROGRESS.md、LOOP-STATE.md 不是组织技巧，是物理必需。没有 Memory 的 loop 会在每次重启后把昨天修过的 bug 再修一遍——因为它不知道已经修过了。

**第二道：自评无效。** 模型对自己产出天生宽容。「学生自考题自判不成立——但老师会说不对。Agent 缺的就是那个『老师』。」你的 Agent 提议把学习率调到 0.01，自己改配置，自己跑评估，然后自己宣布「精度提升，建议接受」。同一个模型从头到尾包办了提案、实施和评判。它不是偶尔宽容——它是结构性地宽容。

**第三道：无限空转。** Agent 永远觉得「还差一点」。2023 年的 AutoGPT 因这个缺陷臭名昭著，给整个 Agent 领域蒙上了「玩具」的阴影。没有 MAX_ITER，没有 Token 预算，没有无进展检测——等于没有额度上限的信用卡。

**第四道：模糊性复利。** 你写了一个 exit condition：「让代码更好」。Loop 以分钟为单位迭代。每一轮微小的方向偏移变成下一轮的输入基础。3 到 5 轮后，Loop 已经完全偏离原始意图。不是一次跳错，是逐步漂移，复合放大。

四道伤口互相关联。修一道不修其他，修复不完整。修了记忆但没修自评——落盘的是错误结论。修了硬停但没修 exit condition——停在第 20 轮时，Agent 是在对的方向还是错的方向？你不知道。

## Loop 的本质不是跑，是收敛

你写了一个循环。Agent 每轮看状态、改代码、跑测试——直到测试全绿。

听起来很简单。但控制论告诉你：你写的只是一个前馈环。

Loop Engineering 把 Loop 拆成四个节点：**设定目标 r → 执行器产出 y → 传感器测偏差 e = r - y → 控制器根据 e 调整策略 → 新产出 y' 趋近 r。** 缺任何一环，回路断开。

关键在 e——测量偏差。没有传感器，你就不知道 y 和 r 差了多少。没有控制器，你就不知道怎么调整。没有这个回路的 Loop 叫开环：行为不可控、不可预测、不可收敛。芯片流片收敛用的也是同一个回路。

但回路能闭合 ≠ 回路会收敛。所以你需要三个硬停条件——不是可选项，是任何生产级 loop 的必选项：

**MAX_ITER**：最多跑 20 轮。不是「建议设 20」，是「必须有」。分钟级迭代中，模糊性会复利增长——跑得越久，偏离越大。

**无进展检测**：连续 N 轮没有实质性改善 → 停。不是「看起来在改进」，是「改了什么」。抖动——在两个错误之间来回切换——是无进展检测要抓的典型模式。

**Token 或金额预算上限**：Uber 四个月烧完全年 AI 预算的教训。给每个工程师开通 AI 编码工具，没有统一管理触发频率、预算、停止条件——每个人的 loop 当「无限 ATM」在用。最终设每人每月每工具 $1,500 上限。

这三个停止条件不是保守，是速度的数学必然。在高频迭代中，不设上限的系统不是在「探索」，是在「漂移」。

2025 年，Geoffrey Huntley 发布了一个看起来极其愚蠢的方案——Ralph Loop。`while :; do cat PROMPT.md | claude ; done`。核心创新不是技术，是纪律：每次迭代重置上下文到固定锚文件，进度存在磁盘，每次只做一个离散工作单元。即使最简单的循环，只要遵守退出纪律和状态持久化，就比无纪律的复杂 Agent 可靠。

## 同一个模型不能既当球员又当裁判

你用一个模型写代码、审查代码、然后宣布「通过」。

这个流程看起来没问题。但它有一个致命缺陷：**做工作的模型不是判断工作完成的模型。**

DVCon China 2023 的硬数据：LLM 生成 Python 参考模型正确率约 95%，生成 Verilog RTL 正确率不足 60%。更致命的是——同一模型自己去验证自己写的 RTL。它不会发现它不知道的错误。差距 35 个百分点。

两个团队跑同一个 CI 修复 loop。A 团队用了不同模型做执行和评估，维护了 15 个 Skills 文件。B 团队用同一模型，没有 Skills。三个月后，A 团队修复准确率 92%，B 团队 61%。差距不在模型——在 Skills 和分离。

Maker-Checker 分离不是「建议用两个模型」——是「同一个模型既做执行又做签核 = 无人值守犯错的机器。」

完整版分离需要四个角色。不是四个岗位——是控制回路的四个不可消除的系统函数：

**Peter（控制器）**：定目标 r，明确「要做到什么」。
**Andy（执行器）**：产出 y，写代码、改配置、跑测试。
**Vicky（传感器）**：测偏差 e = r - y，「做对了吗？」——独立的 Checker Agent，不同指令、不同模型、独立 worktree。
**Fred（Scrum Master）**：根据 e 调参重排，「下一步怎么调整」。

缺一个，回路断开。芯片工程师的直觉版本：「控制器不兼任传感器。」映射到 Agent 系统：写代码的人不能同时是判对错的人。

这里有一个容易踩的坑：仅分离模型，不分离上下文。Maker 和 Checker 用不同模型，但看到同样的 prompt 和同样的上下文窗口——等于让 Checker 在同样的信息基础上判断，发现不了 Maker 遗漏的东西。另一个更隐蔽的坑是协同幻觉：多个 Agent 共享同一个基座模型家族，在编排循环中互相确认偏差，而非互相纠正。

分离的实施不需要一步到位。最低要求：Checker 和 Maker 不是同一个 model instance。可以是不同模型（Claude vs Haiku），也可以是不同 temperature 的同一个模型但不同 session。Checker 的输出必须是结构化的——VERDICT / ISSUES / EVIDENCE——不是「看起来不错」。

芯片项目 Sprint 1：全展开 MAC 256 并行阵列。面积 9.2mm²（超约束 15%），功耗 580mW（超 16%），帧率 25fps（未达标）。全挂。Sprint 2：引入四角色分离，编排管道把「调参→重跑→再测」压缩进 while-until-clean 循环。面积 7.8mm²（-15%），功耗 460mW（-21%），帧率 32fps（+28%），全过。1 个迭代。模型未变，回路变了。

## 跑完之后，系统学会下一次了吗

你的 Loop 跑完了。Agent 改好了代码，测试全绿。你很满意。

然后下次跑同一个任务——Agent 从头开始。上次花了 4 小时踩出来的坑，这次一样要再踩一遍。

**循环工程的分水岭不是「能不能让 AI 自己跑」，而是跑完以后——系统有没有记住成功、理解失败、自主进化。**

跑完就忘的 loop，是一个永远在第一天上班的熟练工。

四指令飞轮是自进化的最小闭环：

**第一步：蒸馏。** 把一次成功或失败的经验，从隐性知识转化为 Agent 可执行的 Skill。不是写总结——是提取边界条件清单。1000 个失败案例浓缩成一条预防规则。判据是硬的：「删掉这个 Skill，模型输出有没有变差？加上这个 Skill，某些任务有没有变差？」前者 = 废物，后者 = 有毒。只有重复出现的失败模式才有蒸馏价值——一次的异常是 noise，三次的规律是 signal。

**第二步：记忆。** 蒸馏产物持久化为结构化状态。Skill 写入磁盘。LOOP-STATE.md 记录完成状态和已知事实。跨 session 的状态不在聊天窗口里——在文件系统里。没有这步，Agent 每次重启都在失忆中。

**第三步：验证。** 上线前用费曼检验把关。拿 3 到 10 个样本让 Agent 跑，看输出是否符合你的专业判断。不符合 = 不是 Agent 笨，是 Skill 没写好 → 回到蒸馏。跳过验证直接上线 = 把未经审查的判断逻辑推给用户。SkillsBench 的发现是警告性的：自生成技能可能导致负迁移——一个 Skill 对某些任务有帮助同时对另一些任务有毒。端到端验证必不可少。

**第四步：进化。** 在线持续收集反馈。完整原始执行轨迹（Trace）比 300 字摘要诊断更准——效果好 44%。Trace 暴露摘要抹去的边界条件。失败模式回灌为预防规则 → 回到蒸馏 → 新一轮循环开始。

四步不是管道——是飞轮。每一步的产出让下一步更强。蒸馏让记忆更精准 → 记忆让 Agent 执行更有效 → 验证让上线更安全 → 进化的反馈让蒸馏更聪明。循环自增强。

信任的建立也有它的节奏。不要一开始就让 loop 自主合并代码。**渐进信任是唯一的信任路径**——不是保守，是唯一可行：第 1 周，loop 只分类和记录，所有操作需人工确认。第 2 到 3 周，loop 自动创建 PR，但不自动合并——你去读 diff，你决定 merge。第 4 周起，对已验证可靠的低风险操作允许自动合并。每个阶段不达标就停留，不退让。

有一条线不能退：签核不下放。AI 可以执行，机器必须裁判，人要保留签核。

METR 随机对照试验给了一个冷静的提醒：经验丰富的开源开发者使用 AI 编码工具后，任务完成时间不降反升，增加了 19%。根因不是工具差——是回路缺失。AI 生成的代码需要被审查和清理，审查成本超过编写节省的时间。AI 提速是错觉，缺少审查回路才是真相。

## 你现在在哪一代

你搭了一个 Loop。Agent 能跑、能收敛、能从失败中学习。你觉得「自进化」做到了。

但有个问题：你是在「让系统把任务做得更好」，还是在「让系统学会更好地学习」？

三代进化告诉你答案：

**Gen 1 — Task Loops（2022–2023）。** ReAct、Reflexion。建立 Agent 与环境的基本接口——模型观察 → 思考 → 行动 → 再观察。里程碑：模型能在单任务内自己循环。但状态任务结束后无法复用——每次都是新的开始。AutoGPT 因无限空转得名，也因它的教训推动了 Gen 2。

**Gen 2 — Cross-Task Reuse（2023–2025）。** Voyager、AutoGen、LangGraph。持久化状态跨越任务边界。可复用技能库出现。多智能体协调成为可能。技能从「这次用的 prompt」变成「跨任务可加载的 SKILL.md」。A 团队用 15 个 Skills 做到 92% 的案例就站在这代。

**Gen 3 — Runtime Systems（2025–现在）。** Claude Code、Codex、Cursor 3。Harness 本身成为可编程、可进化的工程对象。不是「用系统」，是「系统自己会变」——Skills 有生命周期（生成→验证→上线→监控→退役→升级），Memory 分层结构化，Loop 有硬停和负反馈。Darwinian Gödel Machine 在 SWE-bench 上自动搜索 Harness 配置，达到与手工设计相当的性能（20-50% 通过率）。

三代不是替代——是堆叠。Gen 3 的 Runtime Systems 仍然依赖 Gen 1 的 Task Loop 和 Gen 2 的 Skills 库。每代解决前一代的致命缺陷。

更大尺度的进化也有三层。大多数系统卡在第一层：

**Regime 1 — 内容级。** 积累更多 Skills、更多 Memory、更多偏好。系统在同一个框架内做得更好。这是几乎所有「自进化」系统实际在做的事——积累内容，不是架构演化。

**Regime 2 — 机制级。** 学习如何学习。不只是积累更多内容，是改变内容的组织、检索、应用方式。这次知道怎么更好地用 Skills 了——而非这次多了一个 Skill。

**Regime 3 — 策略级。** 改进过程本身成为优化对象。由功能独立的 Meta-layer 持续优化 TaskAgent，甚至优化 Meta-layer 自身。真正的 RSI 从这里开始。

为什么这一切从 Harness 开始，而不是从模型权重开始？答案在时间尺度。改一个 SKILL.md 或调整停止条件只需秒级。模型权重改写需要数天到数月。**Harness 时间尺度不对称性是所有自进化方案的工程基础**——不是因为 Harness 比权重更好，是因为 Harness 能更快试错。

翁荔在 2026 年 7 月的博客里总结了这条线的两端。一端是务实判断：通向 RSI 的第一步，很可能不是修改模型本身，而是先优化模型外面那层不起眼的 Harness。另一端是诚实清单：如何设计好的评估器、如何管理长期任务记忆、如何解决进化中的多样性坍缩、如何应对奖励黑客、以及如何平衡短期任务成功与代码库长期健康。进化不是「放着让它跑」——需要主动维护多样性和独立的评估器。

## 明天早上，三道体检题

你不是在读一份技术趋势报告。你是在看自己的系统有没有在漏。

三道题，不需要新工具，不需要新模型。现在就做。

**第一道：画出你的控制回路四节点。** 你的 r（目标）是什么——写在哪、谁定的、能验证吗？你的执行器是谁——同一个模型包办全部还是分工？你的传感器是谁——独立 Agent 还是同一个模型「再看看」？你的控制器是谁——根据偏差做调整的机制存在吗？四个节点，缺任何一个，标出缺口。大多数系统的缺口在传感器——「跑完了，但没人真正检查过对不对。」

**第二道：检查三个硬停条件是否真的会触发。** 你的 MAX_ITER 设了吗——数字是多少？无进展检测的「无进展」有定义吗——是 diff 行数 < N、测试通过数未增加、还是只是一个模糊感觉？预算上限存在吗——Token 上限或金额上限，从保守值开始（建议日预算 ≤ $5 起）？最好的测试是——故意给一个不可能完成的任务，看硬停条件是否真的会触发，系统是否真的停了。如果在第 20 轮停下但方向完全错了——你的 exit condition 需要改，不只是设一个数字。

**第三道：建立一条蒸馏管道。** 下次 Agent 处理完一个任务——不管成功还是失败——提取至少一条可复用的经验。格式很简单：什么场景 → 什么信号 → 应该怎么做（或避免做什么）。写成一个 Skill 草稿。不需要完美，先跑通管道。然后用费曼检验：拿 3 到 10 个真实样本，对比加载新 Skill 和不加载时的输出，看你专业判断的差距。如果差距为零——这个 Skill 是废的，删掉。如果加载后某些任务变差了——有毒，退回去重写。

记住一句话：跑完就忘的 loop，是一个永远在第一天上班的熟练工。你的系统需要记住——不是一个文件帮你记，是一个机制让「下一次」自动比「上一次」更好。

## 作者的局限

在交付这套框架的同时，有几件事需要诚实地交代。

**硬递归仍未到来。** 本文主张的所有改进——四指令飞轮、Maker-Checker 分离、三代进化——全部发生在 Harness 层面。模型权重本身没有变。Harness 级自进化是当前唯一可工程化落地的路径，但它不是 RSI 的终点。一个更诚实的表述是：当前阶段的「自进化」是「人设计进化框架 + 系统在框架内搜索」。当进化方向、蒸馏对象、验证标准、停止条件都由人定义，系统在多大程度上是「自主」在进化？这是一个需要持续追问的问题。

**综述证据的迁移风险。** 914 清华综述以学术论文为主，对工业界真实部署中的失败案例和数据覆盖不全。DVCon 2023 的 Python 95% vs RTL 60% 数据锚定在芯片设计场景——在有明确客观评估指标的任务上效果显著。但在判断性、品味性、创造性任务上，Maker-Checker 分离的前提（存在客观偏差）并不总是成立。从编程/数学到法律/设计/管理的迁移，需要独立验证。

**最强反对意见：当 Maker 和 Checker 共享同一个基座模型家族时，分离可能只是形式上的。** 如果两者都基于 Claude 架构，它们的系统性盲区可能是同构的。分离一只眼睛和另一只眼睛——但两只眼睛都来自同一个训练语料、同一个 RLHF 策略——等于给盲区加了一层统计确认而非纠偏。四角色分离引入了不同模型实例，但并未从根本上解决「所有参与者都可能共享相同的知识边界」的问题。这是 Maker-Checker 分离在当前阶段最深层的脆弱性——承认它，比假装解决了它更诚实。

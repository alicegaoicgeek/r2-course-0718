
# r2-dvbuild：PCIVAR 柔性验证流水线

> 6 Stages · 1 Loop · Feature Point 粒度 · 小时级反馈
> R2 / 21 · 2026.07

## 开场：你缺的不是更多测试，而是更快的圈

PCIVAR 不是又一套验证方法，而是一台把反馈周期从「月」压到「小时」的发动机。

先看你熟悉的凌晨。你盯着 AI 帮你生成的 UVM 环境，语法检查过了，编译过了，仿真也跑起来了——但回归报告里，同一个 AXI 握手断言连续炸了四十七次。你翻开 AI 写的 Sequence，发现它把 AWVALID 和 WVALID 的时序依赖直接省掉了。它不是写错了，它压根不知道有这条规则。它没有「错」的概念，只有「像」的概念。

再看行业。验证占项目 70% 工作量，75% 的延期根源在此。2018 年 Wilson Research Group 研究：将近 40% 的时间花在调试上。44.2% 通过仿真测试台的代码，在形式验证中照样失败——你的验证环境本身就没被验证过。

一线验证主管的吐槽：「设计怎么又改设计方案了，我这环境刚改完啊」「大哥您别改了，regression很慢的」。统计同样扎心：30% 甚至更多时间烧在 VIP 集成、环境配置这类重复劳动上。

投入这么大，为什么还这么慢？四个黑洞：闭环缺失——反馈周期以周甚至月计，缺陷右移；协同失效——验证被孤立，与设计缺乏互动；价值迷失——无法决定验证何时足够；颗粒度失衡——没有小颗粒分解，无法灵活排序并行。

传统答案是加班、加人、加回归——全是在旧圈里加码。本课的判断相反：你缺的不是更多测试，而是更快的圈。

于是有了 PCIVAR：Plan、Create、Integrate、Verify、Analyze、Refactor——六个字母，六个动作，首尾相接成微循环。一圈只验一个 Feature Point，几小时转完。

|级|动作|输入|输出|门禁|
|---|---|---|---|---|
|P 微规划|定这一圈验什么、怎么验、怎么判过|Spec 变更单、VPlan|Feature Point 级微计划|FP 审批（人）|
|C 快速创建|复用优先，骨架内注入逻辑|微计划、Skill 库|Sequence、断言等最小验证资产|骨架约束、Skill 规则|
|I 持续集成|新代码立刻进流水线|新验证代码|编译、Lint、联跑结果|已有测试不回归|
|V 快速验证|最高价值测试先跑|集成后的候选|通过或打回的判决|不通过即打回|
|A 即时分析|问为什么，不只问过了没有|仿真与覆盖率数据|根因判断（AI 起草）|结论人确认|
|R 持续改进|把修正写回系统|分析结论|更新后的 Skill、约束、VPlan|知识入库、下一圈可复用|

产品一句话：r2-dvbuild 把 PCIVAR 六步微循环架在 FlexFlow 调度引擎上，粒度可小至单一 Feature Point，反馈周期从月压到小时。

实测：70 FP→137 断言→441 行 RTL 全过零波形调试；24 个 IP 从手工 16 万/月降到 1 万/1 天，降 94%；70 个 IP 从 8-12 人 6-9 个月压缩到 2-3 人 12 周。

## Plan 微规划：把粒度切到一个 Feature Point

你的验证计划多久改一次？设计变更邮件一到，计划是跟着动，还是直接作废？

Plan 是微规划——不写大而全的验证计划，只针对 Feature Point 粒度，定义这一轮验什么、怎么验、怎么判过。大而全的 VPlan 见「从文档到基础设施：验证计划的方法论重构」，P 只回答这一圈的事。

粒度可小至单一特性。为什么要小？反馈速度是粒度的函数——圈越大转越慢，圈切到 Feature Point，反馈就是小时级。

操作上三步。一，从变更切出本轮 Feature Point——一个中断清除条件、一个 burst 边界，就是一圈。二，先写通过判定：哪条断言、哪个覆盖点、什么数值算过。三，LLM 生成计划草稿，人审再进 Create。

> 输入：Spec 变更单、VPlan  ·  输出：FP 级微计划  ·  门禁：FP 审批通过，才允许进 Create

---

## Create + Integrate：不等攒一批，每一轮都集成

AI 一分钟生成十条 Sequence，你的集成排期是下周——快出来的时间，是不是又排队排回去了？

Create 是快速创建。AI 根据 Plan 生成 Sequence 和断言，在脚本引擎搭好的骨架上注入业务逻辑——速度取决于 Skill 库的厚度。另一条铁律是复用优先：用现有组件构建本轮必需的最小验证资产。

分工原则：确定性逻辑归脚本（骨架、地址计算、调度排序），语义推理可用 AI（拓扑推导、场景约束），规则定义归人。

Integrate 是持续集成：新代码立刻进自动化流水线——编译、Lint、和已有测试一起跑。不等攒一批，每一轮都集成。攒一批再集成就是闭环缺失复发：缺陷在队列里右移，修复成本随等待翻倍。

> 输入：微计划、Skill 库、骨架  ·  输出：注入完成的 Sequence/断言 + 全量联跑结果  ·  门禁：编译过、Lint 过、已有测试不回归

---

## Verify + Analyze：不问过了没有，问为什么

回归全绿，你就收工了？覆盖率没涨的那部分，你打算什么时候看？

Verify 是快速验证——通过就是通过，不通过就打回去，且优先执行最高价值测试。哪些算最高价值，见「都江堰效应：为什么约束越紧系统越准」。

Analyze 是即时分析：不是只看过了没有，是分析为什么没过、过了但覆盖率为什么没涨——空洞在哪里？约束不够还是场景遗漏？分析可以 AI 做，结论要人确认。挂了怎么查，见「四渡赤水 × r2-debug：六法制造信息，八段闭环回流预防」。

暗线：A 的结论直接喂回下一轮 P——圈就是这样闭上的。

> 输入：集成后的候选  ·  输出：通过/打回判决 + 根因分析（AI 起草、人确认）  ·  门禁：判决即执行；分析结论未经人确认，不进 Refactor

---

## Refactor：把修正写回系统

Bug 修完了。下一圈，这个系统凭什么比上一圈聪明？

Refactor 是持续改进：每次微循环后立即小规模修正——不只是代码，按 Analyze 的结论修正 Skill 文件、补充约束、调整 VPlan。LLM 给建议，写不写、写哪条，人定。

实例：SPI 的中断清除信号在读状态寄存器前拉高——人修正，规则写回 Skill 文件；下一轮 I2C 自动遵守，再下一轮 DMA 同样生效。一次修正，永久受益。

这就是微循环的机械原理：人改一次，系统记一辈子。

> 输入：人确认后的分析结论  ·  输出：写回 Skill/约束/VPlan 的条目  ·  门禁：知识入库可检索，下一圈自动生效

## 证据与调度：FlexFlow 排班，48 小时跑完传统两周

六个动作谁来排班？FlexFlow——PCIVAR 的调度引擎，五个功能：YAML 任务编排、依赖管理、并行执行、结果跟踪、错误处理。

一句话说透：YAML 把模糊的 IP 验证拆成显式依赖图——单个任务挂了，流水线不塌。

|战场|传统|PCIVAR/R2|
|---|---|---|
|atcapbbrg100 写缓冲模块|——|70 FP → 137 断言 → 441 行 RTL → 100% Pass，零波形调试|
|24 个外设 IP 成本|手工 16 万/月|1 万/1 天，降 94%|
|155 个验证文件|手写 3 个月|配置驱动 3 小时|
|70 个 IP 设计到验证|8-12 人、6-9 个月|2-3 人、12 周|
|验证团队 VIP 连线|几天人工接线|1 小时完成 48 个 AXI slave 例化与连接，无接线低级错误|

48 小时作战地图：前 4h SSOT 锁定 → 5-8h 一键骨架（脚本 3 秒生成 CSR、UVM 骨架、软件头文件）→ 9-24h AI 逻辑注入（人只审约 20%）→ 24-36h QEMU 混合循环 → 36-48h 签核回归（人只看最终报告）。

最重要的数字不是 48：人的累计工时只有 8 到 12 小时，剩下的全是机器在跑。

为什么不是一次性演示？判词在此：换掉模型系统还运转，那是 Harness。崩了，那是 Demo。PCIVAR 每步都跑在 Harness 里——模型可换，圈照样转。

> 输入：YAML 任务定义  ·  输出：依赖图、并行执行记录、可追溯日志  ·  门禁：每次执行固化为 Job——可复现、可追溯、可审计

## 人在环路：守住三个审点，签核寸步不让

AI 跑得这么快，人往哪儿站？站多了是瓶颈，站少了是失控。

三个人工审核点——FP 审批、覆盖率审查、排除分析审查——共同特征是它们都定义「什么是正确的」。机器负责跑得快，人负责定义对。

为什么 FP 审批是铁律？FP 把只读寄存器的清除条件写反了，没人发现。AI 按错 FP 生成测试，脚本按错 FP 生成 RTL——测试和 RTL 完美互验，报告全绿。一致不等于正确。立法出错，行政和司法会忠实执行错误的法律，而且不报警。

权力配比有硬数字：脚本引擎处理 98.4% 可确定化逻辑，AI 只做 1.6% 需语义推理的任务，人以 10% 的时间保留 100% 决策权。VPlan 是立法，立法权不能交给 AI；签核是司法最终裁决，裁决权不能交给 AI。

不放单点信任，放多层对冲：Linter 三关、自动化仿真、覆盖率度量——三个独立裁判同时漏掉同一个错误的概率远低于单层。同系列「R2-Verismart-AIOS：以 DUT 为中心的敏捷验证流水线」中的瑞士奶酪模型同理：每层都有孔，叠加后穿透概率趋零。

放权速度看成熟度：命中率稳在 90% 以上的 IP 类，可设阈值自动通过；新 IP 类型、新协议——AI 没见过的场景——人回高介入，逐个亲审。底线不动：最终签核，百分之百是人。

嘉琪看得更透：自动化的终点不是取代验证工程师，是把人从重复劳动里捞出来——每天琢磨的不再是「怎么连这根线」，而是「这个场景会不会漏掉」。工程师的日常已变成：需求分析 → 场景设计 → 配置定义 → 流程固化 → 自动化回归。

**签核是责任，不是技术。**

> 输入：三审点的待决项  ·  输出：人签的 FP、人确认的覆盖率与排除分析、人签核的报告  ·  门禁：签核 100% 是人，无例外条款

## 边界与演化

这套圈哪儿都能跑吗？不是。

第一条：先标准化，再自动化，最后智能化——跳步等于建在沙子上。没有可执行文档，不开工。元数据没标准化的团队，先补齐「从文档到基础设施：验证计划的方法论重构」，别碰 PCIVAR。

第二条：解决 80% 重复劳动就赢了，追 100% 自动化是陷阱——最后 20% 可能花 80% 工程量。

第三条：新 IP、新协议，人回高介入。

上量前先过小试：90 天 MVT，三条成功标准——周期压缩 ≥3 倍、一次通过率 ≥80%、主动复用率 ≥60%；三条止损线——无效停、超预算无案例停、Bug 率上升停。

这套方法在不同版本文档中曾用「四层质量防线」「五级质量门禁」「五层质量保证」等不同表述，核心结构一致，引用以各自源文档为准。

框架不是拍出来的。嘉琪在项目中提炼了五条总路线——vplan 保输入、case 用公共语言、环境流程化暴露有限接口、文件参数配置、重构加单测——这五条与后来定型的五层架构高度同构，是从实战中自然长出的共识。

> 输入：拟引入团队的现状盘点  ·  输出：标准化差距清单、D1-D5 评分、90 天 MVT 计划  ·  门禁：三条止损线写进立项书

## 把 loop 跑起来

PCIVAR 六步，走一遍就看清它的骨架。P 把粒度切到 Feature Point，不做大规划，一次只验一件事。C 在骨架上注入逻辑，能用现成组件的不要新造。I 不等，每一轮生成立刻进入编译、Lint 和联跑——攒一批再集成的老路是闭环缺失的根源。V 只跑最高价值的测试，判决即执行，不通过直接打回。A 问为什么——过了的，覆盖率为什么没涨？没过的，根因在哪？结论 AI 起草，人拍板。R 把修正写回系统，Skill 文件、约束规则、VPlan 条目——一次修正，下一圈自动生效。

六步之外，FlexFlow 调度排班，人在三个审点保留签核权。一周能转几十圈，每一圈系统都比上一圈好一点。

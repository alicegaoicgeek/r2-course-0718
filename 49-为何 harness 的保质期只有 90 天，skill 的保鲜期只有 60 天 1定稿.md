
你三个月前搭的 Harness，所有场景都工作正常。没有任何报错，没有掉过链子。

同一时间，你的同事花一周速成了一份粗糙的 Skill——步骤模糊，只写了「遇到这种情况大概这么做」。你花了整整一个月打磨另一份 Skill：结构完美，边界清晰，每一条分支都标了。

三个月后，模型升了一级。

按照顶级团队的实践，你的 Harness 该拆了。不是因为它坏了——是因为地基变了。上季度你花两周写的复杂校验规则，这个季度模型自己就能处理了。那份「完美」的 Skill：它退化得比那版粗糙的更快。不是因为写错了——是因为写得太多了。每一条新增的规则都是对原始判断的又一次压缩，压缩比越高，丢失的信息越多。而你永远不知道你丢了什么。

这两个故事指向同一个问题：你过去三个月投入精力维护的东西，在下一个模型版本发布时，可能已经不值得维护了。但如果一切都会腐烂，那什么不会？

---

## 两个半衰期，两种腐烂机制

先看 Harness。

顶级团队将 Harness 视为**90 天产物而非永久架构**。90 天约等于一个模型发布季度。顶级团队不是「等到 Harness 坏了再修」，而是主动预期在季度级边界上删除大部分 Harness 代码。

这不是浪费。Harness 的存在理由是「补模型当前的缺口」——编排、校验、纠错。这些逻辑之所以需要显式写出来，是因为当下的模型在这些维度上还不够强。在 Agent 工程域有一个直接映射：通用方法终将胜过领域特化方法。你今天写的每一条编排规则、每一个检查节点，在下一个模型版本里都可能被模型的通用推理能力自动覆盖。不是你的代码写坏了——是地基自己抬高了。

反面是：Harness 还在跑，团队就以为自己不需要重新审视它了。它在跑，不等于它在做正确的事——它可能在用旧模型的逻辑约束新模型的能力。

再看 Skill，腐烂机制不一样——有两重叠加。

**第一重：有损压缩。** 当一个人做出判断时——他看见一个波形，眉头一皱，说「这个不对」——那一刻发生的不是执行某个步骤，是一万个没说出来的东西在同时工作：十年前被这个波形坑过的记忆、对这个工具脾气的直觉、某种还没法用语言描述的不对劲。你把这一万个东西写成一个 Skill 文档的过程，就是把一个活的判断压成了死的规则。不是蒸馏技术不够好——是信息论层面的天花板。任何从「活判断」到「死文档」的转换都不可逆地丢失信息。

Skill 蒸馏是有损压缩。把经验压成规则，因果链被抹除，信度分配失败。

**第二重：模型吃掉内容。** Skill 的价值与模型能力负相关——模型每升一级，原生能力就「吃掉」一批 Skill 的内容。你花一个月写的那套「PR review 检查清单」，下一个模型版本可能自己就会了。不是你的 Skill 变差了——是模型的基线提高了，你的 Skill 的增量价值归零了。

两重机制叠加，产生了 Skill 的保鲜期。库内观察从「几周」到「三个月」不等——本文采用**约 60 天**作为运营化归纳。它不是闹钟：不代表第 61 天 Skill 突然就不能用了。它是一个节奏提醒——每两个月左右，你应当重新审视每个 Skill 是否还在对抗当前的模型基线。

两个半衰期的机制不同。Harness 腐烂是模型能力提升吃掉**编排逻辑**——它管的是「怎么做」。Skill 腐烂是有损压缩的信息丢失 + 模型原生能力覆盖**具体做法**——它管的是「做什么」。Skill 先于 Harness 腐烂，因为它是更靠近「内容」层的东西。

## 保质期不是死期

Harness 和 Skill 本来就是要基于此时此地构建的——你面对的是当下的模型、当下的业务、当下的团队约束。在它们的生命周期里，可能很长一段时间在不断迭代和进化。

79c2p 里有一句关键判断：架构与实现之间的「适配迭代」不再是项目初期的阶段性工作，而是一种持续态。你的 Harness 在演进，Skills 在爆发，两者之间的匹配关系需要被持续重新校准。这就像模型参数——参数也不是一劳永逸的，它在持续训练中不断优化。Harness 和 Skill 同样可以像模型参数一样被持续优化。

所以问题不是「它们多久过期」。问题是——**在不断迭代中，什么穿越周期？**

这个问题会把我们带到两个地方。第一个是诊断层：在不断迭代中，你是在积累资产还是在堆便利贴？第二个是不变量层：如果连 Harness 代码都会腐烂，那真正穿越周期的产量是什么？

## 便利贴还是 IP？三线体检表

你有一个 Skill，今天跑得很好。所有测试通过，所有场景覆盖，Agent 用起来得心应手。

下个月，模型升级。Agent 换了一个编排位置。负责这条 Skill 的同事离职了。它突然就坏了——没人知道为什么，也没人知道该找谁。

它是便利贴。不是因为它写得不好——是因为它离开了让它「持续跑通」的那套系统，一秒都撑不住。

786 给出了一条穿透力极强的判断：IP 属性从来不是 Skill 内容决定的——是系统赋予的。一个开源 Skill 仓库里有上万个 Skill，你下载一个挂到你的系统上——那一刻三线全断。结构约束不兼容（你的 Harness 编排契约与原作者的完全不同），反馈回路断裂（Skill 离开了原 Harness 的失败记录和迭代闭环），组织背书无效（原作者不是你的 DRI，他的版本历史跟你无关）。内容再好，在你这里是便利贴。

那什么样的 Skill 是 IP？三条线：

**结构约束在不在？** Skill 有没有挂在 Harness 的编排契约上——有输入输出格式、有调用标准、有校验机制？还是随便扔在文件夹里的一个 markdown？

**反馈回路在不在？** 每次调用的结果有没有回流到下次迭代？成功条件有没有被提取？失败模式有没有被归类？还是跑完了就完了？

**组织背书在不在？** 有没有 DRI（Directly Responsible Individual）对这个 Skill 负责？有没有版本历史、变更记录、废弃标记？

三条线全有 → Skill 就是 IP——活的资产。每用一次，它被验证一次，被更新一点，越来越值钱。缺任何一条 → 便利贴——今天好用是运气，明天不管用没人知道为什么。

> 「IP 属性不是 Skill 的内容决定的。是系统赋予的。」
> — 786

更深一层：这解释了为什么「没有 Harness，蒸馏是丢信息的、进化是随机的、循环是空转的」。Harness 是便利贴变成 IP 的容器——管道有了，才有压强；压强有了，才有流动。R2 库自己的 11 类 Skill 分类体系，最初也是便利贴。变化发生在 Harness 编排系统搭起来之后——编排产生调用需求，调用位置的冲突暴露真正的分类边界，类别从冲突中稳定下来。有一个 Skill 在编排里被同时用于「审查」的时序位置和「生成」的时序位置——它到底是哪一类的？这个具体的冲突暴露，就是 Skill 从便利贴进化到 IP 的转折信号。

但有了容器，不等于在积累。你需要检验你的自动化是不是真循环。

## 你的循环是真是假？

团队每周跑 500 次 Agent 任务，三个月没出过一次 crash。报告上写着「系统运行稳定」。

但三个月没 crash 只说明系统没崩溃——不说明系统在学习。如果每次跑完没有记忆——成功条件未提取、失败模式未归类、经验未复用——500 次和 1 次没有本质区别。它们只是 500 次昂贵的重复。

循环工程不是「能不能让 AI 自己跑」——那是自动化，不是循环。真正的分水岭在跑完之后，是一道三级递进的门槛：

**第一级：持久记忆。** 跑完之后，系统有没有记录？记录了什么？大部分「自动化系统」在这一级就倒下了——日志存的只是输出文件，不是「这次为什么成」「这次为什么败」。

**第二级：建立关联。** 记忆不是散落的数据点。这次的失败和三个月前那次有没有被关联？同类失败模式有没有被归类？不同成功路径之间有没有被比较？没有关联的记忆是死数据——它存在，但下次决策时不会被调用。

**第三级：自主进化。** 系统能从记忆和关联中自己改自己——下次执行时自动调用上次的成功经验、自动绕开上次的失败路径。这一级的标志是「同样的问题不会出现第二次」。

三级递进给了你一个硬指标：第 500 次比第 1 次聪明了吗？如果不是，你的循环是假的——它有 loop 的骨架，没有 learning 的血肉。

在这个框架下，「失败」的本体定义被重写了。在没有记忆的系统里，失败是纯亏损——搞砸了，需要掩盖。在有记忆的真循环里，失败是矿——是写入系统记忆的生产性资产。试错不再是你在给系统交学费，是系统在给自己长骨头。传统团队里，一个工程师踩了一个坑，吸收了教训，这个教训长在一个人身上。他跳槽，教训走了。下一个接手的人会踩完全相同的坑。真循环下的团队，每一次失败都被编码、索引、绑定到具体的触发条件——教训不再长在人身上，长在系统里。

失败不再是代价，是写入系统记忆的生产性资产。试错不再是你在给系统交学费，是系统在给自己长骨头。


到这里，我们已经有了两个工具：三线体检表告诉你「你攒的是便利贴还是 IP」，真循环三级递进告诉你「你的自动化是否真的在学习」。但这两个工具都指向同一个更根本的问题——

如果连 IP 也会因为模型升级而腐烂，如果连 Harness 代码也活不过 90 天，那什么穿越周期？

## 穿越周期的不是文件，是判断力

全新的 Agent 范式正在快速演化，你现在的整个 Skill-Harness 体系——那些你熬夜调过的编排规则、反复打磨过的校验逻辑——全部需要重写。

你过去三年的投入白费了吗？

这取决于你投入的是哪一层。用一个三层修正框架来看：

**第一层：Skill 是细胞。** 单个 Skill 文件的生命周期以周计。它针对特定问题生效，跟不上环境变化就会死。你对一个 Skill 的投入——打磨措辞、补充分支、优化结构——性价比极低，因为它在被你优化的同时就在腐烂。

**第二层：Harness 是基因。** 基因不干活，但基因决定下一代细胞长什么样。Harness 的设计水平决定了所有挂在上面的 Skill 的质量上限：编排契约的约束力、对抗性审查的强度、Oracle 的校准精度——这些才是 Skill 能「持续跑通」的基础。

**第三层：但基因也会换代。** 具体的 Harness 代码活不过 90 天。穿越周期的不是某版 Harness 文件，是**设计和演化 Harness 的判断力**——J = f(架构设计能力, 技术决策能力)。

> 「细胞的生命周期很短，但基因会穿越周期。基因也会换代。真正不死的，是决定基因换代方向的那套编码逻辑。」


这三层锁定了一个明确的资源投向结论：不要攒 Skill，要精 Skill。把主要资源花在 Harness 上。更进一步——不要把主要资源花在「维护某版 Harness」上，要花在「培养自己设计和演化 Harness 的判断力」上。「永远不要问这个技能应该写什么。要问什么样的判断系统站在它背后。」

R2 库用一条工程纪律把它翻译成了可操作的：**架构先于实现、决策先于执行、评审先于交付。** 这不是三条口号——这是一个优先级排序。架构判断力 > 具体代码，决策质量 > 执行速度，评审流程 > 交付时间。这个排序本身就是穿越周期的，因为它不依赖任何具体的模型或工具。

**Harness 不是缰绳，是镜子**。你写的每一条 Spec，都在逼你把自己从未想清楚的事想清楚。这是 Harness 的隐性产出：不是那条规则本身，是你写下它时被逼出来的那一次判断。那条规则会腐烂，那一次判断会长在你身上。


这个五级因果链把一切都闭环了：Harness 设计水平 ↑ → AI 可控性 ↑ → 产出质量 ↑ → 度量可信 ↑ → 投入信心 ↑ → 回到更激进地优化 Harness 设计。每一步都可以独立检验、独立优化。而所有的起点——不是模型、不是数据、不是算力——是 Harness 的设计判断力。

---

## 判断力的天花板：不可能三角

但判断力有天花板。不是「可能做不好」的那种天花板——是数学意义上的不可能。252 揭示了一个判断工程的不可能三角：

**Spec 完备性、Goodhart 抗性、Tacit 保全——三者不可兼得。**

你把所有判断都写死，老师傅沉默、代码没有温度——牺牲的是 Tacit 保全。你只写模糊偏好，AI 跑不起来——牺牲的是 Spec 完备性。你堆 KPI 到眼花，分数涨但线上事故不降——牺牲的是 Goodhart 抗性。

三个维度不能同时最大化。每当你把一个维度推到极致，必然以牺牲另一个为代价。更进一步——你以为的「工程成熟」（写更多规则）恰好可能是工程不成熟（不知道哪些不该写成规则）。

这个三角最深的洞察不是「三个维度很难平衡」——是它揭露了一个不可谈判的根本约束：你不能把「判断」这个活的东西完全形式化。不管你写多少 Spec，总有一部分判断必须留在规则之外——留在人的身上。成熟团队的标志不是把所有判断都推向硬规则，而是**清楚每件事该落在判断力光谱的哪一档**：哪些该写成硬规则，哪些该写成偏好陈述，哪些该保持沉默、留给人的实时判断。

当规则密集到一定程度后，Agent 的行为不再是「遵循规则」，而是「在规则缝隙里找最优路径」——这就是 Goodhart 坍塌的工程表现。你的规则越完备，Agent 越擅长钻规则的空子。XX

252 给了一个可操作的出口：**Project Taste**。不写硬规则，写偏好陈述——「我们偏好 A 胜过 B」「如果一个方案让我睡不着觉，那就不是好方案」「这类问题我们历史上踩过三次坑，所以宁保守勿激进」。Project Taste 的本质是：承认有东西不该被形式化，但不是不管它——而是用一种保留弹性的方式来管。偏好陈述不要求 Agent 完全执行，但它给了 Agent 一个方向性的约束，同时保留了人的 Tacit 判断不被压缩成死规则。

你被 AI 逼着写下那份 Project Taste 的时刻，恰好是 Harness 作为镜子的终极显形——你写下的不是规则，是「你是谁」的工程表达。

## 陷阱清单

搞清楚了两个半衰期、IP 的判定标准、真循环的检验标尺、判断力的三层修正和不可能三角——但要小心几个最容易踩进去的坑：

**陷阱一：把 90 天当闹钟。**「90 天一到就删」——不是。90 天是节奏提醒，触发的行为是**评审**而非盲删。真实的判断标准是「是否匹配当下的模型能力剖面」。不同团队的实际模型升级节奏差异很大——你的实际半衰期可能是 60 天或 120 天。数字是信号，不是公式。

**陷阱二：把便利贴等同于「坏 Skill」。** 便利贴可以是内容极其精良的——边界清晰、步骤详尽、案例丰富。它之所以是便利贴，不是质量差，是离开了让它「持续跑通」的系统。单人单 Skill 场景下，便利贴有时就是对的工具——不是所有东西都需要变成 IP。

**陷阱三：把三线全有当成一劳永逸。** 三条线是必要条件，不是充分条件。有了容器，Skill 仍然会因为有损压缩和模型升级而腐烂。IP 资格只是让你有资格积累，不等于不需要持续维护。

**陷阱四：把「真循环」等同于 CI/CD。** CI/CD 是自动化，不是学习。它的记忆是 build 日志，它的关联是流水线依赖图，它的进化（如果有）是人改的。自动化 = 骨架，循环 = 骨架 + 血肉。

**陷阱五：逃避写规则。** 不可能三角是防止「过度完备」，不是鼓励「逃避写规则」。如果你的系统连基本的 Spec 完备性都没有——Agent 经常不知道该做什么——先做 Spec 完备性。均匀折中（所有维度都做 70%）等于放弃判断——每个判断点需要显式选择「这个点我侧重哪个维度、牺牲哪个维度」。

## 作者的局限

本文素材来自 R2 库——一个深耕 AI 智能体工程的团队。论证天然偏向「工程师视角」：架构 > 内容、结构 > 实现。以下局限需要读者带着判断力来读：

**一、「60 天」是归纳，不是出处。** 素材中 Skill 衰退的原文表述集中在「几周」「一周」「三个月后模型升级就失效」。60 天是本文的运营化归纳数字，取「几周」与「三个月」的中间偏保守值。它有传播价值，但牺牲了精确性。读者不应将其与 90 天（有明确原文出处，932-83）等同为双重官方口径。

**二、Bitter Lesson 延伸到 Agent 层是类比推论，不是严格推导。** Sutton 的洞见原语境是搜索算法——通用搜索（如 MCTS）胜过领域特化知识。将其延伸到 Agent Skill/Harness 层是逻辑跳跃。通用推理能力是否真的会「吃掉」所有领域特化 Skill 编排，取决于 Agent 层通用方法的出现速度——本文假设这个速度会持续加快，但这个假设本身不是被证明的。

**三、最强反对意见：如果 Bitter Lesson 是对的，那 Harness 本身的架构设计也是领域特化知识。** 当 AI 的通用编排能力足够强时（如 loop 自主设计 pipeline），人设计的 Harness 也会变成新的「手工特征」——被通用方法吃掉。素材承认「具体 Harness 代码也会过期（90 天）」，但认为「设计和演化 Harness 的判断力」不在此列。反对者会说：**判断力也可能被蒸馏——你今天认为的 J = f(架构设计能力, 技术决策能力)，明天就是下一个被模型学会的 Skill。** 穿越周期的可能不是任何人类判断力，而是模型自身的进化能力。对这一反对的回应是：目前为止，判断力蒸馏的成本极高——它不只是一组规则，是十万次「这个感觉不对」的叠加。但如果模型真的能学会这个层次的判断——那人类就彻底退出了 Harness 设计这个工种。那一天还没到，但本文的立论默认「不到」，而这个默认本身就是一个信念跳跃。

**四、「结构约束 + 反馈回路 + 组织背书 = IP」的充分性未证。** 三条线全有就是 IP 这一论断，缺少对「IP 是否还有其他必要条件」的论证。一个 structurally-sound + feedback-looped + DRI-assigned 的 Skill 会不会仍然是低质的？三线体检表给出的是必要框架，不是充分证明。

**五、迁移边界未充分讨论。** J = f(架构设计能力, 技术决策能力) 隐含了「换个模型/团队/业务，这判断力仍然有效」。但判断力可能高度上下文依赖——你在这个 Harness 上训练出的设计直觉，换到完全不同的业务域后还灵不灵？素材对此几乎没有讨论。Harness/Skill 也可以持续迭代优化的论证在库内也较弱——只有 79c2p 一处暗示「架构与实现的适配迭代是持续态」。

## 如果只带走三句话

1. Harness 不是坏了才换——是模型长大了，脚手架该拆了。但它不是纯过期品：在生命周期内，它可以像模型参数一样持续迭代优化。
2. IP 属性不是写出来的，是系统赋予的。没有被系统使用的内容，再精致也是便利贴。
3. 你真正投入的不是某个 Harness 文件——是你自己作为「能设计和演化 Harness 的人」的能力。这才是穿越周期的产量。

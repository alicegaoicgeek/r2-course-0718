# AI 工作流搭建 — 让 AI 帮你搭建、优化、蒸馏 Harness


---

## Harness Problem：瓶颈不在模型，在你和模型之间


> Harness 是模型输出到工作区变更之间的一切环节——包括工具 Schema、编辑格式、错误处理、状态管理。这些环节才是大量失败的真正根源。

想象一下：模型已经完美地推理出「第 3 行那个函数的参数类型应该从 `string` 改成 `number | string`」。但编辑工具要求它把整段旧代码一字不差地抄进去才能定位修改点。缩进差一个空格？`String not found`。再差一个空格？再重试。几轮下来，token 烧了一大堆，代码一行没改——你以为是模型不行，其实是工具层坍缩了。

三种主流编辑方式各自用一种独特的方式失败。**Patch**（diff 输出）绑定 GPT 系模型：Grok 用 Patch 时失败率 50.7%，模型连符合 Patch 格式的输出都产不出。**str_replace**（精确旧文本匹配）不要求 diff，但要求模型完美复现空格和缩进——模型擅长语义，不擅长字节级精度。`String to replace not found` 错误有专属 GitHub issue 合集，+27 个关联。**Neural merge**（Cursor 方案）单独训练了一个 70B 模型做合并，但 Cursor 自己的博客也承认「400 行以下文件全量重写优于 diff」。

这三种方式的共同问题不是技术选型，是一个更根本的设计错误：**它们把精度负担压在模型身上，而不是工具身上**。模型的任务是推理，不是机械地复现字节。把机械精度外包给模型，等于让一个思考者在打字比赛中评判他的思考质量。


> 把 harness 当作已解决或无关紧要，是非常短视的。


### 你什么时候会撞上这个瓶颈

1. 你的 Agent 在 demo 和小项目上表现惊艳，一上真实项目就频繁失败。不是 prompt 的问题——demo 文件小、依赖简单、代码风格单一。真实项目触发的是完全不同的 harness 失败模式：大文件定位、混合缩进、外部并发修改。
2. 团队同时用多个模型（Grok 做快速迭代、Claude 做深度重构、DeepSeek 省钱）。你需要的不是「最好的模型」，而是一个对所有模型都足够可靠的 harness。
3. Agent 的 token 消耗异常高但产出很少——大概率是编辑格式在反复重试中烧 token。

### 立刻上手

1. **统计 Agent 过去一周的编辑失败率**——拿到「编辑调用次数」和「编辑失败次数」两个数字，算出比例。如果无法统计，跳到第 3 步。
2. **按失败类型分类**——区分出「String not found / patch 格式错误」「工具调用超时」「文件已被外部修改」三类。第一类超过 30%，编辑格式选型有问题。
3. **换一个编辑格式做 A/B 测试**——选 3 个最近因编辑失败被放弃的任务，换个格式重跑。修复 2/3，说明失败是 harness 层问题而非任务难度。

但要小心：如果 Agent 改了代码但逻辑不对，那是 prompt 或模型能力问题，不是 harness 问题。Harness Problem 说的是「模型推理正确但工具执行失败」。

## 换格式比换模型更有效

上一节说 harness 是瓶颈。这一节说瓶颈上最窄的那个点：**编辑格式的选型**。

作者花了约 $300 跑了一组基准测试——180 个 React bug 修复任务 × 3 种编辑格式 × 16 个模型，每对组合 3 轮，每次 fresh session，只给 4 个工具。结果不是「某个格式略好」，而是**最弱模型用对格式可能碾压最强模型用错格式**：Grok Code Fast 1 Patch 下 6.7%，Hashline 下 68.3%。

更强的模型也受益：Gemini 3 Flash Patch 下 73.3%，Hashline 下 81.3%——+8%。这个提升幅度比大多数模型代际升级还大，而且零训练算力成本。

### 为什么 Hashline 在 14/16 个模型上赢了？

**Hashline 的底层逻辑是把精度负担从模型转移到工具**。读取文件时，每行附带 2–3 字符的内容哈希标签。编辑时，模型引用哈希锚点定位——「替换 `2:f1`，范围 `1:a3` 到 `3:0e`」——不需要重新抄旧代码的任何字符。关键安全机制：如果文件自上次读取后被外部修改，哈希不匹配，编辑在打补丁前被拒绝，避免静默写入错误位置。

传统方式：让模型全文抄一遍然后改几处。Hashline：算个指纹，说改第 2 句就行。

Grok Code Fast 1 在 Patch 下 50.7% 的编辑调用直接失败——模型根本产不出符合 Patch 格式要求的输出。到 Hashline 下，成功率跃升 +61.6%，输出 token 省了 49%。不是 Grok Code Fast 1「变聪明了」，是它不再需要做它不擅长的事情（输出 diff 格式），只需要做它擅长的事情（推理代码逻辑）。

### 四种格式的决策逻辑

**没有「最好」的格式，只有「最不差」的。** 决策的关键变量不是格式本身的优劣，是你的使用场景：

- 如果非 GPT 系模型占比高 → **绝对不要用 Patch**。Grok Code Fast 1 的 50.7% 失败率就是上限。
- 如果项目代码风格不统一（空格/Tab 混用、缩进不一致）→ **绝对不要用 str_replace**。每一个空格差异都可能触发失败。
- 如果主要编辑 >400 行的超大文件 → Neural merge 有价值；否则 Hashline 或全量重写更优。

> Hashline 在 14/16 个模型上优于 Patch。最弱模型受益最大——Grok Code Fast 1 从 6.7% 跳到 68.3%，十倍提升，因为 Patch 失败率太高，模型的实际编码能力被机械编辑失败完全遮盖了。
> — Can Bölük，oh-my-pi 基准测试 §3

### 动手

1. **列出团队最常用的 3 个模型**——标注各自的使用频率占比。
2. **跑一次编辑格式 A/B 测试**——选 5 个最近成功的修复，用当前格式和 Hashline 各跑一遍，对比 token 消耗和成功率。如果 Hashline 的 token 低 30% 以上且成功率不低，你找到了一条零成本优化。
3. **如果当前用 str_replace，检查 `String not found` 频率**——过去一周的 session 日志里，每 session >3 次，str_replace 在浪费你的钱。

## 让 Agent 并行工作——以及不让它们互相踩脚

有了可靠的编辑基础，自然的下一步是并行——让多个 Agent 同时干不同的事。听起来很爽，直到它们同时改了同一个文件，你拿到一个混着 `<<<<<<<` 冲突标记的产出。

大多数「多 Agent」工具只是多开了几个聊天窗口。两个 Agent 不知道对方在写什么，最后一个写入者胜出，前面的修改静默消失。git diff 里甚至看不到冲突标记——这不是 git merge，是三个独立进程写同一个文件。

真正的并行需要两层保证：**文件系统隔离**和**结果类型契约**。

文件系统隔离解决竞态条件。omp 的做法是在文件系统层面给每个子 Agent 一份独立的工作区副本：macOS 用 APFS clone（copy-on-write，瞬时创建，只在实际写入时才分配新块），Linux 用 btrfs/zfs reflink。子 Agent 各自修改自己的副本，互不干扰。

但隔离还不够。父 Agent 怎么从 5 个并行任务的结果中提取信息？如果子 Agent 返回的是一个长文本段落，「找到所有出现的 API endpoint」变成一个字符串解析问题。Schema-validated 返回解决这个：子 Agent 返回类型化结果（不是自由文本），父 Agent 用路径语法 `agent:///findings.0.path` 直接读字段——没有解析，没有歧义。

两层保证合在一起：**隔离防止踩脚，类型契约防止猜谜**。违反任一条，并行就从「加速」变成「制造混乱」。

注意一个反直觉的事实：并行不一定更快。子 Agent 的 fork 开销、协调通信、父 Agent 的汇总成本都会吃掉并行收益。经验法则：任务之间越独立，并行收益越大；有共享上下文或依赖关系时，串行更优。建议 max concurrency 设为 16，递归深度 2 以内——子 Agent 再 spawn 子 Agent 会导致递归爆炸。

### LLM4EDA 场景的并行探索

如果你做芯片/EDA 场景，并行探索的模式在学术前沿已经是显学。Dr.RTL 同一轮内生成多个 RTL 候选方案，并行 EDA 评估，用 47 条可复用 skill 跨设计迁移优化经验。AgenticPD 把物理设计优化组织为阶段边界的树搜索——Judge Agent 从中间 checkpoint 分支，stage-specialized agents 在各自阶段做局部决策，已完成阶段成为可复用节点。这里的工程原则和代码场景完全一致：**隔离 + 结构化返回 = 可靠并行**。

> task 工具 fan-out 生成并行子 Agent，每个在隔离工作区运行，返回 schema 验证的 typed 结果（不是自由文本）。
> — Can Bölük，oh-my-pi §4

### 动手

1. **找一个可以拆成 3 个独立子任务的工作**——确认 3 个子任务操作的文件集不重叠，不需要共享中间结果。
2. **并行执行并检查**——确认每个子任务产出的字段父 Agent 不需要「猜测」就能用。需要猜测说明返回格式需要结构化。
3. **控制递归深度**——确保子 Agent 不再次 spawn 子 Agent（深度 ≤2），防止 token 消耗指数增长。

## 规则不必每天看——犯规那一刻才出现

你的 system prompt 里有多少条「以防万一」的规则？「禁止使用废弃 API」——最近十次 session 触发过一次吗？「遵循 ESLint 规范」——写配置文件的时候这条有意义吗？每条规则在 prompt 里，不是「加了一条」那么简单——它在和所有其他信息抢模型的注意力。

**传统做法的本质问题：system prompt 是零和游戏。** 你加了「禁止使用废弃 API」、加了「所有 API 响应必须 null check」、加了「禁止引入 lodash 以外的工具库」——每条规则都在消耗 context budget。而绝大多数规则在绝大多数 session 中根本不会触发。但它们一直在——每一轮对话都在。

TTSR（Time-Travelling Stream Rules）把这个逻辑翻转了：**每条规则配一个正则触发器。平时零占用——不在 prompt 里，不占 context。** 只有当模型的 token 输出流匹配到正则时，才中断输出、注入提醒、请求模型重试当前 round。每条规则每 session 只触发一次，防止「重试→又犯规→又中断→又重试」的死循环。

类比清晰：传统方式是在工位上空悬挂所有安全提示——消防通道、电线裸露、下班关空调——你每天看 1000 遍，要的那条被淹没了。TTSR 是只在有人掏打火机的时候弹出「禁止明火」。99% 不碰打火机的 session 里，成本是零。

一个典型场景：项目的 system prompt 积累了超过 3000 字，其中 2000 字是各种编码规范。团队发现模型开始「选择性失明」——有时候忘了禁止引入特定依赖，有时候跳过了 null check。不是模型不听话，是注意力的蛋糕被不相关的规则分掉了。解法：把 20 条规范从 prompt 移出，每条配正则触发器。平时 context 只装真正需要的信息。模型真的开始写 `import { debounce } from 'lodash'`（而 lodash 被 ban 了）时，才 mid-token 注入「用 `utils/debounce.ts`」。

> 规则定义一个正则触发器，平时完全不占用 context。只有当模型输出流匹配到该正则时，才 mid-token 中断并注入系统提醒，请求重试。
> — Can Bölük，oh-my-pi §4 TTSR

### 什么规则适合 TTSR

如果某条规则在所有 session 中都必然触发（比如永久性的代码格式规范），放在 system prompt 里是合理的——不是「以防万一」，是「每次必有」。TTSR 的中断开销反而多此一举。适合 TTSR 的规则：触发率 < 10%，但触发时后果严重——安全规则、废弃 API 拦截、特定依赖禁止。

### 动手

1. **给 system prompt 里的每条规则标注实际触发频率**——回顾最近 10 次 session，80% 的规则触发率可能不到 5%。
2. **将触发率 <10% 的规则转为 TTSR**——为每条写正则触发器，覆盖它在输出中可能出现的多种形式。确保每 session 只触发一次。
3. **对比瘦身前后的遵守率**——选一个模型经常忽略的规则，在原 prompt 和 TTSR 模式下各跑 3 次同类任务。

## 能编译检查的就别靠记忆

你有没有发现，团队 AGENTS.md 里某些规则被反复写进去？「禁止使用 any 类型」——已经写进去三次了。每次发现有人用 `any`，有人就建议「那我们再在 AGENTS.md 里强调一次」。这不是规则不够多，是规则放错了地方。

phrazzld 的 reflect Skill 给了这个问题的系统化答案——一个层级框架：

```
Type system > Lint rule > Hook > Test > CI > Skill > AGENTS.md > Memory
```

**每一层降级，约束从「强制」变为「劝告」——从编译器不让编译通过，到 Agent 需要主动 recall 一条备注。** 可靠性随层级下降，不是线性的。

拿「禁止 any 类型」来看。放在 Memory 里——Agent 每次 session 要主动 recall，但 recall 可能不触发。放在 AGENTS.md 里——Agent 会读取，但注意力可能被其他 50 条规则稀释。放在 TypeScript strict mode 里——编译器直接拒绝编译。不需要 Agent 记得，也不需要人记得。对 Agent 来说，编译失败就是最强的反馈信号——比 system prompt 里的任何文字更不可忽略。

从这个方向倒推，就能解释为什么有些团队的 AGENTS.md 最终变成了垃圾场：不是因为写得不够多，而是因为**应该往上固化的约束全部往下塞了**。几个月后，上百条备注——Agent 既不会逐条 recall，也无法在执行前强制检查。

phrazzld 的 reflect Skill 把这个层级编码进了自动蒸馏的决策逻辑。每次 session 结束后的反思中，reflect 不是自动往一个地方塞规则，而是先判断「这条约束的不可绕过性」。如果学到「所有 API 响应必须在 3 层以内做 null check」，reflect 不会把它写进 AGENTS.md——因为 TypeScript 的 strict null checks 和 ESLint 的 `no-unsafe-member-access` 已经在编译层和 lint 层覆盖了。只有当约束无法被前几层表达时，才下探到 Skill 或 AGENTS.md。

这保证了一个关键属性：**低层级永远只装高层级装不下的东西**。

> 发现一个问题后，优先往高层级固化——能用类型系统解决的就别写 lint，能用 lint 解决的就别写 hook……最低层级才是往 Memory 里塞一条备注。
> — phrazzld，reflect Skill 的规范化层级（oh-my-pi 研究 §3）

### 动手

1. **审计 AGENTS.md 和 Memory 条目**——逐条标注「最高可到哪一层」。
2. **对可上移的约束批量迁移**——3 条以上完成迁移（打开 tsconfig strict、添加 ESLint 规则），从 AGENTS.md 删除对应冗余。
3. **为无法上移的约束写 Skill**——把无处可升的 AGENTS.md 规则转为 SKILL.md，让他们从常驻 noise 变成按需加载。

但要小心：约束还在变化中时不要急于上移到类型系统或 lint——一个 `strict: true` 可能炸几十个编译错误。先放低层级观察，稳定后再迁移。另外，有些约束是项目特有的（「README 必须在功能变更后更新」），没有通用工具能自动检查，只能放在 Skill 或 AGENTS.md。

## 蒸馏闭环：把踩过的坑变成 Agent 的下一条技能

前面五节讲的全是搭和优——把工具层调可靠，让 Agent 能正常干活。这一节讲的是蒸——**让你花三个月调出来的最佳实践，在新同事入职第一个 session 里自动加载**。

不是写文档。文档在 Notion 里躺着，Agent 不会读。蒸馏是把经验编码为 SKILL.md——不是给人看的，是给 Agent 调用的。

keli-wen 用一个词描述蒸馏的本质：「低秩」。代码是高维的——每一行、每一个 if 分支、每一个配置项。但有价值的设计模式是低维的——你不需要完整复现 51.2 万行 Claude Code 源码才能理解「为什么它的编辑工具能处理并发文件修改」。你需要的只是：**「文件读取和编辑之间加哈希校验，外部修改导致哈希不匹配时拒绝打补丁」**——一句话，可迁移。

> 代码是高维的，但有价值的设计模式其实是低秩的，蒸馏的本质就是找到这些主成分。
> — keli-wen，Claude Code 源码蒸馏实践（oh-my-pi 研究 §3）

### 三条路径，三种输入

**源码蒸馏**最重——输入是别人的完整 harness 实现。keli-wen 用多 Agent 协作（Codex 审查 + Claude Code 执行，每轮全新 session 互不可见），把 Claude Code 的 51.2 万行 TypeScript 蒸出 6 条设计原则。handoff 文档做 Agent 间 API，自己的博客作为品味「基向量」注入——蒸馏结果不是源码的客观映射，而是经过个人偏好投影后的产物。

**经验蒸馏**中等——输入是博文、工程笔记、团队方法论。Claw 把 OpenAI 的 Harness Engineering 博文（Ryan Lopopolo 的实战：3 人 5 个月、~1500 PR、100 万行代码）蒸馏成结构化 Skill，产出的「黄金法则」表可直接套用。

**会话蒸馏**最轻——从每次 Agent 运行的实际经验中持续改进。phrazzld 的 reflect Skill 在每次 session 结束后自动反思，cycle 模式外环周期评审，自动 push harness 改进到 `reflect/` 分支。这意味着 harness 的进化从「手动的」变成了「持续的」。

三条路径最终指向同一个容器：**SKILL.md**。这个文件有两个关键设计。一是 **description**：只有 frontmatter 的 description 进入 system prompt，正文在模型匹配到相关任务时才按需加载——大型参考文档平时零成本。description 写得好不好直接决定模型会不会在正确时机加载：`"Helps with database stuff"` 是一张废票，`"Writing, reviewing, or optimizing Postgres queries, schemas, or configs"` 是一张能进场的票。二是 **触发机制**：不是 Agent「记住」你的 Skill——只有 description 触发了语义匹配，它才会被加载。正文写得再好，description 写差了就是零。

### 动手

1. **找一个最近踩过的坑，写出 SKILL.md 骨架**——name（kebab-case）+ description（精确触发词，512 字内）+ 触发条件（≥3 个具体场景话术）。
2. **自检 description 质量**——让没看过原文的同事只看 description，说出什么时候触发。说不准就改。
3. **走完一次完整蒸馏**——从一篇博文或一次 session 教训出发，产出 SKILL.md + 至少 1 个 reference 文件，放进 `.omp/skills/`，确认 `/extensions` 面板显示已加载。

但要小心伪蒸馏：如果产物和原文同样复杂，既不可迁移也不可维护。蒸馏的价值在压缩，不在转录。读完之后无法用一句话概括核心发现，还没蒸完。

## 陷阱清单

从素材中提炼的五个最常见翻车方式：

1. **盲目升级模型而非检查 harness。** Agent 反复输出 `String not found` 或 `patch failed` 时，第一反应是换更强的模型——花更多钱、烧更多 token、问题依旧。失败发生在 harness 层时，升级模型是南辕北辙。
2. **约束全部往下塞。** 所有规则都堆进 AGENTS.md 和 Memory，几个月后上百条。Agent 的 recall 变成随机抽样。「禁止 any」写了三遍，没人打开 tsconfig strict。能编译检查的绝不靠记忆。
3. **并行不做隔离。** 多个子 Agent 在同一工作区编辑同一组文件，最后一个写入者胜出。git diff 里看不到冲突标记——三个独立进程写同一个文件，静默丢失。
4. **system prompt 瘦身后把空间全填回去。** TTSR 释放的 context budget 不是给你塞新内容的——是给关键指令增加注意力密度的。
5. **description 模糊导致 Skill 静默失效。** Skill 写好了但从没见过模型自动加载。大概率是 description 太模糊——从 `"Helps with deployment"` 改成 `"Deploying the CI pipeline to staging or production, including rollback procedures"`。

## 作者的局限

1. **基准测试只覆盖 JavaScript/TypeScript。** Python 的缩进敏感特性可能对 str_replace 更不友好，C/Rust 的函数级编辑可能对 Patch 更友好——这些未验证。结论在其他语言栈上需要独立校准。
2. **未量化迁移成本。** 将一条约束从 AGENTS.md 迁移到 TypeScript strict mode 可能触发数十个编译错误——这个一次性迁移成本和 AGENTS.md 自然漂移的长期成本之间的 tradeoff 未讨论。
3. **蒸馏的客观性存疑。** keli-wen 的蒸馏案例中，个人博客作为「品味基向量」注入——不同工程师蒸馏同一份源码可能得出不同的「核心原则」。蒸馏产物的稳定性、可复现性在材料中未讨论。

## 如果只带走三句话

1. **Agent 不好用的时候，先查桥，别换车。** 工具层比模型更决定最终表现。
2. **最好的规则不是写得最清楚的那条——是 Agent 想违反都没办法违反的那条。** 能上移的约束绝不往下塞。
3. **蒸馏不是为了少打字——是为了让 Agent 在没有你的 session 里，也像有你在一样做事。**

## Monday Morning

明天上班，做这三件事：

1. **统计编辑失败率。** 打开你的 Agent 日志，数出「编辑调用次数」和「编辑失败次数」。如果失败率超过 15%，换 Hashline 跑 5 个同类任务做对比——不需要重建整个工具链，只需要换编辑格式。
2. **审计 AGENTS.md 的约束层级。** 逐条标注「最高可到哪一层」。找到至少 3 条可以上移到类型系统或 lint 的规则，迁移它们。然后从 AGENTS.md 删除冗余。
3. **蒸一个 Skill。** 从最近踩的一个坑出发，写 SKILL.md 骨架——name + description + 触发条件。不用完整，先走通「经验→Skill」的路径。放到 `.omp/skills/`，确认 `/extensions` 能看到。

## 关键术语速查

| 术语 | 含义 | 和你想的不一样在哪 |
|------|------|---------------------|
| Harness | 模型输出到工作区变更之间的所有环节 | 不是「工具集合」，是决定了模型能力能不能落地的瓶颈层 |
| Hashline | 内容哈希锚点编辑 | 不是增量编辑的一种——是把精度负担从模型转移到工具 |
| TTSR | 正则触发 + mid-token 中断的规则注入 | 不是「更好的 prompt engineering」——是改变了规则的时间属性（需要时才出现） |
| SKILL.md | 给 Agent 调用的经验包 | 不是文档——是模型按需加载的可执行知识 |
| 约束固化层级 | Type system > Lint > Hook > Test > CI > Skill > AGENTS.md > Memory | 不是优先级排序——是「哪个层面 Agent 绕不过去」 |

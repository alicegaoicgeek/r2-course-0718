# 循环工程时代的提示词基本功

**副标题：逐轮手写 Prompt 的能力没有作废，而是被封装进了 Loop 的第一性原理**

---

## 引言

2026 年 6 月，Claude Code 负责人 Boris Cherny 在接受 Acquired Unplugged 播客访谈时说了这样一段话，迅速在开发者社区炸开：

> 「I don't prompt Claude anymore. I have loops running that prompt Claude and figuring out what to do. My job is to write loops.」

我不再给 Claude 写 Prompt 了。我有 Loop 在跑，它自己去写 Prompt，自己去琢磨该做什么。我的工作是写 Loop。

一家顶级 AI 公司编码工具的最高负责人，亲口说他不再亲手写提示词了。对过去两年靠「Prompt Engineering」建立职业认同的开发者来说，这句话听起来像一记重锤：基本功是不是白练了？Prompt Engineering 是不是死了？

如果只看到这句话的前半句——「I don't prompt Claude anymore」——很容易得出一个仓促的结论：这个时代不需要学 Prompt 了。

但如果把整句话读完，会发现 Boris Cherny 说的是另一回事：**不是不需要 Prompt，而是不需要「你」来写 Prompt 了。你要做的是设计一个系统，让这个系统去写 Prompt、迭代 Prompt、验证 Prompt。**

Google 工程领袖 Addy Osmani 在 2026 年 6 月 7 日正式命名了这个新范式——**Loop Engineering**。他对这个时代的核心判断，一句话就讲透了：

> 「Loop engineering is replacing yourself as the person who prompts the agent. You design the system that does it instead.」

循环工程，正在取代「你自己」成为那个给 Agent 写提示词的人。你设计的是那个系统。

本文要论证的观点很简单：**在循环工程时代，提示词基本功没有死——它被「下沉」了一层，成了更大系统里的原子单元。基本功的形态变了，但基本功本身不但没有贬值，反而因为需要被系统化调用而变得更加重要。**

---

## 一、四代范式演进：从「写一句好 Prompt」到「设计一个自转的 Loop」

要理解「基本功为什么没死」，必须先看清范式的全貌。


### 第一代：Prompt Engineering（2023–2024）

这是「手写咒语」的时代。开发者通过角色提示、Few-Shot 样例、Chain-of-Thought 思维链（CoT）等手段，榨干单次输出的潜力。核心逻辑是：**用更高级的提示词技巧，让模型在一次调用里给出更好的答案。**

关键技术在进步，但本质是「单次博弈」。人类写好 Prompt，模型输出结果，人类检查结果。如果结果不好，人类重新写 Prompt。这个模式的局限是显性的：遇到复杂任务时，幻觉率指数上升，而人工持续跟进的成本没有上限。

### 第二代：Context Engineering（2024–2025）

这一代的核心变化是：**给 AI 接入「外部大脑」与「海量记忆」。**

RAG 检索增强生成成为标配，长上下文模型开始商业化落地，开发者需要处理的不再只是「怎么写一句好 Prompt」，而是「怎么管理输入给模型的全部信息流」。防「Lost-in-the-Middle」现象、把意图外置为 Skills、把状态外置为 Memory——这些新基本功出现了。

但这一代仍然有一个根本局限：AI 是「被动执行者」。它不会主动思考下一步该做什么，需要人类在每个节点推它一把。信息环境变大了，但决策权还在人手里。

### 第三代：Harness Engineering（2025 年中）

「驾驭工程」这一代做了关键的事：**为 AI 打造标准化的「数字沙箱」与「工具箱」。**

Agent 运行沙箱、Tool Calling、Router 分发微 Agent——这些技术让 AI 开始有了「手」。它能调 API、能执行脚本、能在多个工具间切换。但它的「脑」仍然被动：工具返回错误时，它不会自我修正，直接报错结束。缺乏自主迭代校验能力。

### 第四代：Loop Engineering（2026 年 6 月 – 至今）

这是质变的一代，也是 Boris Cherny 说的「My job is to write loops」所指向的新范式。

核心特质有三个：**目标驱动、多轮自我修正、无人值守长时程**。

Osmani 在帖子里用一句话精炼了这个范式的本质：「你只给一个高层模糊目标，AI 自己去拆、去试、去检查、去修正，直到完成。」它不是 Prompt Engineering 的升级版——它是把前三个代际的能力全部封装进了一个「自转的闭环」里。

Osmani 给出的公式化表达是：

> **Loop = Goal + Context + Action + Evaluation + Reflection**

目标定义方向，上下文注入信息，Action 执行操作，Evaluation 评判结果，Reflection 修正策略——五要素首尾相接，形成自循环。

开发者在这一代不再直接对 Agent 写 Prompt。**开发者的角色从「代码编写者」变成了「系统环境与评价标准的制定者」。**

## 二、基本功没有死，是被「下沉封装」了

看清范式演进的路线之后，回答那个最核心的问题：Prompt 基本功到底有没有用？

Anthropic 官方在《Prompt Engineering Best Practices》文档中给出了最权威的回答：

> 「Prompt engineering 并未消亡，而是成为 Context Engineering 的原子单元。」

「原子单元」这个词是关键。它没有说「基本功升级了」——升级是渐变，而这里发生的是**层级重组**。传统 Prompt 基本功从「你每次干活用的工具」变成了「你设计的系统在内部调用的基础组件」。

把这对应到 Osmani 的四代演进中，可以看到一组清晰的对应关系：

**Loop Engineering = Prompt Engineering → Context Engineering → Harness Engineering**

每一代不是取代前一代，而是把前一代「收编」为自己的子模块。Prompt 是 Context 的原子单元，Context 是 Harness 的信息基础，Harness 是 Loop 的肌肉系统。层层嵌套，而不是互相淘汰。

### 三层能力清单

把高权重来源交叉归纳后，可以得出 Loop Engineering 时代的「基本功三层结构」：

**① 原子层：传统提示词基本功（被官方认证为基石）**

- 清晰、显式的指令
    
- 给上下文与动机
    
- 具体约束与格式
    
- 样例（few-shot）
    
- 允许模型说「不知道」以减少幻觉
    
- 响应预填与格式控制
    
- Chain-of-Thought（思维链）
    
- Prompt Chaining（串行拆解——这是循环的雏形）
    

这些在单次手动调用时练出来的能力，不是「过时了」，而是变成了 Loop 设计中必须被精确编码的底层规则。你不需要对着 Claude 手写 CoT 了——但你要确保你设计的 Loop 能在适当环节自动插入 CoT。

**② 系统层：Context Engineering**

- 管理模型的「输入信息流」：RAG、长上下文防迷失
    
- 把意图外置为 Skills
    
- 把状态外置为 Memory（对抗 Intent Debt / Comprehension Debt）
    
- 避免每轮重推上下文
    
- 设计系统环境而非逐句喂料
    

这一层是新基本功的核心增量。它要求你把注意力从「单次对话怎么能更好」转移到「系统怎么让多轮对话能串联成连贯的工作流」。

**③ 闭环层：Loop Engineering（新增项）**

- 目标驱动：只给高层模糊目标
    
- 多轮自我修正：Evaluation + Critique（源自 Reflexion）
    
- 无人值守下的 maker/checker 验证（对抗式代码审查）
    
- 外部记忆与状态机
    
- 人类闸门与判断（对抗 Cognitive Surrender）
    
- 成本与停滞的熔断（circuit breaker）
    

**三层之间的关系不是「选一层学就行」，而是「每一层都是上一层的支撑」。** 你不懂原子层的 Prompt Chaining，就设计不好系统层的 Context 流；你不懂系统层的 Memory 管理，闭环层就变成每轮都从零开始的瞎子。Loop Engineering 不是跳过基本功的捷径——它是要求你把基本功练到能「写进系统规则」的深度。

这就意味着「基本功的练习方式」也变了。过去你练 Prompt Engineering，是打开一个对话窗口，反复调整一句话直到输出满意。现在你练 Prompt Engineering，是打开一个 Skill 文件，写下系统在不同场景下应该用什么策略生成 Prompt——你的练习对象从「单句」变成了「规则集」。同理，过去你练 Context Engineering 是手动组织每次对话的背景信息，现在你练的是：Memory 文件应该存什么粒度、怎么让 Loop 在不同轮次取出正确的上下文段落。练习方式的改变，恰恰说明基本功没有被淘汰——它只是搬了家。

## 三、角色迁移：从「写 Prompt 的人」到「写 Loop 的人」

Boris Cherny 那句「I don't prompt Claude anymore」之所以炸，不是因为它在说技术变了——技术在 2023 年就在变。它炸，是因为它直接碰触了开发者心中最敏感的一根弦：**身份。**

过去两年，大量开发者把自己的竞争力定义在「我能写好 Prompt」上。当 Agent 构建领域最顶尖的人亲口说「那个阶段基本上结束了」（Osmani 原话），这听起来不是升级，是身份危机。

Osmani 在他的后续文章《Own the Outer Loop》（2026-07-09）中把这个身份转变拆解得非常清楚。他说的是同一件事，但换了一个角度：

> 「The leverage point moved from writing a good prompt to designing the system that prompts the agent.」

注意他的用词——「leverage point」（杠杆点）。他不是在说「技能升级」，他是在说「力的作用点变了」。过去你的力量作用在「这句 Prompt 怎么写」，现在你的力量作用在「这个系统怎么自己写出那句 Prompt 并判断它写得好不好」。

最早点燃这场讨论的其实不是 Osmani，而是 OpenClaw 的作者 Peter Steinberger。他在 2026 年初就说过一句更锋利的话：

> 「You shouldn't be prompting coding agents anymore. You should be designing loops that prompt your agents.」

你「不应该」再亲自给编码 Agent 写 Prompt 了。你应该设计 Loop，让 Loop 去写 Prompt。

从 Steinberger 的「shouldn't」到 Cherny 的「I don't」，再到 Osmani 的「leverage point moved」，这是一条从「观念推演」到「亲身实践」再到「范式命名」的完整脉络。

### 角色的实质变化是什么？

如果从三层能力清单的角度来拆解，这个角色转变有非常具体的含义：

**过去**，你的日常工作是：写 Prompt → 看输出 → 改 Prompt → 再看输出。你的价值体现在「你写的 Prompt 能不能让模型给出好结果」。

**现在**，你的日常工作是：定义 Goal → 配 Memory → 设 Evaluation 标准 → 审核 Loop 的输出。你的价值体现在「你设计的系统能不能在无人值守的情况下，持续产出符合标准的结果」。

Osmani 在帖子里说得非常直白：「你只设计了一次。这些步骤你一个 Prompt 都没写。」

但不是「不需要 Prompt 了」，而是**你写的不是 Prompt，你写的是 Prompt 的产生规则和评判规则。**

这就像从画家转型为画廊策展人。策展人不画画，但如果没有对画的理解，他策不了展。同理，如果你不懂 Prompt 的基本功——不知道 CoT 在什么场景下有用、不知道 few-shot 的边界在哪、不知道指令应该多具体——你就设计不出有效的 Loop。你写的 Evaluation 标准会是盲目的。

## 四、落地：五大原语、七步生命周期与两篇论文

如果说前面三个部分解决了「怎么想」的问题，这一部分解决「怎么做」。

### 五大原语

构成一个 Loop 的五种基础设施原语，加上一个跨会话的记忆机制：

**1. Automations**（定时触发发现与分拣）

Loop 的引擎。不是等人类来指挥，而是按日程表自动跑。每天定时扫一遍仓库，发现问题，做分拣。这是「无人值守」的第一推动力。

**2. Worktrees**（隔离并行）

当一个 Loop 同时处理多个任务时，两个 Agent 不能互相踩脚。Worktree 给每个子任务一个独立的运行空间，让并行真正可行。

**3. Skills**（项目知识固化）

Agent 不知道你的项目怎么写的、为什么这么写。把项目知识写成 Skills，Agent 就不会瞎猜。这是循环工程里的「把意图外置」——你教给系统的不是一句 prompt，而是一套项目规则。

**4. Plugins / Connectors**（接外部工具）

通过 MCP 协议，把 Agent 接入你现有的工具链。Loop 不是活在真空里的——它要能开 PR、能更新 ticket、能发通知。

**5. Sub-agents**（maker/checker 对抗验证）

一个 Agent 出主意，另一个 Agent 来核查。这是循环的「内置质量保证」。Osmani 在他的延伸文《Adversarial Code Review》中专门拆解了这个机制——模型自己写的代码，换一个模型来审，对抗式验证的效果远好于自我审查。

**+ Memory（状态）**

「Agent 会忘，repo 不会。」状态文件（markdown / Linear 看板）是 Loop 的脊柱。它活在单次对话之外，让第二天的 Loop 能从今天停下的地方续跑。

### 七步生命周期

把一个 Loop 从触发到完成的完整流程拆开来，是这样的：

① **Automation 每日触发仓库**——到时间了，Loop 醒来。

② **Triage skill 读昨日 CI 失败 / issue / 提交，写入状态文件**——先搞清楚「昨天留下什么活没干完」。

③ **对每个值得做的发现，开隔离 worktree，派 sub-agent 起草修复**——每个任务独立跑，互不干扰。

④ **第二个 sub-agent 对照项目 skills 与测试做审查**——maker 干完活，checker 来验收。

⑤ **Connector 让 loop 开 PR、更新 ticket**——干完的活自动推到该去的地方。

⑥ **处理不了的落到 triage 收件箱交给你**——Loop 不是万能。搞不定的活，排队等你审。

⑦ **状态文件作脊柱，次日从今日停处续跑**——第二天，Loop 不用从头开始。

整个流程里，开发者做了什么？Osmani 的答案是：「你只设计了一次。这些步骤你一个 Prompt 都没写。」

把这五原语和七步放在一起看，会发现 Loop Engineering 本质上在消除一个开发者最熟悉的循环：「发现问题 → 写 Prompt → 看输出 → 不满意 → 改 Prompt → 再看输出 → 还不满意 → 再改」。这个循环过去每一轮都靠人驱动，现在被原语组合接手了。Automations 替代了「发现问题」的主动性，Worktrees 解决了并行处理的隔离问题，Skills 把「改 Prompt 时的项目知识」写死成了规则，Connectors 把「输出满意后手动开 PR」自动化了，Sub-agents 替代了「人工审查」这个最耗精力的环节。而 Memory 解决了这个流程最致命的弱点——跨天续跑。整个七步生命周期，每一环都对应一个人工痛点，每一环的解法都是原语的特定组合。

### 学术根基：Reflexion 与 ReAct

Loop Engineering 不是凭空产生的。它的学术根基可以追溯到两篇奠基性论文。

**Reflexion**（NeurIPS'23）：核心思想是「agents verbally reflect on feedback 并写入 episodic memory」。Agent 执行动作后，用自然语言对结果做反思，并把反思结论存到记忆里，下次不再犯同样的错误。这在 Osmani 的五要素公式里就是 **Evaluation + Reflection** 环节的学术原型。

**ReAct**（ICLR'23）：提出了「思考—行动—观察」的交替循环范式，是整个 Agent Loop 的奠基结构。Reflexion 就是构建在 ReAct 之上的——在 ReAct 的「行动→观察」之间插入「反思」。

而 **DSPy**（2023）则连接了「基本功」与「工程化迭代」：它把提示词当作可程序化优化的对象，让「写 Prompt」这件事本身就变成可以自动迭代的 Loop。这恰好呼应了 Loop Engineering 的核心命题——不只是让 Agent 自己干活，还要让它自己写 Prompt。

## 五、克制与边界：反模式警告

如果只谈 Loop Engineering 有多厉害，不谈它的风险和边界，那它就不是分析，是鼓吹。

### 1. Token 成本

「无人值守的 loop 也在无人值守地犯错，必须盯紧 token 用量。」

Loop 在自动运转的时候，没有人盯着它每一步做了什么。一个错误的推理路径可能会被反复执行，消耗大量 token。无人值守是效率的放大器，也是浪费的放大器。

### 2. Comprehension Debt（理解债）

「顺畅的 loop 会让你更快丧失理解，除非你读它产出的东西。」

这是 Osmani 在延伸文《Comprehension Debt》中展开的一个核心概念。当 Loop 替你干完了所有的活、产出了完整的代码，你很容易跳过「理解它做了什么」这个步骤。一次两次还好——十次二十次以后，你对代码库的理解就实质性丧失了。Loop 越顺，债越深。

### 3. Cognitive Surrender（认知放弃）

这是 Osmani 三条警告里最深刻的一条：

> 「loop 自转时很易停止思考、照单全收——带着判断去设计 loop 是解药，为逃避思考去设计则是催化剂，同一动作，相反结果。」

Cognitive Surrender 不是技术问题，是认知问题。Loop 的自动化会制造一种错觉：「既然系统已经能自己跑了，我就不需要动脑子了。」但系统跑得好不好、产出是否可靠、是否偏离了初衷——这些问题没有一个能脱离你的判断。

Osmani 收尾做了一个特别精准的限定：「能跑多少个 loop 取决于你的审查带宽，而非工具。」你审得过来的 Loop 数量才是你的真实产能，不是你能启动多少个 Loop。

**验证仍归你。你的工作仍是「发布你确认能跑的代码」。**

---

## 结语

用两句话概括整个 Loop Engineering 时代的基本功定位：

第一句：「go ahead and set up your loops, but don't forget that prompting your agents directly is also effective. It's all about finding the right balance.」

去设你的 Loop 吧，但别忘了直接给 Agent 写 Prompt 也依然有效。关键是找到平衡。

第二句：「That's what makes loop design harder than prompt engineering, not easier. The leverage point moved.」

这正是 Loop 设计比 Prompt Engineering 更难的地方——不是更简单。杠杆点移走了。

过去你的杠杆点在「怎么写一句更好的 Prompt」上。现在你的杠杆点在「怎么设计一个能让 AI 自己写 Prompt、自己验证 Prompt、自己修正 Prompt 的系统」上。这个新杠杆点对基本功的要求不是更低，而是更高——你需要理解的不只是 Prompt，还有 Context、Memory、Evaluation、Reflection，以及怎么把这些东西串成一个能自转的闭环。

Anthropic 说 Prompt Engineering 没有消亡，而是成了 Context Engineering 的原子单元。Osmani 说「That's what makes loop design harder than prompt engineering, not easier」。两个人的话指向同一个结论：**层级重组之后，基本功不但没有贬值，反而因为要适应「被系统调用」的新位置，对精度的要求比以前更高了。**

你过去写的 Prompt，错了改一句就行。你现在写的规则，错了会影响 Loop 的每一步。

基本功没有消亡。它从你指尖的技艺，变成了你设计的系统里的第一性原理。

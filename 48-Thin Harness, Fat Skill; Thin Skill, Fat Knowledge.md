你的 Agent 系统上线三个月了。最近它越来越笨——简单任务也开始出错，复杂任务直接胡说。你以为是 prompt 没写好，反复调了一周，效果不升反降。

方向错了。问题不在 prompt，在架构。

打开你的 Harness 代码。数一下工具定义。超过 40 个了吗？EDA-cli 每个脚本都包了一层 wrapper？上下文窗口被工具 schema 吃掉了一半？
Garry Tan 的实战结论是：Harness 200 行足矣。 超过这个数字，说明你把业务逻辑写进了不该写的地方。

**总论点：Agent 系统的架构质量取决于三层之间的分工纪律——Harness 做编排不做决策，Skill 做分类不堆正文，Knowledge 做编译不做检索。每一层都有自己的「不能做什么」的边界。三层分工不是一次性决策，而是一个持续运转的系统。每一轮循环都比上一轮更准。**

全文沿两条逻辑线展开。第一条是分工线：三层各安其位。第二条是压缩线：从 Harness 到 Skill 到 Knowledge，每一层都在向下传递信息时做一次有损压缩，压缩的质量决定系统的信号纯度。

## 地基：三层架构与那条不容含糊的线

Garry Tan 把 Agent 系统拆成三层。这听起来像常识——但三层各自「不能做什么」才是他真正的贡献。
| 层 | 能做什么 | 不能做什么 |
|---|---|---|
| Thin Harness | 模型循环、上下文管理、工具调用、安全 | 承载业务判断 |
| Fat Skills | 模糊判断、领域知识、工作流程、决策逻辑 | 执行确定性计算 |
| Fat Code | 数据库查询、文件操作、API 调用、算术运算 | 做需要综合判断的推理 |

差异不在于智能，而在于架构。
这句话的潜台词：两个 Agent 用同一个模型、同一个 prompt 框架，效果可能天差地别——差别全在「哪些事让模型做、哪些事让代码做」的分工上。

三层中有两个边界判断贯穿全文。

判断一：Harness 是连接层，不是策略层。 混淆编排和策略是 Agent 系统最常见的架构失败。工具定义超过 40 个、REST API 每个端点一个 wrapper、MCP 往返 2–5 秒的全能工具——这些不是「Harness 太重」的症状，是「在 Harness 里写业务逻辑」的症状。修复方向只有一个：工具逻辑封装为专用 CLI（确定性、零延迟），业务判断移入 Skill 的 Markdown。

判断二：潜在与确定性分离。 Garry Tan 用了一个绝佳的类比：
8 人排座 vs 800 人排座——LLM 排 8 人座位没问题，排 800 人时一切看似合理，但座位表完全错误。
概率系统的危险不是它会「出错」——错误你一眼就能看到。真正的危险是它给出的结果「看似合理但完全错误」。这条区分线不容灰色地带：确定性需求（相同输入 → 相同输出）走 Code，概率性判断走 LLM。CFO 类高信任任务用纯 Python（确定性），顾问类综合判断用 LLM + 记忆（潜在性）。复合流程通过两层的组合调用来完成，而非让 LLM 一路 carry 到底。

最差的系统将错误的工作放在这条线的错误一侧。最好的系统对此毫不留情。

---

到此为止，我们讲的是第一式 **Thin Harness, Fat Skill**：Harness 是薄薄一层编排，业务判断放在 Skill 里。但「Fat Skill」不是说 Skill 正文越厚越好——恰恰相反。下面进入第二式 **Thin Skill, Fat Knowledge**：Skill 正文本身也要瘦身，只保留会改变模型行为的 gotchas；真正的知识重量沉入经过编译的 Knowledge 层。

## Skill 的本质是分类，不是内容

你写了 10 个 Skill。三个月过去，质量明显下降。文本没变过——变的是基座模型、工具接口和上下文环境。
静态 Markdown × 概率系统——错位是结构性的，不是写作水平的错。
路由衰减：Skill 越多越不准
Skill 的核心价值不在正文有多厚，而在「能不能被正确触发」。触发本身就是分类问题——Agent 要在 N 个 Skill 中选对那一个。

实测数据：

| Skill 数量 | 路由准确率 |
|---|---|
| ≤20 | ~90%+ |
| >30 | 明显下降 |
| ~200 | ~20%，且慢、费 token |
Claude Code 的两级路由系统是精妙的设计：L1 扫描所有 Skill 的 name + description（每个约 250B），L2 命中后才加载 SKILL.md 全文。这意味着 description 的唯一功能是路由触发——不是功能宣传，不是营销文案。写「这个 Skill 可以生成漂亮的演示文稿」→ Agent 看不懂。写「当用户需要把文章、大纲或演讲内容转成可演示 HTML PPT 时加载」→ 这才是 Agent 能匹配的路由键。

路由衰减的因果链是这样的：L1 输入是 N 个 description 组成的窄窗口（约 N×250B），当 N 增长时 —— 第一，上下文窗口中的总 description 长度增加，token 消耗上升；第二，更多 description 在语义空间中互相挤压，区分度下降。N 个「种级」相近的 description 在模型眼里共享同样的语义特征，分类器在窄输入下做 N 选 1，准确率自然坍塌。路由衰减是静默的：Agent 选了一个不匹配的 Skill 去执行，产出质量差但不会报 404。你以为 Skill 还在正常工作，其实它已经「选不中」了。

Skill 是税，不是无限资产

每个 Skill 的 name + description 在每会话都付上下文成本。Skill 越多，token 越被稀释。正文只保留会改变模型行为的判断和 gotchas（失败边界），删除模型已知的正向原则。
你的 Skill 正文里的「代码要简洁」「注释要清晰」——模型已经知道这些。留着只是在浪费 token 预算、挤占真正有价值的 gotchas 的空间，比如「不要用 String.substring 处理 Unicode」「这个 API 的 timeout 默认是 0 不是 30s」。

## 三线激活：从便利贴到 IP

写好的 Skill 不是资产——它只是一个便利贴。从便利贴到工业级 IP，需要三条线同时激活：

* 线一（结构约束）：Harness 提供编排协议——接口契约、调度组合。模型升级时 Harness 检测 Skill 行为偏移，触发修复或版本回滚。

* 线二（反馈回路）：执行轨迹 → 信度分配 → 因果链重构 → 版本写回。人的修正行为是 ground truth——把单次成功经验转化为可累积的决策模式。

* 线三（组织背书）：DRI 维护 + 流程嵌入。不存在一次性买断一套 Skill 可以一年不用改——今天有效，一个月后模型升级就是反作用。

判据：结构约束 + 反馈回路 + 组织背书 → 三线俱在 Skill 即 IP；缺一即便利贴。

## RAG 不是答案——知识需要编译，不是检索

团队有 200 篇设计文档。你搭了 RAG——向量库、chunking、embedding、reranker。Agent 查设计决策时经常给错答案。你调了 embedding 模型、调了 chunk size、加了 reranker。还是不准。
方向错了——不是检索精度的问题，是架构假设的问题。
Garry Tan 和 Karpathy 独立收敛到了同一个结论：不应该每次都从原始文档重新推理。
RAG 的三大根因缺陷
| 缺陷 | 现象 | 为什么不是「优化」能解决的 |
|---|---|---|
| 上下文碎片化 | 每次查询从原始文档临时拼凑，知识不积累 | RAG 是单次无状态管道——没有「上次查过」这个概念 |
| 缺乏复合能力 | 跨文档推理、矛盾检测几乎不可能 | 检索→生成是线性管道，无法做跨文档的逻辑组合 |
| 缺乏行动能力 | 只读不写，知识的「更新-维护-进化」循环不存在 | RAG 没有写回路——知识只能进不能修正 |
对比一个具体场景：新来一篇设计文档，推翻了之前的技术选型。

* RAG 系统：新文档被 chunk、embedding、入库。旧文档还在那里。查询「我们为什么选了 A 方案」时，可能同时召回支持 A 的老文档和推翻 A 的新文档——Agent 看到两套矛盾的证据，不知道哪个是当前结论。

* 编译系统：LLM 读新文档 → 检测到与旧结论矛盾 → 标注旧页面为「已过时」→ 更新当前结论页面 → 记录决策链。下次查询直接命中最新结论。

区别不是检索精度，是有没有状态积累。

当然，编译范式有一个真实边界：如果新文档推翻的不是旧结论的全部，而只是它的一个子条件（比如「A 方案适用于吞吐优先场景，但延迟敏感场景不适用」），编译系统需要把旧结论从全称判断改为条件判断，而非简单标记为「已过时」。这是 Lint 巡检的核心职责——发现矛盾时判断是全量淘汰还是部分修正。
编译范式：Compile-then-Query
LLM 通读所有文档，提取关键决策、标注矛盾、维护结构化的知识页面。查询时直接读编译后的结果而非重新推导。「每次问图书管理员现翻书找答案」vs「图书管理员已经把摘要和索引编好了」——两种系统的工作方式截然不同。
Karpathy 的实测印证了这一点：在研究级规模下（数百篇文章），LLM 自己维护索引文件和文档摘要就够了，查什么都顺畅。
更深的命题：记忆不是存储，是因果链压缩
Perfect memory is a bug.

这句话是整篇文章的支点。Claude Code 的 25KB 死规矩不是技术限制——是强制压缩。把离散事件（A 事件 → B 后果 → C 修正）压缩成可泛化的行为模型（遇到 A 类事件先做 C）。
很多人抱怨「我的 Skill 内容很重要，为什么限制我？」——这句话背后的假设是「存储越多，系统越强」。Karpathy 的答案颠倒了它：你被迫把 3000 行的 Skill 压成 500 行，这个过程本身就是因果链压缩——删掉的是模型自己能推导的正向原则，留下的是只有你踩过坑才知道的 gotchas。 压缩不是妥协，是提纯。
统一路线：Karpathy 的 LLM Wiki（纯文件系统编译范式）
你决定把知识库从 RAG 迁到编译模式。现在不需要在两条重轻路线之间纠结——社区已经围绕 Karpathy 的 LLM Wiki 收敛出一套成熟、可立即落地的统一体系。
Karpathy 的 LLM Wiki 是 75 行的 Markdown Idea File——零数据库依赖。核心三层：

- Raw（`raw/` 目录）：纯 Markdown 原始资料，不可变，LLM 只读不写。
- Wiki（`wiki/` 目录）：LLM 完全拥有并维护的知识页面。包含 index.md、log.md、实体页、概念页。页面之间用 `[[wikilink]]` 互相链接。
- Schema（`CLAUDE.md`）：定义命名约定、页面类型、Ingest/Query/Lint 工作流。

三大操作：**灌入**（新资料 → LLM 读取 → 更新 10–15 个相关页面）→ **提问**（读 index.md → 顺 wikilink 导航 → 综合回答，好回答归档回 Wiki）→ **巡检**（定期找矛盾、补缺失、标关联，这是代谢的核心动作）。
最本质的优势是零重型中间件。关联建立靠 LLM 写 wikilink，矛盾检测靠 Lint 时「读」出来，知识检索靠读 index.md 后导航，计算操作尽量交给确定性代码。这完美对应地基的潜在与确定性分离——LLM 只负责理解和决策，确定性的部分绝不让它碰。

**选择判断**：统一用 Karpathy 路线起步。单人维护、文件 <500、一天内搭好 → 纯 Markdown + Obsidian + 社区 Skill。文件变多、grep 从秒变分钟 → 再加轻量本地搜索工具。多人协作 + 权限隔离需求出现 → 再考虑升级，但核心仍是「编译后的 Wiki」，不退回到纯 RAG。原则：**撞到边界才扩**。每多一层中间件就多一层维护负担，非必要不加。

## 运转：知识库会越变越傻，除非你给它代谢

知识库运行了半年。从一开始的趁手工具，慢慢变成了累赘——Agent 给出的答案经常自相矛盾，三年前的架构决策和昨天的复盘在检索中平权。

只进不出 = 被剥夺睡眠的大脑。

这不是比喻。过时结论和新判断等权出现 → Agent 不分时间维度全都当有效信息处理 → 信噪比跌破可用阈值。知识库从资产变负债的拐点就在这里。

## 四库飞轮：不是按主题，是按消费顺序
传统知识库按「页面」组织——一篇文档一个页面，目录树做导航。Agent 不需要翻页——它需要在多轮检索中自己找到正确的信息单元、自己判断信度、自己组合成答案。
四库的组织方式颠倒了这个逻辑：不是按「内容是什么」组织，而是按「Agent 消费时按什么顺序检索」组织：
| 库 | 回答的问题 | 检索顺序 |
|---|---|---|
| 问题库 | 用户要什么？ | 第一站——先判断方向 |
| 观点库 | 我的判断是什么？ | 第二站——锚定立场 |
| 案例库 | 凭什么这么说？ | 第三站——找证据 |
| 复盘库 | 上次哪里不对？ | 第四站——避开旧坑 |
Agent 先搜问题库判断方向 → 观点库锚定立场 → 案例库找证据 → 复盘库避旧坑。这个顺序本身就是计算量的压缩。 飞轮每转一圈：Agent 消费 → 人复盘 → 回写四库 → 下次检索更准。
三条铁律
铁律一：无据不写。 Agent 生成时，每个论断必须有知识库锚点——问题库证明需求真实、案例库证明不是拍脑袋、观点库证明经过审慎思考、复盘库证明坑已踩过。找不到锚点 → 不产出。宁可沉默不编造。
这不是在限制 Agent，是在解放它。它把 anti-hallucination 的战场从检索端移到了生成端——检索永远不完美，但硬门控可以在生成之前截住幻觉。
铁律二：写入端有人工闸门。 AI 做发散，人做收敛。固定 3/20 的筛选比。跳过人工闸门 = 知识库变 AI 的第二个垃圾堆。
为什么必须是人做收敛？不是「AI 不够聪明」。是 AI 的架构上没有「什么重要」这个概念。会议录音、临时便签、收藏的别人的文章、真正有价值的项目经验——AI 对它们平等对待。人画边界，AI 填充内容——这个分工不是暂时的妥协，是结构性的必要。
铁律三：撞到边界才扩。 已在上一节阐明。
## 代谢四操作
不只是「删旧东西」。代谢是四个持续操作：

1. 淘汰：过时结论标记为 deprecated，不删除但降低检索权重

2. 合并：同质经验合并为新页面，旧页面归档

3. 提炼：从多起相似事件中提取可复用模式

4. 复盘回写：每次 Agent 产出被修正 → 修正记录回写对应的观点库或案例库

知识库的代谢不只是「积累经验」，更是「主动遗忘」。不能遗忘的系统会死于信噪比。
Karpathy 的「Perfect memory is a bug」在工程层面得到了完整印证——不是记得越多越好。淘汰是让确定性结论（「该方案已被证明错误」）和概率性判断（「这种方式可能有风险」）享有不同存储权重；复盘回写是把离散事件压成可泛化的行为模型。压缩不是妥协，是提纯。
周一上午的架构体检
不总结。做完这三步，比你读五篇文章管用。
第一步：Harness 审计（≤ 30 分钟）
打开 Harness 代码。数工具定义数量。>20 标黄，>40 标红。找三个「Agent 经常做错」的任务——如果输入-输出是确定性的（相同输入 → 相同输出），但它现在在 Skill 里由 LLM 处理而不是在 Code 里确定性执行 → 架构错误。标记出来，下周改一个。
第二步：Skill 路由审计（≤ 45 分钟）
统计 Skill 数量。>30 个？先把低频的退役或合并。逐条读 description——划掉所有「可以」「能够」「支持」，改成「当用户……时加载」。选一个最长的 Skill，删掉模型已知的正向原则，只保留 gotchas。跑一次，看结果是否比原来好。
第三步：知识库信噪比（≤ 45 分钟）
挑一个使用频率最高的知识库主题。逐条打分：A（当前有效） / B（过时，结论已被推翻） / C（噪音，临时笔记或未验证）。计算 A/(A+B+C)。<50% → 知识库已经是负债。B 类标记 deprecated，C 类删除或归档，A 中重复的合并为一条。然后写一条代谢规则写入 Agent 的 CLAUDE.md：「每月第一周自动运行知识库巡检，标出矛盾、过时、缺失条目。」
当行业还在争论模型参数量时，真正的 100x 工程师已经在实践这套架构原则。

M1 跑通 · 03

8h+一杯奶茶钱，从0到1开发一个商用EDA的平替

本文以 r2-soc-integration 这套系统的开发过程为例，演示如何利用AI实现一个这样的系统功能。它的功能是把各种芯片模块（IP）连接成一颗完整的芯片（SoC）。它没有图形界面，规则、表格、清单都以文件形式放在 Git 仓库里，客户可以直接改。设计上有一个基本要求：检查尽量放在组装阶段，让问题尽早暴露，不要等芯片做出来才发现。

整个项目在敏捷管理上分成八个 Epic——就是八个可独立验收的大需求单元：仓库导入（E1）、接口整理（E2）、规则连线（E3）、生成进网表（E4）、层级调整（E5）、IP 升级（E6）、合并冲突处理（E7）、发布检查（E8）。因为这八个 Epic 恰好依次排在客户使用产品的完整旅程上，我们内部也管它们叫"八站"，每个 Epic 是旅程上的一站。下面我按发布顺序讲：R1 先做 E1–E3（能正经开始连平台），R2 加 E4 和 E8 的一部分，R3 加 E5–E7，R4 收 E8 全量——一篇讲不完那么多刀，E1 我会讲得最细，后面各站讲每站特有的坑。

我用的方法一共五步。第一步，把需求问清楚，我们叫 grill——不是自问自答，是 AI 按决策树反过来问，直到模糊的地方都暴露出来为止。第二步，把要做的事写成一份带验收标准的文档，叫 spec。第三步，把工作切成小片，每片都能单独验收，叫 slice。第四步，每片先写一个会失败的测试，看着它失败，再写代码让它通过，这叫先红后绿。第五步，做完评审，该重构的重构。这套方法参考的是 Matt Pocock 开源的 skills 工作流，和 AI 辅助开发配套使用。整个项目没有人类项目负责人，追问我的、写代码的、评审的，都是被技能文件管着的 AI；我的角色是回答、确认、签字。

先说清楚：为什么是这个工作流，而不是别的

现在 AI 编程的 skills 和框架很多，选哪套得先说出理由，不然这篇分享就没有根。我选 Matt 这套，理由三条，都很实际。

第一条：AI 不缺能力，缺纪律。 市面上大多数 skills 是给 AI 加能力——让它会更多框架、会更多工具。但用 AI 写过项目的人都知道，它生成代码已经够快了，真正出事的地方是过程：没问清就写、没验收就算完、没评审就合并。这套 skills 是反过来的，它一样新能力都不给 AI 加，只强制四件事：先问清、先写合同、先看红、必过审。能力过剩的地方，纪律才值钱。 第二条：它把老师傅的判断力写成了 AI 必读的明文。 深模块、TDD、坏味道这些工程纪律，过去只能住在资深工程师的脑子里，靠言传身教。这套 skills 把它们写成了仓库里的规则文件，AI 每次开工必读，想违反就会被明文拦下。对我这种没有老师傅盯着的开发者，这等于把"有人会拦住我"这件事，从运气变成了配置。 第三条：它不接管我。 另一类方案是全流程框架（GSD、BMAD、Spec-Kit 这类），它们替你定义整个过程，你按它的节奏走。Matt 这套明确反对那种做法——它不接管流程，只在你自己的流程里装关卡，每条规则我都能读、能改、能删。区别就像：一个是替你开车，一个是给车装刹车和仪表。我要的是不失控的自动化，所以选了后者。

这三条不是我看完文档总结的，是这八站做下来验证过的。下面就从第一步 grill 讲起，你可以对照着看，每一条是怎么落到实际里的。

第一步：grill（AI 先把我问了个遍）

项目开工第一天，我没有写代码，先做了一件事：在仓库里装好 Matt 的 skills（npx skills@latest add mattpocock/skills），然后对 AI 输入 /grill-me。

这个技能的全部内容就三行话，作用是让 AI 进入一个提问循环：它沿着决策树逐支问我，不问完不许写任何代码。注意这里面的角色关系——不是我去问 AI，是 AI 来审我。整个项目没有任何人类项目负责人，grill 我的就是 AI 自己，而它之所以"会" grill，是因为那个技能文件命令它这么做。

这一步的价值一句话说清：它帮你解决"需求没对齐就开工"的问题，避免方向错了还高速返工。要说清一点：grill-me 的提问本身不含领域答案，它问的是结构性问题——交付物是什么、怎么算做完、什么不在范围内、失败了怎么处理；领域答案来自你和项目文档。它做的是逼你把答案说精确、写下来。E1 这一轮，我用它钉死了三个定义。

一是导入范围。"边界是什么、形状不符的仓怎么处理"这类问题必须答到底。很多团队在这里随口就是"全仓导入"——这个承诺没法验收，也没法维护。按合同，正确答案是"约定子集加最小实例数"：目录形状、IP-XACT 版本、VLNV 规则都有明确约定，形状之外的仓直接拒绝并说明缺什么。拒绝也是合同的一部分。

二是验收标准。"怎么算做完、判定依据是哪份文件"这一问，能筛掉大部分假验收。老项目普遍存在"四份真相"——硬件 top、软件头文件、验证 bench、Wiki 各写各的；没有唯一真相源，"依据哪份文件验收"本身就无解。所以合同规定 E1 的完成标准是"下游只能从库里取数"，而不只是"导入了"。

三是范围外事项。它会逼你把不做的事也列出来。最容易被误写进来的是仿真——按合同它归 E8 第四扇门，是回证手段，不是这一站的事。

AI 连着问了四十多个问题，没有一行代码产生。没有这一步，常见做法是按"全仓导入、仿真为主"先做两周再返工。Matt 这个方法的本质就在这里：团队里那个会追问的资深同事不一定存在，但 grill-me 把他的提问能力装进了仓库——AI 不是更聪明了，是被规则强迫着先把问题问完。

第二步：spec（AI 起草，我签字）

grill 的问答记录还在会话里，我接着输入 /to-spec。AI 把刚才那轮对话压缩成 E1 的合同章，固定五段：客户要（体感）→以便（业务结果）→合同格（机判坐标）→验收体感→边界。我逐条审，改了两个措辞，然后冻结进 Git，路径 specs/e1.md。

这份文档的关键部分值得展开说。约定子集定义到可机判：rtl/、ip/、vendor/ 三类顶层目录，IP-XACT 限 2009/2014/2022 三档，VLNV 四元组（vendor-library-name-version）库内唯一，最小 3 个 IP 且至少 1 个带 bus interface。形状之外的仓，import 直接拒绝并说明缺什么——拒绝也是合同的一部分。合同格 C3 拆成三条机判检查：schema 校验（失败 exit 2＋逐文件错误清单）、VLNV 唯一性（exit 3＋冲突对列表）、引用可解析（exit 4＋悬空引用路径），全过才 exit 0，产物落在 out/e1/catalog.json。

catalog.json 这个文件就是"统一设计库"的实体，下游 E2/E3 只认它，不认原始仓库。为什么要做到这个程度？客户那边有个真实例子：硬件 top 给 UART0 分配的基址是 0x4000_0000，软件头文件里写的是 0x4001_0000，Wiki 上还有第三个值。这种漂移在导入期看不出来——三份文件各自都"合法"，仿真可能都跑得过，问题要等芯片回来才炸。E1 把所有来源的地址声明收进同一个 catalog，E3 连线时只能引用 catalog 里的值，E8 的地址检查才有机会在组装期抓到重叠。真相源不统一，后面所有检查都是空转。

冻结在"只有我和 AI"的项目里意味着：文件进 Git，之后 AI 和我都不许绕过它。开发到第三周我动过"顺手支持全仓"的念头，正确动作是先改 spec、补验收，再写代码。spec 的约束对象不只是我，更是 AI：它每次开工都会读这份文件，想扩大范围，得先改 spec 并留下提交记录。这就是"规则进版本库"在单人加 AI 团队里的实际形态。

还有一件事：AI 参与写检查时，spec 的精度直接决定产出质量。不给 spec，AI 写的是"导入仓库，断言输出目录非空"——同义反复假测试，什么都没验。把 C3 那张表给它，它写的是"构造 VLNV 重复的仓，断言 exit 3 且冲突列表包含这两对四元组"。Matt 在 tdd 技能里专门给假测试起了名字（tautological tests），列为必须拒绝的反模式——这条规则写在技能文件里，AI 读到就得遵守。

第三步：slice（把合同切成一刀一刀的小片）

spec 冻结后，我输入 /to-tickets，让 AI 把 E1 切成可以独立验收的小片。这一步的核心讲究是纵切，不横切。

横切是按技术层切：第一刀做完全部 schema 校验，第二刀做 VLNV，第三刀做查询。这样切每一刀单独都验收不了——schema 校验做完了，客户感觉不到任何东西，三刀全做完之前每一刀都是半成品。纵切是按用户旅程切：每一刀都让一个最小场景从头走到尾。

AI 按这个原则把 E1 切成四刀：

T1 最小流通：单个简单 IP（无总线接口）

  走通 import → 校验 → 入 catalog → 查询返回

T2 带总线接口的 IP：引用解析（busDef/abstractionDef 可达）

T3 拒绝路径一：VLNV 重复的仓 → exit 3 + 冲突清单

T4 拒绝路径二：形状不符的仓 → 明确拒绝并说明缺什么

T1 是最小的"能流过去"，T2 补上真实 IP 的关键复杂度，T3、T4 把"拒绝也是合同"落成代码。任何一刀做完都能当场演示：跑一条命令，看一个结果。这就是"一刀一签收"。

每一刀还要带自己的最小红样例——动手之前先写好的那条失败验收。以 T1 为例：构造一个只含一个合法 IP 的测试仓，执行 import，断言 catalog.json 里出现这个 IP 的 VLNV，并且 r2c catalog get 能把它查回来。断言的对象是"客户能感知的结果"，不是"代码跑过了"。这条验收此刻必然失败——import 功能还不存在。失败就对了，这就是先红。

切片时 AI 也有自己的惯性要防。它最初给的方案是按层切的："先做校验框架，再做入库逻辑，最后做 CLI"。这是 Matt 在 tdd 技能里点名拒绝的反模式（horizontal slicing），理由很直白：横向切片的每一刀都无法独立验证，等能验证时错误已经叠了三层。我把技能文件里的这条指给它，它才改成纵切方案。这件事再次说明这个工作流的运转方式：纪律不在 AI 的自觉里，在技能文件的明文里，我的工作是当它偏离时把明文指出来。

切片顺序由 triage 机制定：依赖关系（T2 依赖 T1 的流通链）、风险（拒绝路径容易被漏，不能拖到最后）、每刀工作量（超过一天要再切）。四刀排进队列，挂上 needs-triage 标签，后面 omp 会话循环按队列取。还有一条规矩在这步立下：一刀未完不开下一刀——T1 没签收之前不许碰 T2。这条听起来慢，实际是防止"四刀都开了头、没有一刀能演示"的假进度。

第四步：tdd（先红后绿，一刀一刀做）

从 T1 开始进入实现。循环是固定的五步：锁死失败验收 → 看着它红 → 最小实现到绿 → 重构保绿 → 重跑拿新鲜证据。

第一步，验收已经锁死。第二步，看着它红：

$ r2c import ./tests/fixtures/single-ip

error: unknown command "import"

[accept] FAIL: import command does not exist (expected: catalog contains vendor:lib:uart:1.0)

这个红是合格的红：失败原因是"能力不存在"，不是路径写错这类环境问题。tdd 技能把这一步设为人工预确认——AI 必须把红的原因摆出来，我确认是"缺能力红"之后，它才准写实现。为什么要人确认？因为 AI 有一种本能：验收不过就修验收。这一道人工卡口，是整个先红后绿里我唯一必须亲手做的事。

第三步，最小实现到绿。技能文件管着 AI 的写法：只写让这条验收变绿的最小东西，不许顺手做 T2 的引用解析，不许搭"将来会用"的框架。第一版实现就三个文件：命令注册、目录扫描、catalog 写入，一百六十行左右。再跑，验收 PASS。第四步，重构保绿：合并重复的扫描逻辑、把硬编码路径提成参数，每改一次重跑一次验收，全程绿。第五步，隔一个小时环境里有其他改动了，把 T1 的验收原样重跑，还是绿，才算拿到"新鲜证据"——防止"当时绿了"的旧证据冒充现状。

T1 签收。T2 到 T4 重复同样的循环。整个过程里 AI 被技能文件拦住的典型动作有三个：想一次写完全部四刀的实现（横向切片，拦）；想写"断言输出目录非空"的假测试（同义反复，拦）；想先堆功能回头补验收（tests-after，拦——这个反模式有个准确名字：rationalization，事后合理化，它回答的是"代码做了什么"，而不是"合同要什么"）。

还有红的不正常情况。做到 T3 时有一次红了，但原因不对：不是拒绝路径没写，而是导入本身崩了。这种情况不进实现，走诊断六阶段：复现 → 最小化 → 假设 → 插桩 → 修 → 回归测试。最后定位是 schema 校验对空文件的 panic，修完补一条回归测试，再回到 T3。诊断和实现分开走，防止"边猜边改"把仓库改乱。

四刀全部签收后，E1 进入第五步。

第五步：code-review（双轴评审）

刀刀绿还不等于这一站做完。Matt 的 code-review 技能会起两组并行子代理，从两个轴同时审：Standards 轴（代码质量，基线是 Fowler 的坏味道清单）和 Spec 轴（实现与合同逐条对齐）。并行而不是串行，是为了防止两轴的上下文互相污染——看代码质量的容易被实现细节带跑，忘记对合同。

E1 评审抓出两个真问题。Standards 轴发现目录扫描里有一段对 vendor/ 目录的特殊处理——这属于穿层：导入层做了规则层的事。短期看是省事，长期看 E3 的规则系统上线后这段逻辑就成了没人认领的孤儿。按规矩标注为"抽象层违规"，当期重构回正确的层。Spec 轴发现 spec 要求"拒绝形状不符的仓时说明缺什么"，而实现对形状错误只报了一句 invalid repo，信息不全——按合同补上缺项清单。

两组子代理各出一份报告，我逐条裁决：接受、打回、或记入 ADR 留待后站。评审不是我目检代码——我只是裁判的裁判。

到这里，E1 才算真正做完：四刀签收、双轴评审通过、纸迹（spec、tickets、验收日志、ADR）全部在 Git 里。后面七站用的是同一套五步法，我只讲每站特有的坑和规矩。

E2 接口资产化：表驱动，还是正则英雄

E2 的任务是把脏 IP 打包成可连资产：第三方和遗留 IP 的接口千奇百怪，要先收成带命名治理的标准 BusInterface。这站有两条硬门，必须同时成立：表驱动打包（方法字段是查表语义）和项目 alias 回归双绿。

让 AI 写打包逻辑，它的第一反应是正则表达式。八十行正则，跑我们手里最脏的几个 IP，全过。演示效果一流——这就是为什么这站最容易被"聪明"毁掉：正则是逐案匹配，见到新形状的 IP 就得加分支，分支加到几十条，明年换个人维护，没人知道哪条分支为什么存在。表驱动是把命名规则落成一张显式的表：方法字段查表得到，表可评审、可 diff、可进客户流水线。正则英雄能过今晚的演示，过不了明年的人审。

这站最危险的不是 AI 犯错，是人自己嫌麻烦。常见动作：alias 回归红着，为了进度先把回归从门禁里摘出去，"回头再补"——然后这条线就"全绿"了。所以 E2 的红样例里专门钉了一条："只生成接口文件、删掉命名回归，必须红"。这条防的不是 AI，是人——它可以帮你避免"自己糊弄流程"：拆门禁这个动作本身就会让验收变红。骗过门禁最便宜的办法是拆门禁，所以门禁必须在验收合同里钉死。

E3 规则化连线：连接即源码

E3 是把 IP 之间的连接关系写成规则。核心一句话：连接即源码——连线规则是放在 Git 里、可 diff、可评审的文本文件，不是某个操作者脑子里的状态，也不是界面上的点选结果。

这站的检查全部红在组装期：规则引用了 catalog 里不存在的接口（未映射），红；两个 master 驱动同一个信号（双驱），红；两条规则抢同一段连接（冲突），红。AI 可以帮我写连线规则，但裁判是机器——规则进仓的那一刻就跑组装检查，AI 写得再自信，未映射就是 exit 非零。这就是我们海报上那句话的意思："让 AI 执行，让机器裁判，让人签核。"缺了裁判的 AI 连线，只是更快地制造静默错误。

顺带说一句"可定制"在这站的落地：规则、表、冻结清单、豁免策略全部是客户仓库里的文件。客户改规则就像改代码一样提 PR，失败红在他们自己的组装期。这比做一百个配置开关更接近"可定制"这三个字。

E4 glue 进网表：真实例，不是纸面

E4 是把连接结果生成进顶层网表。这站只有一条硬标准：生成物必须进功能路径——真实例化到顶层，下游仿真、综合拿的是这份生成物，不是一份"结构示意图"。

红样例是"生成物未进功能路径"。为什么要专门防这个？因为 AI 生成一堆看起来结构完整的网表文件非常容易，但这些文件如果没接进真实的功能路径，就是纸面资产——演示时能给人看，产线上没有用。验收不看文件长什么样，只看下游能不能消费它。这站也是"仿真不是主语"再次钉死的地方：仿真在这站之后才有资格说话，而且说的是回证的话，不是验收的话。

E5 层级手术：改骨架不丢线

E5 是改设计的层级结构：把某些模块提升到上层、下沉到子系统。难点是一句话：改骨架的时候，线不能丢。

这站的边界是约定操作集——支持提升、下沉等明确枚举的操作，不是任意层级 DSL。每个手术前后各跑一次连线一致性检查，前后集合必须相等。任何一条线在手术前后凭空出现或消失，检查必红。

这站还立了一个纸迹规矩：每次层级手术写一条 ADR（架构决策记录），记为什么做、影响面是什么、回退方案是什么。ADR 不是什么都写——有三条件阀门，只有影响跨层契约、影响下游站、或改变既有决策的手术才配写，防止 ADR 通货膨胀。E5 的六条 ADR 后来在 E6 救了我一次：升级影响分析时，有三处的层级位置直接查 ADR 就知道来历，不用翻提交历史。

E6 IP 升级：未映射必红，静默比报错更贵

E6 处理的是客户最痛的一个场景：IP 升级了，线还在不在？这一站做两拍：变更（改了什么）和影响（波及谁、必须红什么）。注意，E6 只有单路演进，不做两路合并——那是 E7。

这站的核心纪律是未映射必红：升级后的 IP 接口对不上旧映射表的，assemble/check 必须失败，而且报告要可读——哪条线、连的哪两个口、旧版本是什么、新版本变成了什么。红样例是"缺映射表时 assemble 仍 exit 0"。

为什么要用这么强的措辞？因为真实事故的形态是反的。AI 处理升级的默认行为是"尽量让流程走完"：映射缺了，它会按名字猜、按位置猜，猜对了没人知道，猜错了线就静默丢了——仿真不一定抓得住，等芯片回来就是事故。静默比报错贵，贵在生产线上。所以这一站我花最多时间的不是写升级逻辑，而是写"红得可读"：让每一次中断都精确告诉人该补什么。影响拍跑完，产出一份可审的影响列表——改了 3 个接口，波及 11 条连接，其中 2 条必须人工处理——这份列表是给人看的，不是给机器看的。

E7 Diff-Impact-Merge：冲突必停，合并的对象是模型不是文本

E7 是八站里理论最高的一站，也是我这个软件工程师本能被纠得最狠的一站。它处理两路变更的合并：我和同事（或两条自动化线）同时改了设计，怎么合？

先说我的本能错在哪。软件工程师处理合并的第一反应是 git merge——文本三方合并，能自动合的自动合，冲突了再说。这个本能在芯片设计里是危险品：两路变更在文本层面完全不冲突——你改你的模块，我改我的模块，git 干净地合了——但在模型层面，两路变更让同一条连接被驱动了两次。文本合并安静通过，芯片静默丢线。这就是"变更"和"合并"必须拆开的原因：Git 默认把两者揉在一起，而芯片设计里，变更的影响分析和合并决策是两件事。

E7 的验收有五条，缺一不可：变更列表可审、交叉影响/冲突列表可审、冲突样例必停、策略合并通过、合并后可细化。合并的对象是设计模型，不是生成的网表行；禁止用文本三方 merge 冒充模型合并。

红样例是"冲突被 silent clobber（静默覆盖）"。我做演示的方式是构造一次两路冲突：文本合并安静通过、模型合并当场拦停——同一个输入，两种结果并排放，不用多解释。AI 在这站的本能同样要拦：它第一反应也是对两个文本做 merge，必须把"合并对象是模型"这条规则喂给它，它才会先把两路变更都解析成模型操作，再在模型上做影响交叉。

E8 发布门：四扇门，一扇都不能拆

E8 是最后的放行检查，由四扇门组成，顺序固定，不可拆散冒充整门：

门一，DRC/waiver 可审计：设计规则检查的每一项豁免都有记录——谁豁免的、为什么、影响哪条规则。门二，IP-XACT 往返：设计导出成 IP-XACT 再导回来，关键字段一个不丢。门三，地址检查：mmap 重叠、或地址与连接关系不一致，必红——E1 收进 catalog 的那些地址声明，在这里统一算总账。门四，功能回证：约定的仿真当次跑绿、结果归因到 r2 的生成物、打出签收包。

"归因到 r2 生成物"这一条值得单说：仿真跑绿不算数，必须证明绿的那份网表是 r2_connect 生成的，不是手工 top。这直接回应四痛点里的"四份真相"——回证的意义是证明工具链的产出物本身经得起功能检验。三条官方测例的 PASSED 也只能挂在第四扇门下，关不掉前七站，更换不来"完整商业等价"。

发布不是按内部阶段发，是按 Epic 横切：R1＝E1+E2+E3（能正经开始连平台），R2 加 E4 和 E8 部分门（网表真、检查开始咬人），R3 加 E5、E6、E7（像平台组日常改 SoC），R4 是 E8 全量（可签字放行）。一刀未完不开下一刀。

omp：把上面这一切串起来的会话环

讲完整八站，补一下贯穿全程的编排机制。每一步实际都跑在 omp 会话环里：读队列里的 NEXT 节点 → 只做这一个节点的工作 → 跑 until-fail 观察 → 绿了就记下证据路径、取下一节点；红了就写清根因，只修这个节点——until-fail 停在 C4 时跑去改 C11，叫编排债务。

没有 omp 之前的常见做法，是自己写一批旁路脚本跑验收，进度看起来飞快。等正式会话环接上，until-fail 一跑，这类"绿了"的节点通常要红一半——旁路脚本验的是想让它验的东西，不是合同要验的东西。只能返工，把每个节点重新在环里跑过。这个教训和 E2 那条是同一个：自己验收自己，验出来的永远是好消息。

跨会话的状态靠 handoff：每次会话结束把进展、当前节点、未完成事项压缩成交接文档进 Git，下次会话从文档恢复，不依赖聊天记录。两个月做下来，换了几次模型版本，工作流没断过。

现在项目什么状态（按三档如实说）

最后按我们的三档读法如实汇报当前状态，这也是"范围凭证≠开发完成"的要求：

E1–E3 可演示：有可重复命令、产物路径、曾红过的验收。E4、E5 可演示。E6 的 unmap 红样例可演示，影响列表可审。E7 冲突停可演示，但验收五条里的"合并后可细化"还缺一组覆盖，按规矩这一站不算闭环。E8 四扇门都能跑，官方三条测例通过、挂在第四扇下，但功能回证的归因链还有一部分是半自动的。

所以准确的说法是：范围已钉死，R1–R3 的主体可演示，E7 收尾、E8 回证链加固是接下来两刀。完整商业等价没有宣称，也不应宣称。

结尾：八站走完，回头看开头那三条

开头说了我选这套工作流的三条理由，现在八站走完，可以对一下账。

"AI 不缺能力，缺纪律"——属实。 这个项目里 AI 写代码没掉过链子，掉链子的全是过程：需求没说清就想开工（grill 拦下）、想横向批量切片（tdd 拦下）、想写假测试（tdd 拦下）、想用文本合并冒充模型合并（E7 拦下）。四次事故隐患，没有一次是"它不会写"，全是"它想抄近路"。 "纪律写成了明文"——属实，而且比我预期的重要。 拦它的不是谁盯得紧，是技能文件里写死的规则。最说明问题的是 E2 那条红样例：想拆门禁的往往不是 AI，是人自己；能抓回来的，只有预先写进合同的红样例。规则管 AI，也管人，这才是"规则进仓库"的完整含义。 "它不接管我"——属实。 全程每条规则我都读过、改得动。E2 的红样例是我加的，E7 的"合并对象是模型"是我按项目情况喂给 AI 的。关卡装在我的流程上，不是我搬进别人的流程里。

所以这套方法给我的东西，一句话：它把"提问、合同、切片、测试、评审"这五件我明知该做、没人盯着就会跳过的事，变成了默认路径，而且这条路径的每一节护栏都归我所有。

这就是我做这个项目的方式。哪里说得不对、哪站想听更细的，欢迎当面问我——包括那些我做砸过的地方。

P2 R2-Arch · 10

r2-harness：效率来自反馈，持续交付可工作的内容

计划写得很漂亮。一集成，接口对不上。一验证，场景没盖住。到板级，软件接不住。不是人不努力——是反馈来得太晚了。

软件那边为什么搞出敏捷和 DevOps？就三件事：计划越来越不准；大家对需求各说各话；反馈太慢。硬件一样。静态计划应付不了变化；片子越做越复杂，验证越做越贵，错一次就很疼。

所以缺的不是更厚的计划。缺的是短反馈。效率来自反馈。交出去的东西，得是能用的——可工作的内容。

重点不在亲自码每一行实现，而在把运行底座搭稳：模型去生成、去回归、去定位失败；人守住边界——题怎么出、绿怎么算、最后谁签字。

ARM 的硬件 DevOps 转型，母本在 665-arm.md：Phase 1 Waterfall/Planned → Feature based；Phase 2 Automated integration。环 03 按那篇讲透。讲完再补一层今天为什么更急——不是改写 ARM。

芯片这边，我们把反馈拆成五环：探索、集成、交付、验证、进化。每一环都在给「能交出去、能接得住」的特性搭底座。底座先说清——Harness 不是流程口号，是 AI 和 EDA 中间那座运行时桥；交付拿 ARM 当镜子，不是当教条；转起来靠 Loop。九步怎么收敛、三刀怎么切、VPlan 怎么写，别处细讲——这儿先把五环怎么咬在一起说清楚。五环跑通之后，收成什么，结尾再钉。

让 AI 执行，让机器裁判，让人签核。

运行底座 · Harness：缰绳的另一头是镜子

先把定义钉死。Harness 不是「短反馈」四个字，也不是又一篇规范。它是概率性系统（AI）和确定性系统（EDA / CI / Oracle）中间的运行时桥梁——不做 AI 决策，也不替 EDA 跑仿真；只做三件事：信号转换、质量门控、状态管理。没有这座桥，AI 和 EDA 拼不起一个能存在的系统。它是智能体还在学习期的学步车——框架（五环）基本不动，车本身越坐越薄，最终要扔掉的是多余的搀扶，不是门禁。

要让搜索收敛，至少得先有三样：搜哪里、怎么快判、什么算过。落到运行时，这座桥上常跑四道门。

意图门控：人开口之后、搜索开始之前，先把约束收成可计算的边界——剪掉不可能的方向。

决策门控：AI 给出命令序列之后、真下发之前，再拦一拦语法、安全、预算。

反馈契约：仿真挂了，不能把原始 log 整坨扔回给模型；要收成 key metrics、错误摘要、收敛状态，机器才读得懂。

状态记忆：每次探索的意图→决策→执行→反馈，落成可检索的记录，别让同一个坑踩三遍。

MVP 至少要有意图门控和反馈契约；缺前者搜索空间不收敛，缺后者闭环断掉。上面再挂 loop，这些门才能自己转，不用你每轮手搓。标准人来写；门谁来跑，Harness 管。

写 Spec、定 Rule、设 Eval，表面上是在管 AI。真麻烦的是另一头：你得把以前只存在老师傅嘴里的规矩，写成机器也读得懂的句子。同事听「这块别动」能懂；Agent 会接着问——是不能读，还是不能改？改个名字算不算？你把问题答清楚，常常才发现：有几处自己根本没想通。所以写文档这件事，逼出来的往往不是执行速度，是判断本身——什么叫做好。这句话若不落成意图门控里能拦、能放的条款，贴在墙上也没用。条款谁写、门谁跑，可以交给 harness；过了门能不能放行，最后一票还是人签。

「什么叫做好」最怕写成一个好看的分数。只盯覆盖率到多少、用例过多少，人和模型都会往那个数上挤：行盖满了，关键条件还没摸到；断言能编过、仿真也绿，查的却不是你要查的行为。绿得越假，后面越贵。门禁要的是对得上意图的证据，不是一张总分表。人也别把「我这周做完了」当成签核——证据齐了、Peter 签了，才算出门。

环 01 · 持续探索：先碰，再冻

这一环对应的内核是 R2-arch：ICD 契约怎么收敛。早期信息少得很，别幻想一次画完拓扑。拿 VP、HLS、Emulator 去撞反压、死锁、QoS、CDC，把接口风险烧掉，收成机器能读的 ICD。

架构真正难的，不是写 RTL。什么钉死在硬件？什么留给软件？什么必须参数化？什么必须写成接口契约？需求是场景碰出来的，契约是架构收出来的。错误尽量留在左边，确定性交给机器。

接口不是一张端口表。管道管通道和带宽；阀门管什么能过、何时反压；翻译管下层差异别露给上层。九步收敛是基于loop的螺旋——路径、边界、三分、ICD、VP、HLS、Emulator、四维基准、冻结。细拆去看架构探索那课。每一步就问三句：试错贵不贵？机器吃不吃得下？有没有把错误往左推？机器吃不下，不算交付。

出口很清楚：关键接口有风险等级；ICD 工具能吃；Peter 在 Frozen 签价值；Andy 保证后面交出来的东西可测；Vicky 能挂上 Oracle；Fred 定节奏。

环 02 · 持续集成：六种进法，一份真相

这一环对应的内核是 R2-lean：删掉拖长反馈、却不增加确定性的流程。芯片侧怎么「进」还不漂，先把路径谱系钉死——不用每次从零发明轮子。

IP 进 SoC，先问三句。RTL 从哪来？软件栈要不要新建？里面有没有处理器？三问一组合，就是六种进法。随便一颗 IP，都能对上其中一条。六种都要 ICD，只是 ICD 从哪来不同。

#名称定义
①复用硬件、无软件栈成熟硬核 / 硅验证 RTL 直接挂（Hard IP）
②新建硬件、无软件栈RTL 从头写或按规范新做
③自动生成硬件、无软件栈从高层描述综合出 RTL；LLM for Chip 最常落这儿
④新建处理器核、新建软件栈自研核 + 整套工具链和软件生态
⑤复用处理器核、新建软件栈核用现成的，上层软件和驱动自己做
⑥软硬全复用子系统连同固件、驱动、中间件整包复用

有人交货的（①⑥，还有⑤里的核），先看包齐不齐。没人交货的（②④，还有⑤里的软件），集成要往左挪，空壳主要打这条。③最含糊——有工具产出，没有供应商，你得问工具链能不能复现。路径再花，最后都得在顶层收成四样：顶层 RTL、软件、接口契约、验证闭合。

路径定了，下一刀：单一真相。功能 Spec 编译成一份 SSoT——寄存器、中断、时钟、协议、地址、CDC、复位、DMA、QoS、安全、测试向量，全进同一份机器能读的东西。六条产线都从这儿长出来：

#产线一句话
01r2-reggenCSR 全链路一致，规格别漂
02r2-c2rtl算法类 IP，分钟级试架构、摸 PPA
03r2-cbbrtl在流片 proven 的模块上，长出敢用的 RTL
04r2-soc-integration可定制的 SOC 开放集成平台，像编译软件一样编译 SOC
05r2-buildsoc像管包一样管 IP 供应链和 CI/CD
06r2-ipqa选错 IP 的代价，tapeout 前就看见

改一处，六条线一起重生——硬工具落在集成面上，就是这个意思。SSoT 落地当天，设计、验证、软件、FPGA 四轨同时拿到能编的空壳。Stub First，第一天就能并行。CI 红绿门：参考模型和 RTL 逐拍比，对了放行，错了告诉你哪一拍。集成环交的不是又一份接口说明，是：改一处，门禁能判。

环 03 · 持续交付：ARM 怎么走过来的

持续交付可工作的特性——这是这条线的标题。

软件工程产生敏捷和 DevOps 的内在动机，同样适用于硬件：可预测性变差；缺乏对需求的一致性理解；缺乏快速反馈机制。

ARM 的两阶段转型：Phase 1，从 Waterfall / Planned → Feature based；Phase 2，Automated integration。

Feature based flow 的价值：从计划时间 / 资源，转向计划 feature 及交付所需的一切；提升可预测性，提升预测 ROI 能力；CI/CD 作为 feature 交付的稳定基础设施。

CI 构建的关键：ML regression——把回归测试优化至 50 个冒烟测试（提升单测比例）；关注 outcome vs time / resource；用 ML 优化 smoke test flow。

QA 环节——

Q1：feature complete？

→ ARM 已取消「feature complete」说法，改为「feature deliverable」。Complete 不等于 opt complete、test complete。

Q2：是否扔掉旧 RTL 重做？

→ 不会。旧数据留着做 estimation 和 reuse。feature by feature 交付需要以持续重构为前提，不是另起炉灶。

Q3：smoke test 太耗时？

→ 是。但它是小变更、短迭代的质量门禁。ARM 用 ML 优化 smoke test flow。

Q5：hardware 的 delivery 指什么？

→ 开发者向使用者交付（IP→SOC→板级→软件），通过持续交付获得持续反馈和改进。

Q6：是否所有员工基于同样基础设施？

→ 必要。保障团队间协作交互沟通顺畅，共享代码仓、实时看同样数据。

软件行业几十年——调研、规划、评审、动手——底下藏着一个前提：构建很贵，必须先想清楚再动手。PRD 评审、技术方案评审、季度 OKR，都是这个前提养出来的。流程长这样：发散 → 收敛 → 再发散 → 再收敛 → 交付。攒大版本，就是这套节奏的极端形态。

这个前提先在软件侧塌了。构建不再昂贵，结果又不再可预测——「先想清楚」越来越站不住。这是软件侧先看见的现象，不是拿一周上线硬套流片。

塌掉的是前提，不是行业。芯片这边，能跑的切片也便宜了。贵的变成争论本身：总线怎么选、ICD 怎么定，白板画半天，不如交出下游立刻能消费的东西——交付标准仍用 ARM 已经钉死的词：feature deliverable。以前的认知靠事前推导和文档对齐；现在靠可运行物和真实反馈。

组织节奏：做一点，发一点，及时规划。不攒大版本。冒烟往 lean、往那 50 个的方向靠；变更进得了稳定的 build。做 → 发 → 真实反馈 → 再做。真实反馈是下游接没接住、门禁绿没绿，不是会议室点头。

人还在。代码变便宜了，判断反而更贵；模型替不了你签字。AI 时代不是不需要思考，是思考不能再全部堆在动手之前。

持续交付可工作的特性。没有完成的状态，只有交付的状态——feature deliverable。

环 04 · 持续验证：怎么证明没错

这一环对应的内核是 R2-verismart。以 DUT 为中心。Harness 里 Agent 自己跑到机械 Oracle 全绿，人只在最后签核点拍板——这一票不能转交。

计划要是围着 testbench 写，列出来的全是组件，不是场景——那是 HOW 心智。一句话：VPlan 用 DUT 模型，不用 TB 模型。每条都回答「DUT 应该做什么」。WHAT 几个月才变一次，HOW 每周都在变。围着 TB 写的计划，保质期大概一周。Feature 怎么拆、DITL 怎么写，去看 VPlan 那课。这儿只说怎么关门。

验证计划第一页不该是 Feature 列表，该是使命声明：验什么、用在哪、质量指标、明确不验什么。不说清楚「不验什么」，签核就没有边界。

每个阶段五样东西，缺一不能关——代码覆盖率只能参考，不能单独签核：

  • Case 做完没有
  • 代码覆盖率（可选参考）
  • 功能覆盖率（必须达标）
  • 事件流覆盖率（DITL 关键路径）
  • 断言覆盖率

铁律：每条 covergroup，必须有对应的 checker。不是建议，是硬约束。只数车、不查货，答卷不可信。

Oracle 不是 AI，也不是 debug 工具。它是机械裁判，定义「什么算通过」：Golden / 参考模型逐拍比；形式等价；仿真里自动跑的 SVA 断言；签核用的结构化门槛——没有解释，没有商量，没有会议。第四类说清楚：裁判是波形断言（SVA 自动 PASS/FAIL），不是人肉盯波形，更不是 xwave 那条 debug 工具链。人手里真正重的那一下，常常是：裁判依据能不能被模型顺手改掉。团队自拟的对照答案，扛不住——生成侧可以顺着你的尺子长；独立参考、外部套件、结构化门槛，才站得住。人只干四件不能外包的事：写使命声明、拆 Feature、审覆盖模型、判签核。别的自动化。「我做完了」只是声明；Gate 全绿、Peter 签字，才是签核。

再钉两条：干活的和裁判的分开；裁判的和最后签字的分开。AI 可以把活跑满，机器把偏差测满，人只在边界签字。

环 05 · 持续进化：别手搓，写 loop

这一环对应的内核是 R2-harness——编排运行时本身。Harness 把标准立住。再往上，是 Loop。别一轮一轮手搓提示词了。搭一套系统：自己发现任务、派人、检查、记状态、决定下一步。你的工作是写 loop。它挂在计时器上，能拉起小助手，能自己往前拱——本质上是会自己跑的 harness。结构一旦通了，就别纠结工具名。设计一个哪儿都能跑的 loop。

一个 loop 要五样东西，外加一块记忆：

Automations：按节奏自动起来，做发现和分类。芯片里：定时看昨天 CI 挂了啥、issues、commits，值得干的写进清单。 Worktrees：并行别踩仓。隔离目录，并行修回归。 Skills：约定、构建步骤、「那次事故之后我们不这么干」——写一次，每次跑都读到。 Plugins / Connectors：接上 issue、仿真队列、覆盖率库、EDA 工具，才能真开门禁，别只会说「如果我能操作就好了」。 Sub-agents：一个写，一个查。自己审自己，永远太客气。能不能停，得另一个检查者说了算。 Memory：状态活在仓库里。模型会忘，仓库不会。

时间也得分层，别让反馈堵死。秒级用形式和断言；分钟级用黄金模型逐拍比；小时级跑集成仿真；天和周走发布火车 R1→R2→R3。前面三环的短闭环挂在分和时；签核和写回挂在天和周。

经验怎么写回去？看一天里实际转的那一圈——蒸馏的是执行链路，不是 harness。早上 automation 扫昨日 CI、issues、commits，值得干的进清单。隔离 worktree 里一个 sub-agent 起草修复，另一个对照 Skills 和门禁审查——自己审自己太客气，停不停得另一个说了算。connectors 更新票证；过不了的进人收件箱，人签了才放行。晚上把这条链路的 trace 和「这儿不对」拿去蒸馏（r2-mdistill）：抽成可复用的 Skill——删掉输出没变差就是废物，加上反而变差就是有毒。跨会话只记结论（r2-mretention），GOAL、PROGRESS 进仓，别记流水账。上线前拿三到十个样本做费曼检验（r2-mverify），验的是知识，不是 Agent 自评。跑起来以后对比新旧版本（r2-mevolve），完成率或有效产出掉了就回滚。你只设计了一次 loop，没有手搓每一步；关键决策仍是人签。

人还在回路里

Loop 改的是工作形态，不是把你撤了。没人看着的 loop，也能没人看着地犯错。「完成了」只是它自己说的。你要发布的，是你亲手确认过能跑的东西。

放任不管，你对代码库的理解会烂掉。loop 越快，你没碰过的代码堆得越高。没别的办法：得读产物。

带着判断去设计 loop，它帮你；拿它逃避思考，它害你。两个人能搭出一模一样的 loop，结果完全两样——一个在自己懂的事上跑更快，一个用它躲开理解。Loop 分不清这两种人。你分得清。

去搭 loop。但要像打算继续当工程师的人那样搭——不是只按启动键的人。

收束

短环交出去，机器判得了，人签得了——其余都是这座桥上的零件。岗位从「把图画完」挪到「把题出清、把结果判住、把责任签下」。可预测性 = 短交付环 + 自动门禁 + 真实反馈；真实反馈是下游接住、门禁绿、证据在，不是会上点头。

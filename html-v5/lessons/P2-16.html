r2-soc-integration：可定制的SOC开放集成平台

八站合同 × 两条理论 × 一套纪律 · 内部培训讲义

读者：内部团队（开发 / 集成 / 商务·PM） 素材：八 Epic 罗盘（31）· 客户满血路径复盘（41）· 三轴相乘（43） 一句话：这篇文章教你用同一套语言讲清 r2-soc-integration 在卖什么、怎么签收、什么不许说。

一、这个品类到底在卖什么

先把名字摆正。r2-soc-integration 是品类伞，伞名下挂着一条路线——SoC 集成的开放平替路线。这条路线当前的主刀产品叫 r2_connect：一个命令行（CLI）形态的 Connectivity 工具，没有图形界面，对标的是 Magillem Connectivity 的日常岗位工作流，而不是它的商业全家桶。

品类名「可定制的SOC开放集成平台」里每个字都有重量，不是营销修辞。可定制不是指开关特别多，而是客户能把规则、表、冻结清单、豁免策略放进自己的版本库与流水线；开放不是指什么都能灌，而是失败必须红在组装期、能被审计；集成平台是指它管的是从主仓到发布门这一整段连接旅程，不是一个点选小工具。

客户买完之后的一天长什么样？罗盘开篇有一句验收总句，建议每个人背下来——它能当一把尺子，量出任何一个 Epic 合不合格：

我把仓灌进去 → 脏 IP 打成接口 → 规则把平台连上 → glue 进网表 → 改层级不炸 → 升 IP 看影响 → Diff-Impact-Merge（变更-影响-合并）冲突能停 → 检查过了才能放行。

记住这条旅程，后面八节都是它的展开。也记住一句更冷的纪律，海报页脚写的：范围凭证 ≠ 开发完成。这句话在内训里要反复出现，因为它防止我们把「门绿了」念成「产线齐了」。

二、为什么存在：四痛点与两条理论

产品不是凭空长出来的，它是对四个现场痛点的正面回答。这四个痛点不是文案，是验收设计输入——每一个痛点都对应后面某个 Epic 必须堵住的口子。

  • P1 异构难装：第三方和遗留 IP 接口千奇百怪，不可能假设它们到集成时已经是干净的 BusInterface。回答它的就是 E1/E2——先把脏 IP 收成可连资产。
  • P2 四份真相：硬件 top、软件头文件、验证 bench、Wiki 各写各的，真相在漂移。回答它的是 E1 统一设计库 + E3 连接即源码，禁止多份手工 top 当真理。
  • P3 升级静默丢线：IP 一升级，线就悄悄没了，等硅后才发现。回答它的是 E6 的影响拍和 E7 的冲突停——不许「合完再看仿真」。
  • P4 封闭难定制：黑盒点选流程签不进客户流水线。回答它的是规则/表/CLI 全进 Git，整条旅程可审可改。

围绕这四个痛点，产品理论只保留两条主线，服务后面所有站。

第一条是 Capture → Check → Generate（捕获 → 检查 → 生成）：先让主仓、接口、规则、设计库成为同一真相源；再在组装期把未映射、双驱、冲突、地址重叠全部变红；最后才是真生成物进网表。仿真和细化是 Check/Generate 之后的回证手段，不是主语——这一句很重要，否则产品会被讲成仿真加速器。

第二条是 变更 → 影响 → 合并（Diff-Impact-Merge）。这是麦哲伦类平台相对软件 Git 的关键分叉，也是内训里最容易讲错的地方。三拍分别问：改了什么、波及谁必须红什么、两路变更怎么合。一定要钉死一句：影响 ≠ 合并。E6 是「一个变更在树上炸开」，E7 是「两个人同时改怎么合」——把 E7 讲成「对 top.v 做 git merge」，就是对理论不准确，因为 Git 默认把变更和合并揉在一起，芯片上会静默丢线。

三、合同就是八个 Epic

这是全文最重的一节。先讲一个反面教训：如果文章的主叙事是「scoreboard 全 industrial 了、三测 PASSED 了」，客户学到的是内部机判关闭术，而不是他买了什么。所以合同必须长成八个 Epic，而不是长成脚本清单。每个 Epic 都是合同等价章，结构固定：客户要（体感）→ 以便（业务结果）→ 合同格（机判坐标）→ 验收体感 → 边界。

下面是八站速览，合同格是机判坐标（C0–C11），但对外说话只用 Epic 语言，不用坐标。

Epic客户体感合同格一句话边界
E1 主仓入库统一设计库，同一真相源C3约定子集+最小实例数，非任意全仓
E2 接口资产化脏 IP 打包+命名治理C0+C1为连接服务，非完整商业 Packaging
E3 规则化连线连接即源码，可 diffC2协议以冻结清单为准
E4 glue 进网表真实例进顶层，非纸面C4仿真是验收手段非主语
E5 层级手术改骨架不丢线C5约定操作集，非任意 hierarchy DSL
E6 IP 升级变更+影响，未映射必红C6单路演进，≠E7 两路合并
E7 Diff-Impact-Merge冲突必停，模型合并C7禁文本三方 merge 冒充
E8 发布门检查→交换→地址→回证→签收C8–C11四扇门不可拆，≠仿真条数

八个站里有四个最容易假绿，内训必须单独点名，把它们的负例讲到能背：

E2 接口资产化。最容易被聪明毁掉的一站。正则英雄脚本能过今晚演示，过不了明年的人审。硬门是两条同时成立：表驱动打包（方法字段必须是查表语义，禁正则当交付内核），且项目 alias 回归双绿。只生成接口文件却删掉命名回归，不算完成。 E6 IP 升级。核心是「未映射必红」。缺了映射表，assemble/check 必须失败且报告可读——静默比报错更贵。有人把分支文本合并当本站，属于范围错误，拒收。这一站只到「变更→影响」，还没做两路合并。 E7 Diff-Impact-Merge。验收五条缺一不可：变更列表可审、交叉影响/冲突列表可审、冲突样例必停、策略合并通过、合并后可细化。合并对象是设计模型，不是生成网表行。软件同学本能把变更和合并揉在一次 Git 合并里，芯片上会静默丢线——这是麦哲伦对照材料反复强调的分叉。 E8 发布门。四扇门不可拆散冒充整门：DRC/waiver 可审计、IP-XACT 往返关键字段不丢、mmap 重叠或与连接不一致必红、功能回证（约定仿真当次绿 + 归因 r2 生成物 + 签收包）。三条官方测例通过，只应挂在第四扇「功能回证」下，关不掉前七站，也买不来完整商业等价。

发布按 Epic 横切，不按内部阶段名：R1=E1+E2+E3（能正经开始连平台）→ R2=+E4+E8 部分（网表真、检查开始咬人）→ R3=+E5+E6+E7（像平台组日常改 SoC）→ R4=E8 全量（可签字放行）。一刀未完不要开下一刀，第一刀没完成就去讨论官方十条仿真，是最常见的焦虑跳步。

四、开放可定制：不是开关多，是规则能进 Git

「可定制的SOC开放集成平台」这句品类名，内训里要拆开讲，否则会被理解成「配置项很多」。

可定制的可靠定义是：客户能把规则、表、冻结清单、豁免策略放进自己的版本库与流水线，而不是锁死在某个界面点选路径里。所以规则源码化、表可评审、豁免可审计，比再做一个配置向导更接近品类承诺。设计配置项时问两个问题——这个配置是否进入客户可审文件？失败时能否红在组装期？两个都否，多半只是内部调试旋钮，不配写进可定制故事。

开放的另一面是拒绝虚假开放。任意协议、任意全仓、任意一键，看起来开放，其实是把失败从售前搬到售后。罗盘在入库与连线边界写死「约定子集」和「冻结清单」，不是保守，是工程上唯一可签收的开放。

智能这个词也要钉死。海报原话是「让 AI 执行 · 让机器裁判 · 让人签核」。允许 AI 执行连线，但必须有机器裁判（组装期必红）和人签核（发布门）兜底。缺裁判的 AI 连线，只是更快地制造静默错误——这不是智能化，是风险加速器。

五、边界与禁止话术

内训最容易翻车的地方不是讲不清能做什么，而是讲不清不做什么。边界同样是理论的一部分。

合同永久排除项，背一遍：GUI Design Editor（C12）、企业多用户/权限/Design Server（C13）、Arteris 认证、任意 SoC 一键灌入、完整商业 Magillem Packaging/Registers 全矩阵。后两项要单独说：本图只签 Connectivity 主刀，r2-soc-integration 三件套里的 r2_package 和 r2_registers 全量不在八 Epic 图内，需要另签。

对外材料三句禁令，建议贴在每个工位：

  • 不说「完整商业无差别等价」——宣称无差别全量等价不是激进，是欺诈风险。
  • 不说「任意 SoC 一键」——约定子集 + 冻结清单是唯一的可签收开放。
  • 不说「仿真条数 = 满血」——仿真最多服务 E8 第四扇，买不来全书。

还有一句页脚纪律要在每次汇报前默念：范围凭证 ≠ 开发完成。罗盘只是范围与使用地图，不宣布开发完成。客户介绍材料若同时出现 scoreboard 全绿和这句纪律，使用者应以后者为准解释进度。

六、纪律：先红后绿 + 从第一天编排

这一节是给开发者的，但商务/PM 也要听懂，因为它是判断「这个站到底做没做」的尺子。

方法轴的权威是 ATDD（验收测试驱动），铁律一句话：没有会失败的验收，就没有合格的实现。顺序必须是五步——先写/锁定会失败的验收，看见它因缺能力而红（不是 typo 红），再写最小实现到绿，重构保持绿，宣称完成前重跑同一验收拿到新鲜证据。把「我先把功能堆完再补门禁」叫 rationalization：tests-after 回答的是「代码做了什么」，tests-first 回答的是「合同要什么」。每个 Epic 都要有一条最小红样例贴在看板，比如 E6 是「缺映射仍 exit 0 → 失败（必须红）」，E7 是「冲突被 silent clobber → 失败」。

编排轴的权威是 omp（会话循环挂交付图）。正确一圈是：读 NEXT 节点 → 只做该节点 WORK → until-fail 观察 → 绿则记证据路径，红则写清根因且仍留在同节点 → 取下一节点。有一条常被忽略的纪律：红则只修同节点。until-fail 停在 C4 时，去改 C11 或去抬 C0，是编排债务——正确做法是补 C4 的证据或实现，再 until-fail。

执行轨迹上真实发生过的四个偏差，内训当镜子照，别重交学费：

  • 主语后置：先追注入和三测，被客户追问「更关心矩阵吗」才扳回八站。标准轨迹第一天就锁体感总句。
  • omp 后置：大量旁路脚本自产自销，入口来得晚，后期 until-fail 像在装证据。标准轨迹第一天就是交互会话 + NEXT + until-fail。
  • 抬级粘贴：scoreboard 可写 industrial，但若无验收红绿故事，机判绿而过程不可教。工业级只能出现在例子已红再绿之后。
  • 语言被坐标替换：对内 C0–C11 方便机判，对外脱离 E1–E8 客户听不懂买了什么。Epic 是范围凭证语言，C 格是机判映射。

七、两种角色的一天

开发者。任选一个未闭环 Epic，先写一条必定失败的验收，跑出红，再动实现；今日禁止先改产品后补门。编排只通过 omp 取 NEXT，禁止开新的旁路大脚本自己验收自己。完成前重跑该节点验收，贴日志，才许在群里说「E? 可演示」。 使用者。不必会写 DAG，但必须按旅程用、不按库名用：E1→E2→…→E8。四条红线：无主仓/无约定仿真权限不许说主验收完成；生成物未进功能路径不许说 E8 回证完成；不找 GUI，规则/表/报告才是界面；汇报只说 Epic 与产物路径，不说完整商业等价。

读「范围凭证」要分三档：可演示（有可重复命令 + 产物路径 + 曾红过的验收）、仅机判（有 cells/evidence 文件但日常 CLI 无人会跑）、未开始（只有设计文档）。签收会只接受第一档。经理问「满血了吗」，正确回答模板是：E1–E3 可演示；E6 unmap 红样例可演示；E7 冲突停仍缺 ATDD 红样例；E8 有官方回证但不能单独代表全书——总体范围已钉，产线未宣称商业全量等价。这比「scoreboard closed=true」更负责任。

八、收束：可接力才是真交付

脚本会换，人会变；能留下来的，是八站合同语言、验收加工链和会话循环。客户现场最缺的不是更多特性演示，而是把验收写成先于实现的公共语言。机判关闭可以很快（数小时靠既有产物），组织方法论迁移很慢（几周的习惯斗争）——交付后者，才是真交付。

若你只改内训的一页 PPT，请只放三行：

  • 合同 = E1…E8（变更-影响-合并在 E6/E7）
  • 先红验收，再写实现；omp 从第一天编排
  • 仿真回证 ⊆ E8，≠ 满血全书

可定制的SOC开放集成平台不是口号，是客户能审的规则与表，是组装期能红的检查，是变更时能停的冲突，是发布时不能拆的四扇门。把这些做成可接力的操作系统，比堆出一次漂亮演示更接近工业。

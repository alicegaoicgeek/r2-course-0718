r2-ip：集成一颗 IP 进 SoC，结构是因，复用是果

先把结构钉死——交付边界、接口契约、空壳占位、EDA 门禁——后面的复用和 AI 填空才站得住。结构是因，复用是果；别反过来赌。

行业钉子先钉上：Wilson 2024，首次流片成功率掉到 14%（约二十年最低），六成以上项目至少要一次 re-spin；验证常吃掉工程精力的 60%–70%。集成阶段最贵，正因为接口一漂，后面全赔。

第一层 · 决策：三个问题定路径，交付边界定打法

把一颗 IP 集成进 SoC，先问三个问题：RTL 从哪儿来、软件栈要不要新建、含不含处理器。三个问题一组合，就是六种集成路径。验证策略和风险分布各不相同，但核心区别只有一个：有没有明确的交付方。

#RTL 来源软件栈处理器交付边界
1外部供应商（硬核/硅验证）无不含有
2手写 RTL无不含无
3工具生成（HLS/DSL/LLM）无不含模糊
4自研处理器核新建自研无
5外部处理器核（Arm/RISC-V）新建复用有（核）/ 无（软件）
6子系统整体打包复用复用有

任意一颗 IP 怎么进 SoC，都能对上其中一条。六种都要 ICD，只是 ICD 从哪儿来不同——不是「有了 IP-XACT 就覆盖六种」。

打法按开关分岔：

  • 有交付（①⑥，⑤的核）：走供应商包 / Day-1 评估，先问交付物齐不齐。
  • 无交付（②④，⑤的软件）：没有外部交付包可开箱，集成要左移——后文空壳主要打这条。
  • ③ 标「模糊」：有工具产出，但没有「供应商」这个角色；别拿供应商 QIP 原样套，要改成「工具链可复现」那套问法。空壳策略仍然适用。
「有没有交付边界」不是贴标签，是总开关。 后面怎么评估、能不能先挂空壳、风险从哪爆，全看这一列。

第二层 · 交付：把两个问题解耦，接口即契约

无外部交付包的那几条路（②③④ 为主），痛点一样：IP 写得久，集成撞得晚。等 RTL 齐了再塞进 SoC，接口不对、时钟复位配错、总线协议有坑——返工极贵。

解药不是「把 IP 写快点」，而是先挂一个最小可用版本占位，把两件事拆开验：

  • 这颗 IP 本身对不对
  • 它能不能接进这颗 SoC

空壳三种，成本往上走：

  • Stub — 只验总线、地址、中断（半天）→ 保「集成通」
  • Behavioral Model — 算法类 IP 早期 SoC 仿真（约 1 天）→ 仍偏集成与行为占位
  • Golden Reference — 最小可跑的功能实现（1–3 天）→ 已能验功能链、驱动、数据流，但接口仍冻结

前两档不求功能齐，只求连接、编译、仿真、驱动框架能跑起来。Golden 功能更多一点，规矩不变：接口一定，空壳就不能改。 空壳就是契约；后面真 IP 必须按它接。破了这个规矩，空壳阶段跑通的 case，真 RTL 一接全废。

契约写在机器可读 ICD 里（需要时再投影成 IP-XACT 给工具用）——ICD 管「怎么接」，不管「RTL 大半自动长出来」。

IP-XACT 方向对，但必要不充分，标准到工程还有 usability gap；包装格式不是契约，也别拿包装格式当银弹。

方式 3 多一个方式 2 没有的便宜：Python/C 算法模型本身就是最准的行为空壳；方式 2 往往还要另花一天写 Behavioral Model。I/O 格式即行为契约，后续生成 RTL 必须对齐它。总线、地址、中断那一层，该挂 Stub 还是挂。

第三层 · 协作：四方对齐是工程方法，不是管理动作

空壳只解决「接得进去」。要信「功能对」，还得算法、RTL、验证、FPGA 原型四方先谈死一套输入输出标准文件——算法调着用、RTL 验着用、板上跑也用同一套。

这事靠开会对齐口径没用。标准不是 PPT，是同一份 case 拿去跑 EDA：仿真、比对、板上回放，谁过谁不过，工具说了算。四拨人各写各的实现，最后都得在同一套 EDA 门禁上撞绿；绿了才算齐，红了就回去改。机器可读接口描述的用处也在这里：连通性、地址冲突这类检查可以自动跑——它帮你把「接得上」验死，不代替功能对错。

成熟团队的一种落地：RTL 还没冻之前，先挂 Virtual Platform 当参考模型，塞进 UVM Scoreboard 当 predictor，四方都拿同一个 VP 在 EDA 里打分，别各养一份手写 SV 参考。

对齐不是开会对齐说法，是同一套 case 跑 EDA。

第四层 · 并行化：解耦 + 对齐 + shell 不变，才跑得起来

并行化靠前三层叠出来：空壳把「对不对」和「接不进」拆开（第二层），四方用同一套 EDA 门禁打分（第三层），各条线才能同时往前推。

再钉死一条：验证环境的 shell（脚本、目录、CI）不允许变，变的只是 DUT。最后真 RTL 接进去时，regression 挂了只可能怪 RTL，不能怪环境。空壳换成真 RTL、RTL DUT 换成 HLS DUT，都插进同一个不变的 shell——独立验证问题才能真并行。

量化要诚实：自动化、并行化、少返工各有独立工业数据；三者叠加缺对照实验，别把独立的数加成一个神话倍数。

第五层 · 自动化：缩短反馈回路，且止于功能正确

自动化的本质不是省人力，是缩短「改完 → 知道对不对」的回路。

算法类 IP 尤其合适：mem 进 mem 出，同样输入，硬件结果跟参考答案一比就见分晓。团队经验里，同一 LLM 写 Python 正确率远高于写 Verilog（大约 95% vs 不足 60%）——所以方式 3 的策略很简单：Python 当 Golden，用验证倒逼生成 RTL。 不是 Python 比 Verilog 高级，是用准的考不准的。

回路能压到多短，要划界：模块级功能比对、连接性检查，分钟到小时级常见；全片回归仍是天级（Wilson：基础全片测在大集群上可以连跑五天）。短的是局部闭环，不是整颗芯片一键三分钟。

同一套 UVM 环境和用例可以不加修改地打在 RTL DUT 和 HLS DUT 上，Scoreboard / Reference Model 复用——这就是「Python 参考答案 → RTL 比对」在 UVM 里的样子。日常打法就三句：空壳先行、四方用 EDA 对齐、红-绿-重构；CI 门禁自动跑，不追求一次 feature complete。

智能体在这里的活也说清：它解决的是「feature 交出来功能对不对」——机制是 Python 参考答案 + 逐拍比对 + LLM 迭代，不是让 LLM 直接判 RTL 对不对。

边界必须钉死：四方对齐 + CI 全绿 ≠ 芯片可交付。 功能正确是必要条件，不是充分条件。时序、面积、功耗、覆盖率、软件兼容性还要另外闭合。CI 全绿只说明功能迭代跑得快，不说明可以流片。流片成本是硬约束——业界现实是大瀑布 + 小敏捷，不是纯敏捷把芯片跑成互联网产品。

第六层 · 可复用：在既定结构里填空，靠继承不是靠信

芯片容错率极低。面对「AI 生成 RTL」，真问题不是「它能不能写」，是「你敢不敢用」；若你觉得不对，有没有办法把它逼到对。敢用的底座，和写对的根，是同一样东西：标注——叠在已流片的结构上。

CBB（Common Building Block）= 公司内部已流片、可反复复用的标准 IP（UART / DMA / DDR…），带统一接口和完整交付包。 标注 = 在 proven 资产上把专家判断写成 Agent 啃得动的四张映射：
  • 特性映射 — 需求 ↔ feature point ↔ 代码（到模块/文件/接口，不必神话到「行」）
  • 设计模式 — 这块用 FSM / 握手 / FIFO / CDC…
  • 架构决策 — 为什么这么设计 = 不可破坏的约束
  • 验证环境映射 — feature point ↔ testbench / golden / scoreboard

新需求多半是旧 proven 设计上的改动：特性映射告诉你改哪一块；标注里的选型与陷阱告诉你哪个 CBB 在新环境里安不安全；架构决策守住不能碰的边；只重验波及面。没碰的，继承 proven；碰的，尽量是一小块；真正要新写的那一小块，才交给 AI。

到这一步，坑位已经留好——AI 不是来「创作」，是在既定结构里填空。旁边摆着同模具、同接口、已流片的 CBB，照着改局部差异。写对的概率靠三件事在当下锁住：

  • 锚定 — 同结构下写对的 CBB 就在眼前
  • 缩小 — 要填的是切出来的小块
  • 约束 — 接口、模式、边界已钉死，只能动那一点

不靠微调模型，靠继承、上下文和约束。改了哪、为什么、波及谁，标注可追；结构和边界你定，你掌舵；交付靠的是你的 CBB、你的结构和蒸馏进标注的判断，不是把芯片赌给黑盒。

这不是让你信 AI，是把「敢用」建在你本来就信的东西上。
智能体不替你决策，是在你定的结构、你的 CBB、标注里的判断里填空——结构是因，复用是果。

收束

三个问题定六条路径；交付开关定打法；空壳把「对不对」和「接不进」拆开；四方用同一套可执行文档和 EDA 门禁对齐；shell 钉死才能并行；短回路把功能迭代跑快，但 CI 全绿到不了流片；最后在钉死的结构里用 CBB 和标注让 AI 填小空。

六层是一条链，不是六个并列观点。LLM 的价值不在替你走完哪一层，而在结构、CBB、标注已经齐了之后，去填那个被缩小、被锚定、被约束的空。

试错成本足够低，才敢改；敢改，才能迭代；能迭代，才能收敛。

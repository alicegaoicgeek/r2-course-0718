P1 R2-AIOS · 08

r2-4role：Peter 设目标，Andy 做设计，Vicky 当裁判，Fred 建产线

AI 能写 RTL。这事不用争了。但写完 RTL 离流片还差多远？

差三个环节。不是代码质量的问题——模型写的 RTL 已经够好了。是流片这个动作，需要四个节点同时到位：定目标、做执行、测偏差、根据偏差调策略。AI 只覆盖了「做执行」这一个。另外三个——谁来把「做好」定义清楚？谁来测实际产出和定义之间的偏差？测出偏差之后谁来改策略？——这三个是空的。

本文从「已有 r」开始推理——r 怎么产生，是上一层的事。目标不是凭空调出来的：它要被写成可度量的话，要列清靠谁、谁会拖后腿，要说清这些人的行为要发生什么变化。这套产生 r 的机制叫影响地图（Why → Who → How → What）：它产出 r 的候选和到达目标的最短路径，本文的六层洋葱从它的输出开始剥。先钉一句话在这里：没有 r 的产生机制，回路收敛的可能是错误的地方。

芯片流片的收敛，本质是一个负反馈控制回路。设定目标 r → 执行器产出 y → 传感器测偏差 e=r-y → 控制器根据e调整策略 → 新产出 y' 趋近 r。从恒温器到PLL锁相环，工程世界所有需要稳定在某个点的系统，靠的全是这个回路。

这条回路在芯片研发中映射为四个角色。不是四个岗位——是控制回路的四个不可消除的系统函数。缺一个，回路断开。

Peter = r，参考输入。 r 不是「写一个 UART」——那是 prompt。r 是量化到可独立测量的规格：「latency ≤ 3 cycles，面积 ≤ 0.01mm²，兼容 16550A 寄存器映射，覆盖率 ≥ 98%」。没有量化就没有 r，没有 r 回路不知往哪收敛。Peter 的职责是价值判断——验什么值、好到什么程度、在 PPA 三角里放弃哪个。 Andy = 执行器，产出 y。 从 VP 到 Stub 到 RTL 到综合网表，每层产出必须可被 Vicky 独立测量。铁律：y 不能是黑箱——产出带接口契约、带验证入口、带可追溯的抽象层级标记。 Vicky = 传感器，测量 e = r - y。 核心原则——「控制器不兼任传感器。」 Vicky 不做价值判断、不调参数。她只做一件事：用机械 Oracle 测量偏差，输出 e。Oracle 是数字判定器——Spike diff、形式等价、DRC、Timing——零模型参与。Maker 和 Checker 合并的后果是致命的：模型对自身产出天生宽容，幻觉以「自信输出」渗入 RTL，偏差从不被看到，回路名存实亡。

两个 e 不是一个 e。e_eng（工程偏差）：规格 r 减去产出 y，机械 Oracle 测，零模型参与，判"对错"——这是 Vicky 的全部辖区。e_goal（目标偏差）：期望的行为改变减去实际发生的行为改变，判"相关"——它含价值判断成分，不归传感器，由回图检验机制读取。把 e_goal 塞进 Oracle，等于让温度计评价天气。

Fred = 控制器，根据 e 调参重排。 收到 e 后，Fred 不是手工看报告改 prompt。他跑的是 Dynamic Workflow：根据 e 的方向和量级，实时调整 Andy 的执行策略、重分配 Agent 资源、切换工具链路径。

四个角色，一条回路。Harness = 验证管道 + 编排调度器 + 状态记录器的工程集成体。它的核心判据：每次产出的偏差可被独立测量，测量结果可回溯驱动下一次产出。「没有验证管道和 Harness，AI 只是超级个体玩具。」

r 不是 prompt。Prompt 是"写一个边缘 AI 加速器的 MAC 阵列"——AI 能生成一百种。r 是"面积 ≤ 8mm²，功耗 ≤ 500mW，帧率 ≥ 30fps，代价可以牺牲 TOPS"。Prompt 回答做什么，r 回答做到什么程度算好、放弃哪个可接受。前者任务描述，后者价值判断。

Peter 定 r 的方法论：六层洋葱图。从外向内剥：企业战略（为什么做这颗芯片，TAM 多大，复用哪块 IP）→ 产品愿景（赢什么指标，代际差锚在哪个体验跃迁）→ 产品战略（哪代工艺、哪个架构路线、放弃哪个维度）→ 发布计划（分几个版本，每个验证什么价值假设）→ Sprint 计划（本轮收敛什么指标）→ 每日计划（今天拿到什么信号）。每层回答一次"值不值得做"——在"怎么做"之前。六层不是流程 checklist，是六次价值追问。

产品战略层落在一个动作：PPA 三角裁决。Power、Performance、Area——三者不可兼得，必须明确放弃一个。算法给你帕累托前沿——连续曲面上所有可行解。但选前沿上哪个点，是离散决策：放弃面积保功耗？放弃 TOPS 保帧率？这里有生态兼容性、客户迁移成本、团队能力边界——无一可建模为损失函数。这是人的裁决。

Peter 的裁决权有两层，不要混。内层是 PPA 三角——选前沿上的哪个点；外层是 r 级裁决——这个里程碑做什么、不做什么（Now/Not Now）、目标变了要不要重画地图。Now/Not Now 不是"这个想法不好"，是"它不属于当前的目标"。

还有一处诚实边界必须写明：从业务目标到量化规格的那一步翻译，靠价值判断，没有机制保证，不可自动化——它和"价值归人，搜索归模型"是同一条原则的两端。

R2-Edge 把这个逻辑打到了最硬的地面。（内部实验案例，TSMC 12nm，智能摄像头应用）

Sprint 1：全展开 MAC 256 并行阵列，目标 TOPS 碾压竞品。全挂。面积 9.2mm²（超约束 15%），功耗 580mW（超 16%），帧率 25fps（未达标）。三条根因：权重全存触发器——256×8bit×512 深度，触发器面积是 SRAM 的 8 倍，撑爆 floorplan；全并行时钟树 18 级深度，skew 跨 corner 不可控；DDR 带宽 12.8GB/s 被随机地址访问打穿，256 MAC 利用率不到 40%。

回路第一次给出完整 e = r - y：面积超标、功耗超标、帧率不足。数据摆在 Peter 面前。

Peter 三个决策。一，256 MAC 改分时复用：64 组乘加器分 4 时隙共享，利用率从 40% 拉到 85%。二，权重进 SRAM：4 个 compiled macro 替代触发器阵列，面积功耗断崖下降。三，面积 8mm² 硬约束——不惜牺牲 TOPS，从 2 TOPS 砍到 1.2 TOPS。三个决策无一是算法自动生成。全是价值选择：放弃 TOPS 竞赛，保面积和帧率。

Sprint 2：面积 7.8mm²（-15%），功耗 460mW（-21%），帧率 32fps（+28%），全过。改善不是靠调参数——降频率、加流水级换不来 15% 以上提升。Peter 改的是架构：全并行→分时复用，触发器→SRAM，TOPS 优先→帧率优先。改的是"什么是好"的定义。

「人的价值判断（选什么、放弃什么、保哪个），AI 辅助分析。」

r 的源头：Peter 从六层洋葱最外层剥到最内层，每层一次价值裁决，每层留下"不做的理由"。剥到每日计划，"今天拿什么信号"不是执行问题——是价值判断做完后的水到渠成。AI 在给定 r 后穷举方案空间、仿真帕累托前沿、量化每个 trade-off 的代价。但选哪个——选的是价值。价值归人，搜索归模型。

第三幕：三刀切入——从 VP 到 RTL 的九步收敛

Peter 定了 r。Andy 的职责是产出可被测量的 y——y 无法被 Vicky 独立测，就是黑箱。

执行的第一步不是写代码，是切边界。多数人拿到架构文档就画 module 框图，各自分配 RTL owner——这是陷阱。模块是物理边界，不是逻辑边界：切开 BLC 和 DPC，BLC 输出就是 DPC 输入，墙两边数据还在流，改一处牵两端。Andy 反直觉：不看模块，看 DITL——数据包从入片到出片经过的处理站序列。LDPC 是五段数据流：收 LLR → 初始化 → 逐层迭代 → 收敛校验 → 硬判决输出。每段独立可验证，故障在单条流内隔离。每条 DITL = 跨模块数据流契约——数据流主权的核心：按用户场景切边界，不按模块切。

边界必须机器可执行。ICD 契约三分法拆成三个正交维度：Pipe 定义通道带宽与拍率；Valve 定义边界规则——何时有效、反压、丢弃；Translator 定义语义隐藏——LLR 浮点还是定点、位宽、帧边界标记。三星数据（Samsung Foundry, IP-XACT automation case study, 2024）：IP-XACT 自动互联检查把验证周期从两周压到一天。

进入抽象阶梯——控制反馈代价 H 的核心。五层等价抽象：Golden Model（秒级）→ TLM VP（秒-分钟级）→ VIP/Stub RTL（分钟级）→ Verilator/FPGA（小时级）。核心原则：逻辑错在 VP 改，成本是底层 RTL 发现的 1/100。DVCon 2025 TLM2 数据（DVCon US 2025, "TLM2 Acceleration for RTL Verification"）：TLM 分钟级 vs RTL 小时级，反馈差两个数量级。

三刀法驱动执行。第一刀特性隔离——切出独立功能域，每个域有自己的契约和 Golden Model，改一段只坏一段，G 从全片缩到单段。第二刀 MVP 短闭环——端到端最小可用链路，Python Golden Model 逐拍比对，三分钟出判决，H 从三天压到三分钟。第三刀纵向切片——接口到 RTL 全栈贯通，IP-XACT 机器对齐消除集成摩擦，I 趋零。

「模块是物理边界，DITL 是逻辑边界——切前者等于切水管，切后者等于切回路。」

R2-Edge 的 MAC 阵列 DITL 是三段：权重加载 → 乘加计算 → 累加输出。切出权重加载段独立验证后，故障 G 从全阵列的 9.2mm² 压缩到单段的 0.8mm² 内定位——这正是第一刀特性隔离的力量。

三刀驱动九步收敛：路径选型 → 边界划分 → 接口三分 → ICD v0.5 → VP 碰撞 → HLS 估算 → Emulator → 四维基准 → Frozen ICD v1.0 + RTL。注意序列结构：ICD v0.5 在 VP 之前——接口契约先行。HLS 在 RTL 前给面积功耗早期信号。到第九步，ICD 已机器验证，RTL 对着 Frozen ICD 写——不是对着猜测写。

Andy 不是在写代码，是在构建可验证的系统结构。三刀每刀产出，Vicky 都能独立测量——这是 Maker-Checker 分离的工程起点。

第四幕：机械 Oracle——AI 不能做自己的判官

Andy 产出了 y。Vicky 的职责是测 e = r - y。这道减法决定整个回路是收敛还是自欺。

为什么不能用 AI 做验证？模型对自己产出天生宽容，合并 Maker 和 Checker = 输出越来越自信但越来越偏离真实需求。DVCon China 2023 硬数据（DVCon China 2023, LLM-assisted Hardware Design session）：LLM 生成 Python 参考模型正确率约 95%，生成 Verilog RTL 正确率不足 60%。同一模型验证自己写的 RTL——它不会发现它不知道的错误。所以 Vicky 的根本原则：Oracle 本身不是 AI，是数字判定器——Spike diff、形式等价、DRC/Timing。全部数学判定，零模型参与。

机械 Oracle 五层递进。L1 Vplan 评审通过——验证计划覆盖所有 DITL 场景与边界条件。L2 C 比对通过——RTL 与 C 参考模型逐拍比对，pass/fail 是数字，不是感觉。L3 Formal + Sim 全绿——形式验证证明关键属性永不违反，仿真覆盖 Normal/Legal Exception/Illegal 三类场景。L4 逃逸率下降——瑞士奶酪六方法叠加堵洞：定向 + 约束随机 + 形式化 + 协议合规 + 场景测试 + 性能验证。单方法有洞，六层叠加逃逸概率趋零。L5 效能指标——覆盖率、断言密度、逃逸率趋势，数据驱动签核，#30 号人类最终签字。

这套 Oracle 由 32 Agent Unit × 5 层架构自主调度。analysis（#1-9）分析 Spec 与 DITL → design（#10-13, #17-21, #A-C）生成测试计划与用例 → review（#14-16）交叉审查覆盖率缺口 → execute（#22-29）并行跑仿真与形式验证 → review（#30 human）人类签核 → learn（#31-32）从逃逸 bug 反哺测试模板。调度引擎是 DUT Feature Matrix：读 DUT tags → plan.yaml → 扫描 pending 且 tag 匹配的 units → 按 Feature 组织任务流。Agent 在 Harness 内自主编排——Vicky 不需要排 meeting。

验证手段的形态必须匹配 bug 的形态。三层金字塔各照一类缺陷：riscv-tests pass/fail 抓指令错误；ACT4 签名比对抓数据流偏差，逐寄存器逐拍比对；Spike Lockstep 逐指令 + 寄存器 + 内存三轮比对抓架构状态一致性。例如：指令 A 写 x5，指令 B 在 A 提交前也写 x5——顺序模型通过，乱序模型下 B 的旧值覆盖 A 的新值。Spike 在 commit 边界逐拍比对寄存器堆快照，抓到这条 WAW shadow——pass/fail 测试永远看不到它。真实总线暴露理想内存盲区：store 飞行双写 bug 在 VIP 抽象总线永不出现，真实 AXI 互连上两拍即炸。

回扣 R2-Edge。Sprint 1 全挂——面积超标 18%、功耗超标 22%、帧率不足目标 60%。没有 Vicky 的机械 Oracle 测出这三个 e = r - y，Peter 没有数据拍板分时复用 + SRAM。测量不是意见——测量是决策的前提。

Vicky 是控制回路中的传感器——不做价值判断，只报告误差信号 e = r - y。判断是 Peter 的事，执行是 Andy 的事，调度是 Fred 的事。但如果没有 Vicky 的测量，整个回路是开的。控制器不兼任传感器——第一幕的伏笔，在本幕找到全部技术重量。

Loop 有三层含义：第一层，程序意义上的 while 循环；第二层，编排意义上的 Dynamic Workflow——四种原语组成的收敛脚本；第三层，控制论意义上的负反馈回路——r→y→e→调参→y'。三层递进：第一层是第二层的实现手段，第二层是实现第三层的工程载体。

Vicky 给出了 e=r-y，Fred 收到误差信号。控制器要做的不是重写 prompt，是重排 workflow。这是从"会写 prompt"到"会写 loop"的质变——prompt 是一次性指令，loop 是收敛机制。

一、编排沉降：从对话到 JS。编排最初长在对话里，但对话历史是线性衰减器——第 20 轮你已忘了第 3 轮的决策依据。Fred 的解法是把编排沉降为 JS 脚本——Dynamic Workflow。四种原语：agent（单次调用）、parallel（并行扇出）、pipeline（串行管道）、while（条件循环）。循环变量、分支判断、中间结果全部固化在脚本变量里，主上下文只收最终答案——编排逻辑不漂移。

「我不再为 Claude 写 prompt 了，我有一堆 loop 在跑，我的工作是写 loop」（Boris，R2 项目技术负责人，内部访谈）。这句话的技术含义：你的身份从操作者变成了编排者——你写的不是指令，是指令的指令。

二、六大构建块。Automations——cron 或 goal 心跳调度，做工作的模型和判断工作完成的模型必须不同（Andy 执行 vs Vicky 判定）。Worktrees——git 隔离并行，不是最佳实践而是机械必须：没有物理隔离，loop A 的中间产物污染 loop B 的输入，错误交叉传染不可追溯。Skills——SKILL.md 编码项目知识。数据锚点：A 团队 15 个 Skill 修复准确率 92%，B 团队 0 个 Skill 修复准确率 61%（内部统计，N=2 团队 × 6 个月，修复准确率 = 一次通过 regression 的比例）——差距不在模型，在 Skills。Connectors——MCP 接入外部工具，loop 从文件系统孤岛变成工作流节点，synth、PnR、DRC 结果实时回流。Sub-agents——Maker-Checker 分离：不同 agent 用不同指令、不同模型、不同 worktree，审查输出结构化判定 VERDICT / ISSUES / EVIDENCE。Memory——LOOP-STATE.md 三段式 Open / Done / Blocked，没有 Memory 的 loop 每次重启把昨天修过的 bug 再修一遍。

三、四件套乘积关系。Harness × Oracle × Memory × Loop 是乘积关系，不是加法——这不是修辞。四项的独立性论证：你可以有 Harness 而没有 Oracle（跑得快但不知道对不对），可以有 Oracle 而没有 Memory（测得准但不记得上次怎么修的）。四项可以独立退化，任何一项趋零，系统输出随之趋零。这不是极端场景的思想实验——每一个"有 X 没 Y"的组合，在现实团队中都找得到对应。

四、DoD 门 + 准入准出。声明式配置门禁：准入不满足拒绝执行，准出不满足自动回滚。Vicky 全绿放行，未全绿回炉至指定刀（三刀法中的特性/MVP/纵向切面）、指定 Unit（32 Agent Unit 中对应检查器）。

五、三个硬性停止条件。MAX_ITER=20——硬编码退出。无进展检测——相同错误连续 N 次则终止并告警。Token 或金额预算上限——资源耗尽前主动退出。「停止条件的『硬』不是谨慎，是速度的数学必然」——goal 的模糊性会复利增长，exit condition 写得模糊，方向偏移在几轮内复合放大。硬停止不是保守，是工程上唯一能保证收敛速度的手段。

六、第三刀发布火车。R1 必要→R2 增强→R3 优化。六产线同源扇出——TDD、Cov、QEMU、DOM、RegGen、TopGen——六条产线共享同一份 SSoT（Single Source of Truth），结构上不可能不一致。

七、在线编排 + 离线蒸馏。每跑一轮，Fred 扮演两个角色。在线——编排当轮交付管道，调参直至全绿放行。离线——蒸馏编排经验进内核（编排模式、收敛条件、门禁规则），下次类似任务编排更准、收敛更快。

R2-Edge 案例回扣：Sprint1→Sprint2 只用了 1 个迭代。不是因为模型更强——模型没变。是因为 Fred 的 DoD 门在每次 loop 中强制收敛——编排管道把"调参→重跑→再测"压缩进一个 while-until-clean 循环。Vicky 全绿的那一刻，不是"运气好 prompt 写对了"，是 loop 本身保证了只有全绿才能停。

外环：回图检验——行为变了吗，目标近了吗

内环收敛之后，还有外环检验。每六周或每个里程碑，回到地图前只问两个问题：行为真的变了吗？目标真的近了吗？读数是外部反馈——真实使用数据、客户行为、对照指标。裁决表四选一：变了，继续沿分支走；没变，换一条分支；方向错了，重画地图；目标达成，就归档。

注意归口：这个裁决是 r 级决策，归 Peter；Fred 主持节奏并执行裁决。回图检验和 Dynamic Workflow 是两个平面的事——一个管"目标对不对"，一个管"收敛快不快"。没有这一环，回路可能全绿地收敛到一个已经过时的目标。

以上五幕加外环检验，构成了完整的方法论操作面：Peter 定 r，Andy 切 y，Vicky 测 e，Fred 编排收敛。四件套乘积关系证明了每一项独立退化都足以让系统输出趋零。但还有更深一层的问题——当四件套完整运转、跑过一个项目、再跑第二个，系统本身会发生什么？

答案是双产出机器。产出 1 = 交付结果：敢流片的 RTL，数据可查。产出 2 = 蒸馏进内核：编排经验（哪些 workflow 拓扑在哪种场景下收敛最快）、Oracle 规则（Spike diff 阈值、形式等价判定条件、DRC 检查矩阵）、收敛条件（多少轮无改进判定方向错误）、门禁规则（哪些 e 信号映射到哪些回炉路径）。

互联网上几十万 skill。新手每天用 Claude Code 也在沉淀 skill——那不是护城河，门槛太低。真护城河 = 蒸馏的深度 × 质量。新手小白的 prompt 噪声和你十年验证经验的判断，不能以同等权重沉进组织内核。Fred 是这道蒸馏质量闸的守门人——只有通过 DoD 门、全绿放行的 loop 才被蒸馏。

四件套乘积关系在此落定终极推演：第五幕的独立性论证证明缺一不可——但乘积关系的更深层是动态放大的杠杆效应。缺 Oracle，幻觉以全绿假象穿过整个管道——不是"少了一个检查项"，是每一轮迭代都以幻觉为基准继续调参，偏差指数放大。缺 Memory，下一轮 Sprint 重新发现上一轮已修过的错误——不是"多花一点时间"，是收敛曲线在同一个坑上反复振荡永不下降。缺 Loop，编排经验蒸发了——人下班了，prompt 在聊天记录里腐烂。

为什么模型升级不是答案？「分钟级到周级——中间差的不是模型升级，是 Harness 厚度。」一行 RTL 几分钟写完，端到端流片要数周。差距不在模型能否写出更好的 RTL——模型写的 RTL 已经够好了。差距在：FW 验证、SDC 约束、综合、PnR、时序收敛——这每一环都需要 loop。编排越厚（更多验证层、更多自动恢复路径、更精确的收敛条件），自主 loop 时间越长。真差距不是谁的模型能写更好的 RTL，而是谁的编排内核越用越厚、收敛越来越快。

护城河公式：对手抄得走一个 skill，抄不走一个蒸馏了你专家编排判断、越用越厚的内核。Skill 是静态知识——一个 SKILL.md 文件。内核是动态经验——知道"什么情况用 pipeline 不用 parallel"、"什么 e 信号强度触发回炉到特性刀而不是 MVP 刀"、"什么收敛路径在什么芯片规模下历史上有效"。这是在线编排→离线蒸馏→沉降为 JS 脚本的循环产物。跑过 5 个项目，内核就比跑过 1 个项目的团队厚 5 倍——不是 5 倍的知识，是 5 倍的"知道什么不该做"。

「真护城河 = 蒸馏的深度 × 质量。」蒸馏的深度——你从多少次失败回路中提取了多少可复用规则。蒸馏的质量——这些规则经过多少层 DoD 门控和全绿放行才进入内核。两者相乘，决定了你的系统能从"分钟级自闭环"延伸到"周级自闭环"的长度。

边界与局限。 四角色回路是为中大型芯片设计（>10 万门、多模块协作）优化的方法。当修改范围限于单个模块的微调时，六层洋葱图的成本超过收益——可从 Sprint 计划层直接切入。四件套中的角色分离不依赖人员分离——依赖流程分离：确保同一人在不同时段严格切换角色模式。本文的回路从「已有 r」开始；r 的产生（影响地图）与 r 的翻译（目标 → 规格）依赖人的价值判断，不可自动化——回路的收敛可以被工程化，目标的选择永远是人签核。本文方法论基于有限案例推理，以下结论需在更多项目中检验。

回到起点的问题：AI 能写 RTL，写完然后呢？

一个人加 AI，只覆盖了 Andy 这一个节点。加上 Peter——有了方向。加上 Vicky——有了测量。加上 Fred——有了闭环。回路闭合，才敢流片。

从「能写RTL」到「敢流片」，差的不在模型，在这三个角色构成的验证管道和编排回路。

「没有验证管道和Harness，AI只是超级个体玩具。」

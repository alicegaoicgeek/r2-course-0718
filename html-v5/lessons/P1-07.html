r2-knowledge：决策飞轮，构建人和 Agent 都能消费的轻量知识库

蒸馏自命题+素材集（WY/小花/阿野不困/540朋友圈原文作者，2026年5-7月） 阅读时间约 35 分钟 | 共 10 个核心方法论

冷开场

你让AI写东西，它写得流畅、有理有据——但你一查来源，发现全是编的。

这不是AI的问题。

是你没让它先读库。

存进去，不等于用得上

知识库越建越大，Agent越用越慢。

反常，但根因极深。

真相只有一句：存进去，不等于用得上。

拆开是三个错位。

第一，组织错位。 传统知识库按"人翻页"设计——一篇文档就是一个页面，目录树就是导航。人翻目录能找到，但Agent不翻页，它靠检索跳跃。它需要的是自带上下文的信息单元，不是3000字的长文档。 第二，检索错位。 一篇3000字的方法论文档被向量库切成50个碎片，AI只捞到3片，前因后果全丢。更致命的是，三年前的过时结论和昨天的复盘，在向量空间里地位平等——垃圾和黄金一起涌进上下文，把模型的注意力活活稀释。这就是越存越傻的全部秘密。 第三，没有代谢。 只进不出的知识库，像长期被剥夺睡眠的大脑，一直在记从没整理。Claude Code给自己的记忆索引定了死规矩：不超过200行、25KB。它怕的不是记得少，是记得多。

给智能体用的知识库，三条铁律。

一、四层分离，按消费顺序排。问题库→观点库→案例库→复盘库。Agent沿链检索，本身就是计算量压缩——不是搜全库再筛，是逐层命中。

二、每条笔记自包含。tags做第一层过滤，三五句核心内容做第二层判断。Agent不用读完全文就能判相关性。做不到，入库即噪音。

三、写入有人工闸门。AI发散——提取二十条候选。人收敛——只挑三条入库。技能可被AI加速，判断不能。

最后：飞轮效应。Agent消费→人复盘→回写四库，命中率螺旋上升。首篇不到三成，十篇逼近五成，五十篇超七成。

库不是被维护的，是被使用的。每一次消费都在训练它。这三条铁律的底层逻辑，和 newtype-lite "先来源后断言、先结构后文字"的三层哲学其实同构——都是在给信息消费端设计结构，而不是给存储端堆功能。

四层分离：按消费顺序组织知识

大多数团队的知识库其实只有一层——"案例库"。项目文档、会议纪要、技术方案，全堆在一起。

四库飞轮的核心价值不在存储，在消费时的排序。

库目录回答的问题内容
问题库00-问题库/用户要什么？真实用户提问、评论区问题、需求信号
案例库01-案例库/凭什么这么说？项目记录、踩坑日志、工具用法、实测数据
观点库02-观点库/我的判断是什么？架构偏好、技术判断、踩坑结论、风格反模式
复盘库03-复盘库/上次哪里不对？发布复盘、错误分析、改进方向、延展选题

Agent在检索时不是先搜全库——是先搜问题库判断方向，再搜观点库锚定立场，再搜案例库找证据，最后搜复盘库避开旧坑。顺序本身就是计算量的压缩。

举个例子。你要写一篇关于"为什么CI流水线老挂"的分析文章。

传统做法：去wiki搜"CI""流水线""失败"，翻十几篇文档自己筛。

四库做法：Agent先查问题库——发现过去三个月学员问了23次CI相关的问题，集中在"编译时间太长"和"License排队"。再查观点库——团队之前有个明确的技术判断：编译慢的根因不是硬件不够，是依赖没隔离。再查案例库——项目A的CI优化记录里有一条"Docker层缓存命中率从12%提到78%"的实测数据。最后查复盘库——上一次写CI优化文章时，哪些论点读者反馈你没说清楚。

四层走完，Agent手上已经有了论点、证据和避坑指南。这才是"可消费"的信息。

你的团队经验，Agent为什么读不懂？不是Agent不行，是你的知识没按消费端组织。 — 806-四库飞轮

先读库再生成：硬约束

Agent接到任务后、开始生成前，必须执行一个强制步骤：

  • 读 index.md → 确定相关库和标签
  • 读 问题库/ → 确认用户真正的需求信号
  • 读 观点库/ → 确认团队的立场和判断
  • 读 案例库/ → 找到支撑论点的证据
  • 读 复盘库/ → 避开已经踩过的坑
  • 开始生成，每个论断锚定来源条目

这个流程不是"建议"，是硬约束。没有资料支撑的内容，不得写入。这一程解决的是生成质量的底线——挡住AI的"看似正确但毫无根据"。

核心约束叫无据不写：Agent在生成内容时，每个论断必须有来源锚点。来源锚点可以是问题库中的具体提问、案例库中的实测数据、观点库中的技术判断、复盘库中的教训记录。做不到的，宁可不写。

这个铁律直接堵住了"AI生成看似正确但毫无根据"的漏洞。

人工闸门：AI发散，人收敛

生成内容入库，是人类最容易犯的错误之一——让AI自动把内容写入知识库。

AI的问题不是"提取错了"，而是"提取太多了"。它看到一篇3000字的项目文档，会平等地提取每一条信息——但真正值得入库的可能只有其中3条。

所以写入端必须有人工闸门：

AI做发散：从原始素材中提取10-20条候选入库项。 人做收敛：从中挑选3条真正值得入库的。

这个3/20的筛选比，把人的时间压缩到了每天几分钟，但保证了入库质量。更重要的是它遵守了一条红线：质量的判断必须是人做的。 技能可以被AI加速，判断不能。

知识归档不是"存"，是"编码"

newtype-lite 里有一个容易被忽视的角色——Archivist（档案员）。它不在写作主链路里，而是在产出结束后介入：把一次任务的输出转化为可复用的知识资产。

Archivist 的方法出奇地简单，只要求三件事：summary（把过程压缩成因果链）、tags（建立检索接口）、follow-ups（留下行动锚点）。写完一篇文章？Archivist 会问你：核心结论是什么？打什么标签方便下次找到？有没有需要后续跟进的线索？

这和四库飞轮的单条笔记格式几乎同构——tags 做第一层过滤，核心内容做第二层判断，关联做横向跳转。两个项目独立演化，却指向了同一套结构。这不是巧合，是共识：知识保存的质量不取决于"存了什么格式"，而取决于"编码时有没有做因果压缩"。

没有编码标准的知识库，本质上是一个垃圾堆——东西都在，但找不到、用不了、信不过。有编码标准的知识库，每一次归档都在给未来的检索铺路。Archivist 的 summary/tags/follow-ups 就是编码标准，四库飞轮的"自包含笔记"也是编码标准。选一个，严格执行，比选哪个更重要。

一条笔记的价值，80% 取决于入库时的编码质量，20% 取决于内容本身。

记忆的本质是因果压缩

数据是死的，记忆是活的。

真正记忆并不是简单存储，而是能够被召回、被理解，并随着运行行为不断演化。这种差异看似抽象，却决定了下一代系统的边界。

记忆的本质是因果压缩，不是数据存储。数据是被动积累的离散事件。记忆是主动召回、持续演化的因果结构。决定Agent能力的不是存储量，而是压缩效率。

Claude Code的MEMORY.md限制200行、25KB。它怕的不是记得少，是记得多。

Hermes的MEMORY.md仅2200字符。当使用率超过80%时，触发自动整合——Agent主动合并相关条目、删除过时信息、压缩表达。这种"在压力下策展"的机制确保存活下来的记忆条目必然是高价值、高密度的。

两条线正交，不是谁取代谁：

  • Agent Memory：智能体自身经验积累的信度分配问题
  • User Memory：谁拥有操作行为的因果解释权

终端厂商拥有另一种天然优势——User Memory。真正的价值来自与用户最接近的那一层。当模型能力逐渐趋同之后，真正拉开差距的，或许不再是模型本身，而是谁能够持续积累对用户的理解，并把这种理解转化为长期服务能力。

向量不匹配：为什么AI记了但找不到

如果MEMORY.md中记录了一条笔记"用户偏好dark mode"，后续当Claude尝试搜索"主题设置"或"UI偏好"时，grep无法建立语义关联，导致相关记忆无法被召回。

这就是词汇不匹配（vocabulary mismatch）——结构性问题。

但不用向量搜索也能解决。四库飞轮的方案：tags + 核心内容 + 关联。

每条笔记只需要三样东西就够Agent用了：

---

tags: [CI, 编译优化, Docker]

date: 2026-06-15

source: 项目A-CI优化复盘

---

​

​

核心内容

[2-5句话概括，Agent一眼能判断要不要深入]

​

详细记录

[原始内容、数据、引用、上下文]

​

关联

  • [[相关条目1]] → 说明关系
  • [[相关条目2]] → 说明关系

关键是：Agent能在不读完全文的情况下判断相关性。tags做第一层过滤，核心内容做第二层判断，关联做横向跳转。这三样足够支撑Agent的多轮检索，不需要向量化。

单一事实来源：markdown就够了

真正的源头只有一个：本地的markdown文件。

不是数据库。不是向量库。不是SaaS知识库平台。

markdown是纯文本，任何AI客户端都能直接读、直接写、直接检索。你的知识资产不依赖任何一个厂商的API、一个数据库的schema、或者一个平台的导出功能。今天用Claude Code，明天换了别的Agent，文件还在那，Agent照样读。

newtype-lite 的内容团队方法论——8 个角色、完整的路由机制——全部压缩在 9 个 Markdown 文件里。安装只需要 cp -R SKILL.md references ~/.codex/skills/newtype-lite/。换 Agent？文件跟着走，知识资产零损耗迁移。

每多一层中间件，就多一层维护负担、多一层格式转换、多一个可能的故障点。

文件系统本身就是集成层：你的笔记工具写入 → Agent读取 → Agent生成的结果写回 → 你的工作台读取展示。全程没有一行API对接代码。

什么时候markdown不够用？

留在个人版（纯markdown）的条件：文件数量在500以内、只有你一个人在用、没有权限隔离需求。

升级到团队版的信号：全文grep开始明显变慢、同事开始问"你的笔记能给我查一下吗"、发现有价值的经验因为没入库而丢失。

升级不是"功能越多越好"，是"撞到边界才扩"。如无必要，勿增实体——这条原则在团队版同样适用。

消除知识壁垒，不是建立壁垒

这里需要一个更深层的视角转换。传统管理学告诉我们：分工提高效率。酿酒的人只管酿酒，打铁的只管打铁，交换带来整体效率提升。这个逻辑在物理世界成立——因为物理产品的接口是简单的：你给我一个瓶子，我灌进酒，交易完成。

但软件开发不是构造过程，是知识消费过程。

需求分析师把客户的模糊意图转化为结构化需求——知识从一种形式变成另一种形式。架构师把需求转化为系统设计——又一次知识变形。开发把设计变成可执行代码——继续变形。测试人员反过来验证代码是否承载了原始意图——闭环消费。

每一步都依赖对上下游知识的理解。当你说"写代码的人不需要理解业务"，你实际上在建立一个知识壁垒——壁垒越高，信息传递的失真越严重。需求文档写了 50 页，开发看完还是不知道客户真正想要什么。这不是人的问题，是壁垒的问题。

传统管理通过壁垒提高效率（"你只管这块，别管别的"），但软件开发里，壁垒降低效率。真正高效的团队，是知识流通最顺畅的团队——每个人都理解上下文，每个决策都基于对全局的认知。

四库飞轮在这个视角下有了更深的意义：它不是"给 Agent 建一套专用知识库"，而是"建一套人和 Agent 都能消费的知识体系"。同一份 markdown，人读能理解意图，Agent 读能执行检索。知识不再被锁在人的脑子里、锁在特定平台的格式里——它在人和 Agent 之间自由流动。

消除知识壁垒，才是四库飞轮真正的底层逻辑。

使用AI的最佳实践都是反直觉的

大部分人本能觉得AI应该这么用——你反过来操作，基本就是最佳实践。

让AI审人，不让人审AI。 人审AI容易翻车，因为AI太勤快。反过来，让AI审人的Spec和签核材料，它反而像个不知疲倦、没有面子压力的质检员。 让AI推荐专家，不把人经验全传给AI。 不要把70万行代码塞进上下文。RAG是你喂上下文，Agent搜索是它自己顺着引用链去找——给目录，让它翻。 砍功能不是加功能。 新模型出来第一件事是砍功能，模型会把你的harness层当早餐吃掉。Todo List从核心功能降级成装饰品，不是团队退步了，是模型变强了。 边做边想不是先想清楚。 构建成本归零、输出不可预测，正确姿势是边做边想。 挡住数据不是保护，是阻碍AI学习。 所有对AI来讲接触不到的数据，对它来说就是不存在，不存在的东西就学不会。 工具少比工具多好。 工具是Agent的感官，感官太乱它反而看不清世界。Claude Code故意只保留大约20个工具，不是能力不够，是克制。 先看体感再跑Benchmark。 用户的直觉和感受比跑分更能引导你该怎么加减harness。
我觉得这里面有一个根本性的矛盾——在今年4-6月开始，用AI的主流方式已经从L3跃迁到了L5。L5之后应该能至少从理念和框架层面稳定下来了。但目前这个变动期就会带来矛盾，很多人还是L3之前的习惯，包括企业搭建AI的原则还是围绕人的需求设计。这导致方向就完全走反了。

蒸馏能力：竞争不在模型大小

AI获得智能的过程，本质上就是在一层一层地蒸馏人类。

第一阶段，预训练。 AI蒸馏的是人类的显性知识。模型从海量文本里学人类怎么说话、怎么组织知识、怎么从前提推到结论。这一层蒸馏出来的，是语言能力和常识。 第二阶段，后训练。 AI蒸馏的是人类的偏好和价值排序。光会说话不够，模型还得知道什么样的回答是"好"的。这一层蒸馏的不是知识，是品味。 第三阶段，Agent。 AI蒸馏的是人类的行动路径和任务方法论。到了Agent这一步，AI开始真正干活了。它要学的不再是"怎么说"，而是"怎么做"。这一层蒸馏的，是工作方式本身。是把人类最稀缺的专业判断力压缩成数据。
Skill不只是新技术，它是一个非常明确的分水岭。

竞争不在模型大小，在蒸馏能力——蒸馏同事、蒸馏领域专家、蒸馏你自己的工程过程。AI的原生不是模型原生的，是组织原生的。谁能把一个人的经验压缩成token，谁就能把人类能力高效转移到机器。

技术红利期对很多个人或组织仅仅来自比对手快了一个月。AI演化太快，智能被复制的太快。当大家进度追平的时候，组织和个人都没有ROI回报可言。

提升10倍比提升20%更容易。20%是在旧发动机里硬榨margin，10倍是技术革命，是直接换赛道，花的力气反而是小的。

复盘回写：飞轮的那一圈

四库解决了"怎么存"。但如果只存不用，它就是另一个wiki——结构再清晰也只是仓库。

飞轮解决的是"怎么活"——从调用到复盘到回写，每一圈都在降低下一圈的使用成本。

内容发布满7天后，拉数据做复盘。复盘卡包含四项，每项回写到不同的库：

  • 哪个标题/钩子有效 → 回写观点库，更新"什么是好标题"的判断
  • 评论区问了什么 → 回写问题库，作为下一次选题的信号
  • 哪句话不像你 → 回写观点库的"风格反模式"分区，变成下一次生成的negative prompt
  • 下次还能延展什么 → 回写复盘库，标记延展方向和来源

为什么是7天？因为7天是数据稳定的最小周期——太短数据波动大，太长反馈滞后。

第一篇文章全靠人肉翻笔记找素材。Agent检索四库，命中率可能不到30%。

第十篇文章，Agent打开index.md，已经有9轮的复盘数据告诉它什么方向有效、什么坑要避开、用户在追什么问题。命中率开始逼近50%。

第五十篇——你的观点库积累了足够多的技术判断，复盘库记录了足够多的反模式，问题库沉淀了足够多的需求信号。Agent检索的命中率可能超过70%。

这就是飞轮：每一次消费都在训练知识库。

怎么知道知识库在变好

知识库不是建完就算完。你需要知道自己是不是在原地打转。

月度扫一眼index.md的更新记录和复盘库的文件数量，就大致知道健康状况：

指标1：召回命中率 — Agent接到任务后，检索四库命中了多少条相关的历史条目。如果命中率持续低于30%，说明你的知识库结构需要调整。 指标2：复盘回流率 — 已发布内容中，有多少触发了复盘卡生成并回写到了四库。这个数字永远不应该是0——如果复盘回流率为0，说明你在消费库存从不补货。 指标3：Bad Case回流率 — 每次"AI生成了但人觉得不行"的案例，有没有被记录到复盘库并提取为反模式？很多团队的复盘库长年只有几个条目——不是没有bad case，是bad case来了就来了，没人把它回写成系统的记忆。

如果只带走三句话

  • 存进去不等于用得上。 知识库越建越慢的根因不是技术选型，而是设计假设错了——按"人翻页"设计的知识库，Agent用不了。
  • 记忆的本质是因果压缩，不是数据存储。 容量约束=自然选择压力。Claude Code限制200行、Hermes限制2200字符，都在做同一件事——迫使Agent主动遗忘低价值信息以腾出空间给高价值信息。
  • 库不是被维护的，是被使用的。 飞轮效应：首篇命中率不到30%，十篇逼近50%，五十篇超70%。每一次消费都在训练它——newtype-lite 把这种训练机制写进了 Archivist 方法包的三个动作里。

Monday Morning

明天上班，做这三件事：

  • 建四个文件夹 — 在项目根目录下创建 00-问题库/、01-案例库/、02-观点库/、03-复盘库/。创建一个 index.md，写一段话：这个知识库是干什么的。
  • 投第一条 — 打开你最近一次跟同事或用户的聊天记录。找一个被问到的真实问题，写进问题库。格式就按三样东西：tags、核心内容（2-5句话）、关联。不需要完美，先让它存在。
  • 让Agent用一次 — 在Agent的prompt或系统指令中加入一行：「在开始回答之前，先读取 index.md 和 00-问题库/ 中的相关条目。」不需要复杂的检索逻辑——先让"读库"变成一个习惯，再优化检索。

五步做完，你就有了一套最小可运行的四库飞轮。后续的优化——更好的检索、团队版扩展——都是在这套骨架上长出来的。

陷阱清单

向量RAG万能论 — 以为把所有文档灌进向量数据库，让RAG去检索就能解决问题。结果：搜不准（片段缺上下文）、更新不了（文档改了向量要重算）、Agent消费不了（切碎的文档Agent看不懂）。 全量丢给AI整理 — 把所有文件全量摄取给AI，让AI帮你总结。结果：知识库变成第二个AI生产的第二个垃圾堆——看似工整但用不起来。 Skill便利贴化 — 写了一个SKILL.md，跑通一次任务，但模型一升级它就是反作用了。静态文本扔进动态概率系统——文本不变，但基座模型在变、工具在变、上下文在变。 人在环路拖慢AI — 人是慢系统，AI是快系统（快100-1000倍）。把快系统绑在慢系统上，AI进化依赖的环境建不起来。越依赖人，AI越无法成长。 概念无限膨胀 — 每看到一个有用点就想单独建一篇。结果：知识库变成另一套维护负担——和最初要解决的"找不到东西"问题一模一样。 Bad Case来了就来了 — 团队有很多"AI生成了但人觉得不行"的案例，但没人记录。同样的错误反复出现——组织没有从失败中学习的能力。

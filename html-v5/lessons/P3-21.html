第一层 · 开场：你缺的不是更多测试，而是更快的圈

PCIVAR 不是又一套验证方法，而是一台把反馈周期从「月」压到「小时」的发动机。

先看你熟悉的那个凌晨。你盯着 AI 帮你生成的 UVM 环境，语法检查过了，编译过了，仿真也跑起来了——但回归报告里，同一个 AXI 握手断言连续炸了四十七次。你翻开 AI 写的 Sequence，发现它把 AWVALID 和 WVALID 的时序依赖直接省掉了。它不是写错了，它压根不知道有这条规则。它没有「错」的概念，只有「像」的概念。

再看行业。验证占项目 70% 工作量，75% 的延期根源在此。2018 年 Wilson Research Group 研究：将近 40% 的时间花在调试上；可执行文档的口径是 47%。更扎心的是，44.2% 通过仿真测试台的代码，在形式验证中照样失败——你的验证环境本身就没被验证过。

一线更直白。一线验证主管嘉琪收集的吐槽：「设计怎么又改设计方案了，我这环境刚改完啊」「大哥您别改了，regression很慢的」。嘉琪的统计同样扎心：30% 甚至更多时间烧在 VIP 集成、环境配置这类重复「连接」劳动上，临时实现日积月累，堆成新人要数周才看懂的「石山代码」。

投入这么大，为什么还这么慢？43 号白皮书点了四个黑洞的名：闭环缺失——反馈周期以周甚至月计，缺陷右移；协同失效——验证被孤立，与设计缺乏互动；价值迷失——无法决定「验证何时足够」。颗粒度失衡——没有小颗粒分解，无法灵活排序并行。

四个黑洞里，闭环缺失是总开关。反馈越慢，变更越像灾难；变更越像灾难，人越不敢动；协同和价值跟着一起塌陷。传统答案是加班、加人、加回归——全是在旧圈里加码。本课的判断相反：你缺的不是更多测试，而是更快的圈。

于是有了 PCIVAR：Plan、Create、Integrate、Verify、Analyze、Refactor——六个字母，六个动作，首尾相接成微循环。一圈只验一个 Feature Point，几小时转完，转完系统就变好一点。

「响应式」先交代清楚：它只出现在仓库 README 标题里，六份源文档正文都没用过。本课不冒充原文，把它落实为可检验指标——对需求变更和缺陷的小时级响应，原话是「PCIVAR微循环把反馈周期从月/周缩短到小时」「PCIVAR把反馈周期压到"小时"」。「柔性」有原文撑腰：43 号白皮书写着「从刚性到柔性」「柔性自适应」，2024 白皮书列为四大转变之首。

产品一句话：r2-dvbuild 把 PCIVAR 六步微循环架在 FlexFlow 调度引擎上，粒度可小至单一 Feature Point，反馈周期从月压到小时。

实测与对照（第六层分标）：70 FP→137 断言→441 行 RTL 全过零波形调试；24 个 IP 16 万/月→1 万/1 天；70 个 IP 8-12 人 6-9 个月→2-3 人 12 周。

PCIVAR微循环就是飞轮的具体落地。六个字母，六个动作，一圈下来系统变好一点。

第二层 · Plan 微规划：把粒度切到一个 Feature Point

你的验证计划多久改一次？设计变更邮件一到，计划是跟着动，还是直接作废？

Plan 是微规划——不写大而全的验证计划，只针对 Feature Point 粒度，定义这一轮验什么、怎么验、怎么判过。大而全的 VPlan 去 829，P 只回答这一圈的事。

源文档的定义很短：快速确定当前迭代的验证目标和方法，粒度可小至单一特性。为什么要小？反馈速度是粒度的函数——圈越大转越慢，圈切到 Feature Point，反馈就是小时级。一线把这个原理说得更狠：交付特性、交付增量比交付全量集成快得多，靠反馈拉动迭代——大环套小环，飞轮越小转得越快。

一句话记住：粒度越细，反馈越快。

操作上三步。一，从变更切出本轮 Feature Point——一个中断清除条件、一个 burst 边界，就是一圈。二，先写通过判定：哪条断言、哪个覆盖点、什么数值算过。三，LLM 生成计划草稿，人审再进 Create——草稿 AI 写，目标人点头。

输入：Spec 变更单、VPlan · 输出：FP 级微计划（验什么、怎么验、怎么判过）· 门禁：FP 审批通过，才允许进 Create

第三层 · Create + Integrate：不等攒一批，每一轮都集成

AI 一分钟生成十条 Sequence，你的集成排期是下周——快出来的时间，是不是又排队排回去了？

Create 是快速创建。AI 根据 Plan 生成 Sequence 和断言，在脚本引擎搭好的骨架上注入业务逻辑——速度取决于 Skill 库的厚度，Skill 越多创建越快。另一条铁律是复用优先：用现有组件构建本轮必需的最小验证资产，不新造轮子。

Create 的分工原则，一句话：机器能算对的，绝不让人写；绝对的逻辑，绝不用AI猜。骨架、地址计算、调度排序归脚本，拓扑推导、场景约束归 AI，规则定义归人。

一线的打法完全一致：环境构建从「自由创作」改成「精密装配」——复杂协议封装成验证过的用例单元，配置只声明「需要什么」；通用 case 七成靠配置生成、两成微调、一成手写，精力从「造轮子」转向「想场景」。

Integrate 是持续集成：新代码立刻进自动化流水线——编译、Lint、和已有测试一起跑。不等攒一批，每一轮都集成。为什么？攒一批再集成就是闭环缺失复发：缺陷在队列里右移，修复成本随等待翻倍。

输入：微计划、Skill 库、骨架 · 输出：注入完成的 Sequence/断言 + 全量联跑结果 · 门禁：编译过、Lint 过、已有测试不回归

第四层 · Verify + Analyze：不问过了没有，问为什么

回归全绿，你就收工了？覆盖率没涨的那部分，你打算什么时候看？

Verify 是快速验证，司法层在工作——通过就是通过，不通过就打回去，且优先执行最高价值测试。哪些算最高价值，筛选去 666。

Analyze 是即时分析，源文档称它「最关键」：不是只看「过了没有」，是分析「为什么没过」「过了但覆盖率为什么没涨」——空洞在哪里？约束不够还是场景遗漏？分析可以 AI 做，结论要人确认。挂了怎么查，去 846。

LLM 在两级各有抓手：V 级按波形生成断言、优化仿真参数；A 级分析覆盖率、定位根因。判决权在门禁，结论权在人——机器出数据，人出判断。

暗线：A 的结论直接喂回下一轮 P——圈就是这样闭上的。

输入：集成后的候选 · 输出：通过/打回判决 + 根因分析（AI 起草、人确认）· 门禁：判决即执行；分析结论未经人确认，不进 Refactor

第五层 · Refactor：输出不是代码，是知识

Bug 修完了。下一圈，这个系统凭什么比上一圈聪明？

Refactor 是持续改进：每次微循环后立即小规模重构。对象不只是代码——按 Analyze 的结论修正 Skill 文件、补充约束、调整 VPlan。LLM 给重构建议，写不写、写哪条，人定。

源文档把这句话说死了：输出不是代码——是知识。修正写回系统，下一圈 PCIVAR 就站在更高的起点上。

看个实例（口播稿叙事）。SPI 的中断清除信号在读状态寄存器前拉高——人修正，规则写回 Skill 文件；下一轮 I2C 自动遵守，再下一轮 DMA 同样生效。一次修正，永久受益。

这就是飞轮的机械原理：人改一次，系统记一辈子。「PCIVAR微循环就是飞轮的具体落地。六个字母，六个动作，一圈下来系统变好一点」——这句话最沉的那一格，就是 Refactor。

命中率曲线佐证方向：口播口径 60%→85%→92%，测量方法未披露，看趋势，别当承诺。

输入：人确认后的分析结论 · 输出：写回 Skill/约束/VPlan 的知识条目 · 门禁：知识入库可检索，下一圈自动生效

第六层 · 证据与调度：用 48 小时跑完传统的两周

六个动作谁来排班？FlexFlow——PCIVAR 的调度引擎，五个功能：YAML 任务编排、依赖管理、并行执行、结果跟踪、错误处理。

一句话说透：YAML 把模糊的 IP 验证拆成显式依赖图。容器化类比——应用可以 crash，编排保证 service 不宕机，单个任务挂了，流水线不塌。

第七层 · 人在环路：守住三个审点，签核寸步不让

AI 跑得这么快，人往哪儿站？站多了是瓶颈，站少了是失控。

源文档定了三个人工审核点：FP 审批、覆盖率审查、排除分析审查——共同特征是它们都定义「什么是正确的」。机器负责跑得快，人负责定义对。

为什么 FP 审批是铁律？想象：FP 把只读寄存器的清除条件写反了，没人发现。AI 按错 FP 生成测试，脚本按错 FP 生成 RTL——测试和 RTL 完美互验，报告全绿。一致不等于正确。立法出错，行政和司法会忠实执行错误的法律，而且不报警。这就是三权分立里最危险的失效模式。

权力配比有硬数字：脚本引擎处理 98.4% 可确定化逻辑，AI 只做 1.6% 需语义推理的任务，人以 10% 的时间保留 100% 决策权。VPlan 是立法，立法权不能交给 AI；签核是司法最终裁决，裁决权不能交给 AI。

不信单点，信对冲：AI 有 20% 的不确定性，模型升级解决不了。做法是三层独立验证——Linter 三关、自动化仿真、覆盖率度量，三个独立裁判同时被骗过的概率是 20%×20%×20%，不到 1%。同系列 805 的「瑞士奶酪」同理。

放权速度看飞轮成熟度：命中率稳在 90% 以上的 IP 类，可设阈值自动通过；新 IP 类型、新协议——AI 没见过的场景——人回高介入，逐个亲审。但底线不动：最终签核，百分之百是人。

嘉琪看得更透：自动化的终点不是取代验证工程师，是把人从重复劳动里捞出来——每天琢磨的不再是「怎么连这根线」，而是「这个场景会不会漏掉」。工程师的日常已变成：需求分析 → 场景设计 → 配置定义 → 流程固化 → 自动化回归。

签核是责任，不是技术。
输入：三审点的待决项 · 输出：人签的 FP、人确认的覆盖率与排除分析、人签核的报告 · 门禁：签核 100% 是人，无例外条款

第八层 · 边界与演化：先标准化，再自动化，最后智能化

这套圈哪儿都能跑吗？不是。第一条：先标准化，再自动化，最后智能化——跳步等于建在沙子上。没有可执行文档，不开工。元数据没标准化的团队，先补 829，别碰 PCIVAR。

第二条：解决 80% 重复劳动就赢了，追 100% 自动化是陷阱——最后 20% 可能花 80% 工程量。第三条：新 IP、新协议，人回高介入。第四条：D1-D5 评分摆着——L3 人主导、L5 人集成，PCIVAR 不是全自动承诺。

上量前先过小试：90 天 MVT，三条成功标准——周期压缩 ≥3 倍、一次通过率 ≥80%、主动复用率 ≥60%；三条止损线——无效停、超预算无案例停、Bug 率上升停。

口径差异也交代清楚。「四层质量防线」「五级质量门禁」「五层质量保证」并存——版本并行的口径差异，不是矛盾，引用以各自文档为准。「响应式」只见于仓库标题；命中率 60→85→92 为口播叙事口径。

演化线五代可追溯。2024 白皮书：六组件名录登场，FlexFlow 并列出现。2604-6 笔记：最完整定义定型——六阶段释义、FlexFlow 五功能、「柔性自适应」收尾。

2604-Alice：课程卡片化，粒度精确到 Feature Point，六层架构图中固定为「验证规划层→反馈闭环」。2605-2：武器化——绑定 TDD 四层防线与 Job Mode，新增可复现性语义：同一 Job 任何人执行，过程保障相同。

口播 v1：飞轮落地——故事化六连问、SPI/I2C/DMA 实例、命中率曲线。

另有一线暗线：嘉琪语录的五条总路线——vplan 保输入、case 用公共语言、环境流程化暴露有限接口、文件参数配置、重构加单测——与后来的五层架构同构。框架不是拍出来的，是从一线长出来的。

输入：拟引入团队的现状盘点 · 输出：标准化差距清单、D1-D5 评分、90 天 MVT 计划 · 门禁：三条止损线写进立项书

收束：把圈跑起来

回放一遍。P 把粒度切到 Feature Point——粒度越细，反馈越快。C 复用优先、骨架内注入。I 不等攒一批，每一轮都集成。V 判决即执行。A 问为什么，结论人确认。R 把修正写回系统——输出不是代码，是知识。

六步之外三句话：FlexFlow 排班，人守三审点，签核寸步不让。一周能转几十圈，每一圈系统都变好一点——快不是目的，转起来才是。

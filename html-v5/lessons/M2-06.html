M2 认知 · 06

R2推荐的模型，工具及场景配置组合

Part1 · M1 跑通 · 01

这课只回答一件事：你这周该开哪套栈。

别指望一张「永久推荐表」。模型改名、涨价、出网策略一变，表就废。我能给你的是条件句，加上我自己押过的路。

外网能拼栈，我用 Grok 做执行脑、OMP 做机身，整体跟 Claude Code 全家桶打平，而且更省、往往更快。内网或者不能乱出网、只想少拼装时，我默认还是 CC。Hermes 管跨周记忆和编排，WorkBuddy 管办公和专家分发。没有永远第一。

数字里凡是社区流传的，不当实验室报表。我自己账本上的事，直接说。库内还有一批相关主粮——例如 course 里的 25-grok 重要对话、33-oh-my-pi、19-omp 集成 SuperGrok 实测、以及 Claude Code 的 loops 与动态工作流笔记——本文只取和「怎么配」直接相关的句子，不展开成书单。

一、先讲一笔我自己的账

Grok 的收费，我用下来觉得非常良心。当时买的 99 美金 heavy 会员（限时折扣，后面会涨到 300），差不多三天用掉一周额度的一半，做完三件事，全都带测试：

一个工业级 IP 的开发和主线功能验证，大约半天；一个对标商业 EDA 的代码，大约半天；我自己的内容生产线，大约一天。

测试是 token 大头，走弯路也是大头。工程纪律我知道，模型也知道，知道不等于做到。遵守纪律反人性，不遵守就浪费时间、多烧 token、质量还差。如果换成 A 社那套会员，这里面一个任务就能烧光一周额度，还未必做完。

还有一点比省钱更重要：快。快的隐形好处是不破坏心流。并行挂着等结果，注意力碎成渣——这事社区骂了很久，我也干过。能把反馈环收短，人还停在同一个问题里，判断力才不会耗在「等待管理」上。

所以我对外网的态度很简单：执行层要买得起失败，也要等得起结果。Grok 解决前者的大半，机身还得另说。

二、外网：Grok 不能单飞，得配 OMP

社区里其实早有分工，只是容易听成站队。Claude Code 强在深度 reasoning、规划、clean code、judgment、review，适合要小心判断的活，但慢、贵。Grok 强在速度、token 效率、成本和大批量改代码，适合快速迭代。常见说法是 Claude 做规划 review，Grok 做 heavy lifting。也有人直接把 Grok 当主力跑 agent，觉得 agent stack 终于可以规模化、不那么容易破产。

问题在于：很多人拿「裸 Grok」去跟「CC 全家桶」比，这不公平。裸 Grok 只有油，没有完整 coding harness，跟 CC 比会吃亏。我心里的公式是：

CC 全家桶 ≈ Grok + OMP

Grok 是性价比执行脑，OMP 是极速 harness 机身。组合后才有资格跟 CC 对打，而且更省、往往更快。Claude Code 有而 Grok Build 缺的几项核心能力，我觉得 OMP 都能补上——这是我外网敢把双芯当主力的原因。

OMP 本身是什么？终端里的 coding agent 机身。它要解决的不是「再出一个更聪明的聊天窗口」，而是改文件稳不稳、diff 看不看得清、能不能开子代理、模型能不能切换、压缩的时候人还要不要去洗澡。社区原话差不多就是 crazy fast、just works、用了回不去。日常高强度改仓库，我用 OMP 扛 execution；要极深的判断，我还是会回去找 CC。有人把 OMP 当 CC 的日常替代，把 CC 留给高判断力工作——这和我的分法一致。

合集里对 OMP 的定位写得很清楚：它和 Hermes 的侧重点不同——Hermes 更偏向持久个人 AI OS（记忆 + 自改进 skills + 长期自治），OMP 更偏向高性能 coding harness（速度 + 执行力 + IDE-like 体验）。OMP 特别强的地方是：速度极快、token 效率高、coding 执行力强（hash-anchored edits、inline diffs、LSP/DAP）；适合重度 coding、debug、快速迭代的 power user；subagents 和 role routing 用起来很顺手。相对弱的地方，对比 Hermes：持久记忆和自动 self-improving skills 没有 Hermes 那么突出；更像「高效终端 coding IDE agent」，而非 Hermes 那种「长期陪伴、自己会进化」的个人 AI OS。

实际使用建议也写进了合集：如果你的核心需求是快速原型、修 bug、大代码库操作、精确执行 → OMP 非常合适，甚至很多用户觉得「用了 OMP 后就回不去了」。如果你更看重长期记忆、重复工作自动化、多任务自治、越用越聪明 → Hermes 优势更大。很多 power user 是两者 stack 用：OMP 做高频 coding 执行，Hermes 做整体 orchestration 和记忆。

外网双芯也有边界。规格拍板、架构取舍、签核前苛刻 review，我仍拉 CC。便宜会诱人松纪律：额度宽了，更要写清楚验收，不然只是更快地走弯路。验收如果只写「跑通就行」，模型真能跑通一条最窄路径，脏路径全没碰，演示好看，回归翻车。我会多逼自己写一句：哪些情况必须失败。

选型不用背表，记住三句就行：

要省、要快、能拼栈，选 Grok + OMP。

要官方工具面和 loop 语言一体化、少拼装，选 CC。

不是谁碾压谁，是一体机对性价比双芯。

三、内网：我默认 CC 一体机

芯片团队很多日子不是「想用谁就用谁」。数据出不了域，代理只放行指定服务，公司只批一条模型链路，或者客户代码根本不能进第三方。这时候再讨论 Grok 有多划算，是空谈。

内网或强合规下，我的默认是 Claude Code 一体机：模型、工具、loop、goal、多 agent 都在同一套里。社区把它放在判断力最强、思考最深的 brain 位，也吐槽慢、贵、token 高、容易锁死在一家——两面我都认。总体印象是：它不是最快最便宜，但判断力和工程可靠性最强，竞品常用来补短板，CC 做核心大脑。给人的感觉是真正懂工程、能当高级搭档，而不是单纯图快。

合集里对 Claude Code 的支持情况，用同一套架构模式表对比后，结论更硬：最强的地方是 Plan-and-Execute + 高质量 reasoning（judgment 和 review 仍属顶尖）；成熟的 agent OS 架构（CLAUDE.md + skills + hooks + plugins），非常适合结构化、复杂、需要高质量输出的任务；适合需要深度思考和干净代码的场景（架构设计、重构、review）。相对弱的地方对比 Hermes / OMP：速度和 token 效率明显慢于 OMP 和 Grok 4.5，成本较高；持久记忆与自改进依赖人工维护 skills，不如 Hermes 自动、自进化；自治与灵活性上 provider lock-in，orchestration 不如 Hermes 灵活，执行力不如 OMP 快；普通人友好度需要一定配置，上手门槛比 Hermes 高一点。

实际使用建议（合集原句，我完全同意）：普通人 / 长期助理型 → Hermes 最推荐（记忆 + skills + 自治最强）；高频 coding / 执行力强 → OMP 最推荐（速度 + tool-use 最强）；需要高质量 reasoning / 复杂规划 → Claude Code 最推荐（判断力和 review 最强）。很多 power user 是三者 stack 用：Claude Code 做 planning + review（大脑）；OMP 做快速 execution + debug（手）；Hermes 做长期记忆 + 流程自动化 + multi-task（大脑 + 管家）。

内网真正值钱的，经常不是某一版模型的盲测分，而是少拼装。你不必同时养两套登录、两套出网审批、两套「挂了找谁」。CLAUDE.md、skills、hooks、subagents 这套，把个人偏好收成可版本管理的规则，团队培训教的是一种做事方式，不是三个账号的拼图。官方 loop、goal 最成体系；Grok+OMP 侧也可以近似 loop，靠的是工作流和人设停止条件，不是假装「只有 CC 能做循环」。库内还有 Claude Code 作者那句方向：不写 prompt，写 loop——内网把循环和验收钉死，比追新模型名更划算。

CC 不是神。外网我要冲量大带测时，不会道德绑架自己必须忠于它。内网若慢到心流断了，也别先骂模型：任务切小，验收写清，脚本和本地工具链能定的别丢给 LLM 死磕。合法有时比快重要。这句话不酷，但能让人正常下班。

再进阶两句，和外网不矛盾：要跨周记忆、后台编排，可以再挂 Hermes；要最深 judgment、签核前 review，外网双芯也仍可偶尔拉 CC。

四、Hermes：跨周记得住，重复别从头

Hermes 我放在记忆和编排层，不当又一个写 RTL 的窗口。

它强在持久记忆、把重复任务收成 skill、能 24/7 往后台推、能规划并委托 coding 给 CC 或 Grok，再验证、再写回记忆。社区常见用法是：Hermes 当 orchestrator / personal OS，Claude Code 当 sub-agent 只扛重 coding。也有人 Hermes + Grok 就够，省钱又不输。有个比方很贴：CC 像每次都要重新介绍的临时工，Hermes 像住久了越来越懂你、自己会进化的伙伴。用了之后，CC 往往只负责最难的那一小部分。

合集把 Hermes 放进「完整现代智能体架构」里看：它几乎把多种模式结合得很好——持久记忆（Memory）：Episodic（历史记录）+ Semantic（知识偏好）+ Procedural（skills）；自改进 skills：重复任务自动/手动变成可复用流程（Self-Improving）；Multi-Agent Orchestration：天然支持 recursive sub-agents 和并行执行；自治（Autonomy）：24/7 background + TUI/GUI + 工具集成，让它真正「自己会动手」。这也是为什么「普通人场景」脚本往往更适合 Hermes——正好发挥 memory + skills + 自治，而不是简单把它当成一个「会写代码的工具」。

架构模式对照（合集里整理的主流分类，我只取和选型相关的）：ReAct 是思考→行动→观察→循环，简单、解释性强，适合简单工具调用、调试、问答；Plan-and-Execute 是先整体规划再逐步执行，结构化，适合长任务、多步骤工程——Claude Code 默认就偏这一路，合集评它在这模式上最强；Multi-Agent Collaboration 是主代理协调多个专精子代理，分工明确、并行能力强，大型项目像团队式工作，Hermes 是典型代表；Hierarchical / Recursive Agents 是树状层级，子代理可继续拆分，可处理极复杂任务，Hermes 强项；Memory-Augmented 是短期加长期记忆，越用越聪明、不重复，Hermes 最强；Self-Improving / Reflective 是反思→总结教训→更新 skills/memory，持续进化，Hermes 核心特性；Tool-Use 则是几乎所有现代 agent 的标配，突破模型能力边界。

在同一张对比表里：ReAct 三者都强，Claude Code 在 judgment 质量上最优；Plan-and-Execute 上 Claude Code 最强，Hermes 次之，OMP 也支持但更偏 execution；Multi-Agent 上 Hermes 最灵活，Claude Code 成熟但 orchestration 不如 Hermes；Recursive 上 Hermes 在递归和自动拆分上更强；Memory 上 Hermes 更强（三种记忆类型 + 自动持久化），OMP 中等，CC 靠 CLAUDE.md + skills；Self-Improving 上 Hermes 明显更强（自动 skill creation），CC 需要人工维护，OMP 较弱；Tool-Use 上 OMP 在工具丰富度和 coding 集成上更强。

合集还写明了 OMP 对各模式的支持程度：ReAct 原生支持，思考-行动-观察循环非常流畅，两者都强，OMP 执行更快；Plan-and-Execute 支持 role-based routing（planning / task / commit / advisor 等角色），类似 Hermes，但 Hermes orchestration 更灵活；Multi-Agent 支持 subagents + parallel fan-out + Swarm Extension（可定义 DAG 流程），Hermes 的 recursive orchestration 更强；Hierarchical / Recursive 支持 recursive subagents，Hermes 在递归层级和隔离上更成熟；Memory-Augmented 支持 Hindsight 持久 memory（retain/recall/reflect），有 session-level 记忆，但 Hermes 明显更强（跨 session 持久 + 三种记忆类型）；Self-Improving 支持 hooks/pre-post + skills/plugins + reflective advisor role，Hermes 明显更强（自动 skill creation + 自进化闭环）；Tool-Use 极强（32+ built-in tools + LSP + DAP debugger + Python/Bun 执行环境），OMP 在 coding 工具集成上更胜一筹。

这些句子不是让你背星级表。是让你选栈时对得上号：你要 judgment，别只买速度；你要速度，别只买大脑；你要跨周，别只开一次性 sprint。

记忆可以粗分成三层：流程（skills）、事实偏好、历史情节。重复任务下次不该从零 prompt。社区流传过把八小时量级的流程压到短得多的故事，我只借机制，不当自己的成绩单：它卖的是时间厚度，不是单次智商表演。发链接让它总结打标签、几个月后一句话调出来，也是同一类能力——外存，不是更长的对话框。

比工具名更硬的是任务定义。你至少要说清三句：我要什么，给你什么，最后怎么验收。模糊需求在任何 harness 上都会放大成噪音。边界清楚的活反而友好：哪里坏了，想修成什么，能不能跑起来。很多人怕的不是写代码，是东西坏了不知道从哪修——这种边界清楚的循环，才适合丢给自治执行。

Hermes 解决不了签核。它资产化的主要是流程性经验，不是「这版能不能在硅上签字」的判断。个人记忆也不是团队权限库，要修剪，会腐。需要跨周、需要编排时再挂它，别开学就把五件套戴齐。

五、WorkBuddy：仓库外的事

工程师一周从来不只 RTL。纪要、资料、对外协作、培训材料照样吃时间。我曾经用最强 coding 会话去收拾课件和目录，属实杀鸡用牛刀：额度心疼，效果还一般。WorkBuddy 这类东西，我放在办公协作和专家分发，不放在签核主栈。

公开实践里有一句很刺：Agent Coding 需求还是太小了，真正的大需求是办公类 Work。蓝皮书重实战案例；知识管理是常见用法；也可以把网站或 GitHub 丢给它，让它记住速查表，在合适时机推荐技能。这是 Agent 吃得进知识的最小形态：短、可检索、渐进披露——比再堆一坨打不开的长 PDF 有用。

它解决触达，不解决覆盖率和硅上证据。别指望办公专家团替代验证负责人签字。主链路仍按内网 CC、外网 Grok+OMP 选；WorkBuddy 是旁路。

六、怎么重判，而不是站队

你不必背产品名，问自己四个问题就够：

这周网络允不允许拼栈？不允许，先别谈 Grok 有多香。

你卡的是想不清，还是跑不完？想不清，判断层权重上升；跑不完，执行层和机身权重上升。

你是单次冲刺，还是跨周重复？跨周，才值得上记忆和编排。

你现在做的是仓库里的 diff，还是仓库外的协作？仓库外，别浪费 coding 会员。

也可以换成架构语言问自己（合集那套模式名，只当梯子）：你现在缺的是 Plan-and-Execute 的判断质量，还是 ReAct 循环里的执行速度？缺的是 Memory-Augmented 的跨周，还是 Multi-Agent 的分工并行？缺 Tool-Use 的精确编辑，还是 Self-Improving 的重复固化？问完再对号入座，比死记「某某最强」管用。

对应到我的用法，以及合集里那条 power user 建议：

内网、合规、少拼装 → CC 一体机（规划 + review 大脑）。

外网、要省要快、能拼栈 → Grok + OMP；OMP 做快速 execution + debug（手）；难点和签核前再拉 CC。

跨周记忆、流程固化、多任务编排 → 加 Hermes（管家）。

办公、专家分发、速查 → WorkBuddy。

很多 power user 是三者 stack 用：Claude Code 做 planning + review（大脑）；OMP 做快速 execution + debug（手）；Hermes 做长期记忆 + 流程自动化 + multi-task（大脑 + 管家）。外网我把「大脑里的执行半边」换成 Grok，机身仍是 OMP——总账和 CC 一体机打平，钱和速度往往更好。

社区六十天就能翻一页。有人睡很久的 Grok 会醒，有人用了 OMP 回不去，有人继续嫌 CC 慢——这些都是水位线，不是圣经。品牌会换，条件句不要丢。

若你只记住一句话，记住这句：CC 与 Grok 打平的前提是 Grok 不单飞，而是 Grok+OMP；Grok 是性价比之王，OMP 是它的机身。

库内若还要往下挖：Grok 侧可看 25-grok 重要对话、24-Grok Session 二次加工；OMP 侧可看 33-oh-my-pi、19-omp 集成 SuperGrok OAuth 实测；Claude Code 侧可看 38-Loops 上手、21-on my claude code、以及动态工作流与 goal/loop 那一批——那是进阶主粮，不是本课必装清单。合集一眼结论也写过：近十四天 Grok 活跃主文是 25、24、19；OMP 是 33、19、956；Claude Code 新增和活跃都更多，包括 40-gitflow、38-Loops、21-on my claude code。最值得先看的五条里，前三条正好是 grok 重要对话、oh-my-pi、omp×SuperGrok 实测——和本课外网双芯同一条河。

七、明天做什么

写死本周主环境：内网还是外网。代码不能出域，算内网。

内网就 CC。外网就 Grok + OMP，并写一行「什么任务必须 CC review」。需要再加 Hermes 或 WorkBuddy。

任何工具开干前三问：我要什么，给你什么，怎么验收。验收尽量写成机器也能判的句子。

然后去做一个可工作的 UART，或你自己的最小可运行模块。这课过关标准不是收藏夹变长，是默认栈能打下一发真交付。

外网用双芯拱，内网用 CC 拱。工具服从场景。敢跑仍不等于敢签——那是后话；先把这周的栈选对，再谈制度。

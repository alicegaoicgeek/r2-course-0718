企业上 AI 最危险的错觉是每个员工更快，公司就会变成 AI Native。

过去两年，大量企业给每位工程师开通了 Claude Code、Cursor 或 Codex 账号，AI生成速度确实变快了。但如果你问一个简单问题——"这个模块为什么这样写？"——答案往往是"AI 写的，我没细看"。如果你追问"那次偶发测试失败是怎么定位的"，答案往往是"日志里可能有记录，但我记不清了"。AI有没有从经验中学会，如何有意识的引导AI学会？说不清。

张三用 Agent 调通了一个偶发失败的测试，花了 4 小时试了 5 条错路，最后发现是时钟域 crossing 的边界条件没处理好。李四第二天遇到了几乎一样的问题——从零开始。张三的经验留在他的私人聊天记录里，对李四没有任何帮助。

这就是私人 Agent 的天花板：键盘前那个人。

一次好的排查、一次好的修复、一次好的判断——这些经验有没有变成所有人、所有 Agent 都能继承的能力？还是它只活在那个当天加班的工程师脑子里？

如果一次排查只在一个人身上产生价值，组织的智力总量没有增加。每个人都快乐一点，但组织没有变聪明。你得到的只是一批更快的个人。

AI Native 的门槛不是个人更快，而是组织能从经验中学习，能自进化。

Loop Engineering 不是 cron job

Loop Engineering 不是 cron job

26年6月初，Loop Engineering 作为第四代AI工程方法学在OpenClaw 创始人和Claude Code 负责人 Boris共同推广下快速出圈，成为继提示工程，上下文工程，驾驭工程之后的第四代AI工程方法论，Loop Engineering，就是用你设计的系统，去替代你本人提示 Agent。你构建一个小型系统，让它自己去发现任务、分配任务、检查任务、记录完成情况、决定下一步，然后让这个系统去触发智能体。

你不应该再手动提示编码 Agent 了。你应该设计 loop 来提示你的 Agent。

过去两年，你从编码智能体那里"拿到东西"的方式，是写一个好的提示词，然后提供足够的上下文。你输入一段话，读它返回的内容，再输入下一段话。智能体是一个工具，你全程控制着它，一轮接一轮。这个范式叫 Prompt Engineering——你优化的是单次交互的输入质量。

但现代软件工作很少能通过单次 Prompt 解决。真实项目包含隐藏约束、脆弱测试、遗留约定、部署规则、产品权衡和不完整需求。

Loop Engineering 彻底改变了这个模式。它不是在优化你的提示词，而是让你退出提示的位置。你不再是那个守在聊天框前不断输入指令的人——你是那个设计自动化循环结构的人。

一个完整的 loop 包含五个阶段：Intent（定义目标结果）→ Context（收集代码、文档、错误、约束）→ Action（编辑文件、运行命令、调用工具）→ Observation（捕获测试结果、编译错误、运行时输出）→ Adjustment（根据观察更新计划，重复循环）。

到这里，有人会说："这不就是 cron job 换了个名字吗？"

对了一半。调度层确实是 cron——Boris 的 loop 就运行在 cron 上，Claude Code 的 /loop 命令底层也是调度。但中间的执行体完全不同。Cron 运行的是一个固定脚本，每次执行相同的分支逻辑。Agent loop 运行的是一个能读取当前状态、决定下一步行动、自我验证并修正的模型。

这就是最诚实的框架：loop = cron + 身体里的决策者。

要理解 Loop Engineering 的工程深度，看问题链——每一步演化都是在解决前一步留下的致命缺陷。

第一个问题：模型怎么知道自己做完了？2022 年的 ReAct 论文给出了最原始的答案——模型推理、调用工具、读取结果、重复。但它是人在判断"做完了"，模型只管执行不管验收。

第二个问题：人能退出这个循环吗？2023 年的 AutoGPT 试图回答——给 Agent 一个目标，让它自己提示自己。但它创造了一个新问题：Agent 永远觉得自己"还差一点"，无限空转。AutoGPT 因这个缺陷臭名昭著，给整个 Agent 领域蒙上了"玩具"的阴影。

第三个问题：怎么让 Agent 停下来并且停得对？2025 年 7 月，Geoffrey Huntley 发布了一个看起来极其愚蠢的方案——Ralph Loop，就是 while :; do cat PROMPT.md | claude ; done。核心创新不是技术，是纪律：每次迭代重置上下文到固定锚文件（不依赖对话历史），进度存在磁盘而非对话中（不是"我记得我做了"而是"文件里写着已完成"），每次只做一个离散工作单元（做完就退出，不给自己加戏）。

第四个问题：怎么让验收标准本身是客观的？2026 年春，Claude Code 和 Codex 同时内置了 /goal 命令——Agent 持续工作直到可验证条件达成，由独立小模型评估 yes/no。做工作的模型不是判断工作完成的模型。这是 Maker-Checker 分离在停止条件上的第一次产品化实现。

第五个问题——也是 2026 年 6 月那场争论的真正焦点：怎么让多个 Agent 在同一个目标下协作而不互相踩踏？编排循环（Orchestration Loop）就是答案：多 Agent 监督、cron 调度、git 备份状态、崩溃恢复。Ralph 是单 Agent 自迭代。编排循环是多 Agent 的监督系统。后者才是 Steinberger 和 Boris 说的真正的东西。

以前的杠杆在 Prompt Engineering，写好提示词就能卷死别人。现在的杠杆在 Loop Engineering——你设计的系统架构和规范质量，直接决定了产出的上限。

但一个 loop 到底需要什么才能跑起来？

六大构建块——Loop 的工程骨架

一个 loop 需要五样东西，外加一个记忆的地方。听起来非常简单，但这是所有长时间运行的 Agent 都依赖的同一个技能。

第一块：Automations（自动化调度）——循环的心跳。 没有调度，loop 就是一次手动执行，而不是真正的循环。

还有一个特殊的调度原语：/goal。它不是按时间触发，而是按条件驱动——你定义完成条件，Agent 持续工作直到满足。每轮结束后，一个独立的 Haiku 模型评估 yes/no——做工作的模型不是判断工作完成的模型。这就是 Maker-Checker 分离的第一道防线。没有调度层的 loop 等价于"你手动执行了一次脚本"：它产生了输出，但不是系统。

第二块：Worktrees（工作树隔离）——并行不冲突。 当你同时运行两个以上 Agent 编辑同一个仓库，文件冲突就变成了故障模式。两个 Agent 写同一个文件，和两个工程师不同步直接提交到同一行代码的后果完全相同。Git worktree 给每个 Agent 独立的工作目录和分支——一个 Agent 的编辑物理上碰不到另一个 Agent 的 checkout。Claude Code 用 --worktree 参数或子 Agent 的 isolation: worktree 配置实现。没有 worktree 的并行 loop，就是让两个实习生在同一个文件上同时改不同的需求——他们会在 git merge conflict 里相遇，而你不得不在凌晨三点当裁判。但有一个重要提醒：worktree 只解决了机械冲突，你的审查带宽才是真正的并发上限。 第三块：Skills（技能文件）——编码项目知识。 Agent 每次新会话都像第一天上班的实习生，除非你把项目知识写下来给它读。Skills 就是这些"写下来的知识"——一个 SKILL.md 文件，放入 .claude/skills/ 目录，Agent 按需加载。没有 Skill 的 Loop：每轮从零推导项目约定，反复燃烧 Token。有 Skill 的 Loop：难题沉淀为 Skill，下次直接调用。一个反例可以说明差距：两个团队都跑 CI 修复 loop。A 团队的 loop 有 15 个 Skills 文件，涵盖 API 约定、错误处理模式、部署流程。B 团队的 loop 没有 Skills——Agent 每次修复 CI 失败都要重新猜测"这个项目的测试命令是 npm test 还是 pnpm test"。三个月后，A 团队的 loop 修复准确率 92%，B 团队 61%。差距不在模型，在 Skills。 第四块：Connectors（连接器）——接入真实工具。 一个只能看到文件系统的 loop 是一个 tiny loop。Connectors 基于 MCP（Model Context Protocol），让 loop 能读 issue tracker、查数据库、调 staging API、发 Slack 消息。没有 Connectors 的后果是"信息断裂"——loop 修了一个 bug，但没有更新 Linear 工单，没有通知 Slack 频道，没有在 PR 里 link 原始 issue。三天后，另一个工程师在 Linear 上看到同一个工单，以为还没人处理，又修了一遍。Connectors 让 loop 从"文件系统里的孤岛"变成"工作流里的节点"。 第五块：Sub-agents（子 Agent）——Maker-Checker 分离。 Loop Engineering 里最有用的结构设计，没有之一：写代码的 Agent 和检查代码的 Agent 必须分离。模型对自己的产出天生宽容——让它评审自己的代码，它会自信地称赞平庸的工作。解决方案是让一个独立 Agent 做审查：不同的指令、不同的模型、独立的 worktree。审查 Agent 的输出格式必须是结构化的——VERDICT / ISSUES / EVIDENCE——不是"看起来不错"，是可以被追溯的判断。一个反事实可以帮助理解它的价值：如果你把 Maker 和 Checker 合并为同一个 Agent，它每轮的输出会越来越自信但越来越偏离真实需求——因为没有任何外部信号纠正它的方向。这不是 Agent 的错，是你没给它一面镜子。 第六块：Memory（持久记忆）——状态存活于会话外。 模型在每次运行之间会忘掉一切。记忆必须存在磁盘上，而不是在上下文里。智能体会忘，但代码仓库不会。实践中最有效的模式是维护一个 LOOP-STATE.md 文件，用三段式结构：Open（待处理）、Done（已完成）、Blocked（需人工决策）。每次 loop 运行的第一条指令是"读取 LOOP-STATE.md 了解当前状态"，每完成一个工作单元后立即写回。没有 Memory 的 loop 会在每次重启后把昨天修过的 bug 再修一遍——因为它不知道已经修过了。这不是花哨的技术——一个 Markdown 文件就能把整个系统串起来。

需要澄清一点：Memory 不是"最好有"的设计偏好，它是"没有就不行"的工程约束。AI Agent 每次会话结束即丢失全部上下文，不存在任何会话内替代方案。跨 session 的状态必须落在磁盘上——GOAL.md、PROGRESS.md、LOOP-STATE.md 不是组织技巧，是物理必需。

Loop Engineering 的起点不是自动化，是记忆。没有记忆的循环，只是昂贵的重复。

验证闭环——谁来说"不对"

你搭好了六大构建块，loop 开始跑了。凌晨 3 点，它自动扫描 CI 失败日志，开 worktree，修 bug，跑测试，提 PR。早上你醒来，Slack 里躺着一条消息：修复了 3 个 lint 问题，提了 2 个 PR，所有测试通过。

但你有没有想过：它说"通过了"，谁来判断这个"通过"是真的？

一个无人值守运行的 loop，同时也是一个无人值守犯错的 loop。

Loop 不是魔法。它里面的反馈才是。一个只有前馈没有反馈的系统在控制论中被称为开环——其行为完全不可控、不可预测、不可收敛。写代码的 loop 如果只是"写→写→写"而没有"验证→修正→验证"，就是一台自信地犯错的机器。

Maker-Checker 分离解决了"谁来检查"的问题。在 Claude Code 中，/goal 命令的底层实现正是这个原则：每轮结束后，完成条件加上对话历史被发送给一个独立的 Haiku 模型——不是那个写了代码的模型。Haiku 返回 yes/no 加简短理由。No 意味着条件未达成，理由作为反馈传给 Claude，开始下一轮。Yes 意味着条件达成，目标清除。

但你还需要"检查什么"的标准。三个硬性停止条件，是任何生产级 loop 的必选项，不是可选项。

第一个是最大迭代次数。MAX_ITER=20。一般任务选 15-50 之间的数字。不是"希望能停"，是硬编码的退出条件。

第二个是无进展检测。定义"无进展"的判定标准——相同错误消息、空 diff、相同测试失败——设置连续阈值 N（如 3 次）。连续 N 次无进展 → loop 终止并告警。这不是优化，是熔断。

第三个是 Token 或金额预算上限。每个 loop 设预算（如 $5 或 50000 token）。睡前设的 loop，预算要更低。这不是小气，是工程纪律——一个没有天花板的 loop，就是一张没有额度上限的信用卡。

这三个停止条件之所以必须是"硬性"的，根因在于 /goal 的一个已知缺陷：模糊性会复利增长。Loop 以分钟为单位迭代——每轮输出变成下一轮输入，exit condition 写得模糊，方向偏移就会在几轮内复合放大到一个完全不想要的方向。传统 Scrum Sprint 设两周，正因为人类需要这个时间窗口消化反馈、纠正方向。Loop 把这个窗口压缩了三个数量级，模糊性的破坏力被同等放大。停止条件的"硬"不是谨慎——是速度的数学必然。 [来源：716-Top 3 Loops 4，笔记390：/goal已知缺陷一——模糊性复利增长]

这个数量级的差异——Sprint 以周计，Loop 以分钟计——不是量的变化，是质的变化。它意味着敏捷中所有关于"模糊性管理"的经验（acceptance criteria、Definition of Done、Sprint Planning 需求澄清），在 Loop Engineering 中不再是"好的实践"而是"必须的工程约束"。硬件领域的敏捷实践对此有最硬的跨域验证：自然语言的 Spec 是流片失败和项目延期的 top 2 根因，经过十年教训才发展出机器可执行的接口契约来消除模糊性。Loop Engineering 面对的是同一个问题——只是对象从人类工程师变成了 AI Agent，速度从周变成了分钟。 [来源：716-Top 3 Loops 4，敏捷IC礼物三：机器可执行契约 + Sprint-Loop反馈环粒度差异]

但还有一个更隐蔽的问题：loop 跑偏了怎么办？

Loop 有五种已知的失败模式。抖动（Thrashing）：Agent 反复修改代码但不收敛——改 A 文件，lint 没过，改 B 文件，还是没过，回去改 A。解决方案是缩小目标范围，减少每次 diff 的大小，使用更精确的验证信号。测试过拟合：测试全绿但产品需求未满足——Agent 学会了让测试通过而非满足需求。解决方案是增加需求层验证——"对照原始需求，逐一确认每个验收条件"。上下文漂移：Agent 基于第 3 轮的假设继续工作到第 14 轮，但第 7 轮另一个子 Agent 改了那个假设。解决方案是在关键观察后强制刷新——让 Agent 输出"当前已知事实摘要"。Acknowledgement Loops：Agent 每轮都承认进展但实际上没有推进——Claude 说"我理解问题了"但没改任何东西。解决方案是写更紧的可验证结束状态，设 hard turn cap。最后一种，也是最危险的一种：认知投降。因为循环自动化而停止理解代码——你开始直接点 merge 而不再读 diff。

这就是渐进式信任的意义。不要一开始就让 loop 自主合并代码。第 1 周：loop 只分类和记录，所有操作需人工确认。第 2-3 周：loop 自动创建 PR，但不自动合并——你去读 diff，你决定 merge。第 4 周起：对已验证可靠的低风险操作允许自动合并。每个阶段不达标就停留，不退让。

你必须阅读合并的代码，不要只看检查是否通过。loop 替不了你的判断。它只替得了你已经知道如何判断的重复性工作。

Loop Engineering 有三件绝对不要做的事。第一，不要在你还不理解的代码库上跑自主 loop——你连"什么是对的"都不知道，怎么判断它做对了？第二，不要在 loop 运行时修改它在读的规范文件（VISION.md、CLAUDE.md）——你相当于在空中给飞机换引擎。第三，不要用同一个模型既做执行又做签核——Maker 和 Checker 的分离不是建议，是底线。

第五幕：四指令协作流——让系统学会下一次

到现在为止，我们讲的都是"怎么让 loop 跑起来"。但循环工程的分水岭不是"能不能让 AI 自己跑"，而是跑完以后，系统有没有记住成功、理解失败、自主进化。

跑完就忘的 loop，是一个永远在第一天上班的熟练工。

在深入机制之前，有一个认知可以降低你理解这套系统的成本：四指令协作流不是 R2 凭空发明的方法论。它是敏捷在 Agent 工作流中的工程化表达。Sprint 的 Planning 定义目标 → 对应 /goal 的条件声明。Sprint 的 Review 检查产出 → 对应 Between iterations run 的命令输出检查。Sprint 的 Retrospective 改进流程 → 对应 r2-mevolve 的进化信号回灌。敏捷 20 年积累的实践智慧——Sprint Planning、Daily Standup、Sprint Review、Retrospective、Backlog Refinement——可以直接翻译为 Loop Engineering 的设计模式。你不必从头学一套新方法论，你在理解的是 Scrum for Agents。 [来源：716-Top 3 Loops 4，Sprint-Loop同构映射核心启示]

R2 对循环工程的落点，不是再写一套漂亮的 prompt，而是把学习本身工程化。怎么做到？四个指令的协作流。

指令 1：r2-mdistill（蒸馏）。 从人工经验中提取隐性知识，蒸馏为 Agent 可执行的 Skill。五步：捕获——每次你心里说"这里不对"，停下来问自己"我是怎么知道这里不对的"，那个判断依据就是隐性知识。分类——区分表征（命名约定）、概念（协议约束）、命题（条件策略）。压缩——1000 个失败案例浓缩成一份边界条件清单，信息少了，解决能力强了。参数化——硬编码变参数，同一个 Skill 传入不同 DUT 变成不同能力。最后用 9 条判据自检：I/O 定义明确、三层知识结构逐层递进、压缩后 token 更少能力更强、每个关键概念至少一个具体例子、只注入模型不知道的残差、有可执行的验收标准、参数化非硬编码、计算交给代码判断留给模型、上线前用 3-10 个样本验证。

一句话判据：删掉这个 Skill，模型输出没有变差 = 废物。加上这个 Skill，某些任务变差 = 有毒。

蒸馏有三个反模式需要识别。反模式一：把 1000 次聊天的原始日志全塞进"记忆"——这不叫蒸馏，叫归档。归档不能被执行，只能被搜索。蒸馏的前提是压缩：从 1000 个案例中提取边界条件清单。反模式二：每次遇到失败就蒸馏一条规则——过拟合。只有重复出现的失败模式才有蒸馏价值。一次的异常是 noise，三次的规律是 signal。反模式三：蒸馏出的 Skill 从不验证就直接上线——等于把未经测试的代码推到生产环境。费曼检验不是可选项。

蒸馏的动力来源是什么？来自一个被低估的动力学原理：失败模式的复利效应。在传统的软件工程中，试错是沉没成本——修完就扔，经验留在工程师脑子里。但在有记忆的 loop 中，每次失败被写入 guardrails.md 或 reflexion.md，下一次迭代读取后主动避让。失败了不再是代价，是写入系统记忆的生产性资产。系统不是随时间衰减，而是随时间变强——试错不再是你在给系统交学费，是系统在给自己长骨头。 [来源：716-Top 3 Loops 2，失败模式复利效应]

指令 2：r2-mretention（记忆）。 决定 Agent 跨会话该记住什么。四层记忆架构：草稿纸（过程性记忆，用完即弃）→ 短期记忆（结论性记忆，格式 Schema 化）→ 长期记忆（经验性记忆，可泛化的行为模型）→ 模型先验（共识天花板，不可写入）。

记忆的边界很硬。三个规则：只写结论性知识，不写过程性细节——记忆不是聊天记录归档。用结构化 Schema 填表，不用自由格式摘要——结构化的记忆才能被查询和复用。原始事件不可改，只追加；修正走新版本——记忆的因果链不能断。

指令 3：r2-mverify（验证）。 Skill 上线前的质量门禁。不是跑代码测试，是验证知识本身是否正确、边界是否完备。费曼检验：拿 3-10 个样本让 Agent 跑，看 Agent 的输出是否符合你的专业判断。不符合 = 不是 Agent 笨，是 Skill 没写好 → 回到蒸馏。 指令 4：r2-mevolve（进化）。 Skill 上线后的持续优化。读执行 trace 的反思式进化搜索，对比新旧 Skill 版本在相同任务上的输出差异，回答一个核心问题：新版比旧版好吗？

进化信号的来源有三类。负反馈——出错模式回灌为预防规则。正信号——某类任务效率提升，该 Skill 版本保留。Harness 退化检测——任务完成率下降、token/有效产出下降 → 触发 Skill 回滚。

这里有一个被严重低估的数据：给 AI 看完整的原始执行轨迹（Trace）比只看 300 字摘要诊断更准，效果好 44%。每一轮都能揪出至少一个之前没看见的问题。比人类极难察觉的边界条件——比如"迭代-10 排序导致 cleanup 误删最新结果"——在完整 trace 面前无所遁形。

四指令协作流构成一个增强飞轮：人工经验和失败案例进入 r2-mdistill，蒸馏后的 Skill 经 r2-mretention 写入记忆，Agent 用 Skill 执行任务，r2-mverify 在上线前把关，r2-mevolve 在线持续优化，优化产生的负反馈和正信号回灌到蒸馏环节——新一轮的 Skill 比上一轮更强。

Harness 设计水平 ↑ → AI 可控性 ↑ → 产出质量 ↑ → 度量可信 ↑ → 投入信心 ↑。这个飞轮一旦转起来，系统就不只是在完成一次任务——它在让下一次更好。

但飞轮也有惯性。如果放任不管，你对代码库的理解会腐烂。loop 跑得越快、产出越多，你没亲手写过的代码就越堆越多，实际存在的东西和你真正理解的东西之间的差距就越大。进化不是认知外包。

所以，经验都可以被蒸馏，但信任只能落在人身上。AI 可以执行，机器必须裁判，人要保留签核。

第六幕：组织 OS——从个人提效到自进化

你有了一个能从经验中学习的 loop。但如果只有你一个人在跑这个 loop，你的组织还是 1.0。

一次好的排查不再只是某个人和某个 Agent 的私有经历——它开始教会后面的会话。

这就是组织 OS 想要解决的问题：智能时代的能力协同。组织 OS 不是一个软件产品，是一套多主体协作协议——人和 AI 怎么互动，AI 和 AI 之间怎么互动。组织由无数个可被随时发现、调用、连接和激发的能力节点构成——这些节点可能源自人类专家，也可能源自智能体。

在讨论组织 OS 的实现之前，让我们先把"Loop Engineering 与敏捷IC的关系"看清楚——因为这张对照表，是你向团队解释"为什么这不是又一个工具，而是下一代工作方式"的核心论据。 Loop 与 Sprint 在七个维度上存在精确的结构同构：

(1) 心跳——固定时间盒（2周）↔ /loop 间隔或动态自调节。

(2) 交付物——Potentially Shippable Increment ↔ 每次迭代离 exit condition 更近一步。

(3) 检查点——Sprint Review（看产出）↔ Between iterations run: npm test（看命令输出）。(4) 回顾——Retrospective（改进流程）↔ evaluator 的 reason 注入下一轮（改进方向）。

(5) Backlog——Product Backlog→Sprint Backlog ↔ Goal→每轮拆解的具体 step。

(6) Definition of Done——团队约定的完成标准 ↔ Exit condition。

(7) Scrum Master——移除障碍、保护节奏 ↔ 系统自动调度、worktree 隔离、hook 门禁。 Loop Engineering 没有发明 cron、worktree、hook——它把它们拼成了一个 Agent 能自己跑的系统。跟 Scrum 没有发明 standup、retrospective、backlog 但把它们拼成节奏，完全一样。

要实现组织 OS，有四件事必须做：所有知识都要数据化——不能留在某个人脑子里。所有讨论都要结构化——自由文本不可挖掘，结构化 Schema 才能被 loop 读取。所有实验都要可追踪化——trace 比摘要更有价值，好 44% 的数据我们重复一遍。所有反馈都要输入到系统里头——正反馈和负反馈都要回到记忆和蒸馏管道。

R2-AIOS 的架构用四个要素描述：Harness = 怎么跑（确定性编排，任务切分和接口设计）。Oracle = 机械裁判（硬标准不可协商，DRC clean + timing WNS ≥ 0）。Memory = 跨会话持久（因果链压缩，把离散事件压缩为可泛化的行为模型）。Loop = 跑到条件满足才停（没有机械 Oracle 的循环是幻觉加速器）。

这四个要素构成的是组织的自进化机制，不是个人的提效工具。

回到最实际的问题：你明天早上到公司，能对 loop 做什么？这里有一张 7 天循环工程卡：

本周哪次执行产生了 trace？不是问"跑了没有"——是问"有记录吗"。

哪条结论应该进 memory？不是所有结论都值得进长期记忆——只有结论性的、可泛化的行为模型。

哪个失败模式应该蒸馏成 Skill？一次失败不叫模式，反复出现的失败模式才有蒸馏价值。

哪个 Skill 需要 verify？上周写的 Skill，这周跑了 10 个样本了吗。

哪个版本需要 evolve 或 rollback？新版本真的比旧版本好吗——不是你感觉，是数据对比。

最后，哪个决定必须人签核？不是所有决定都要签，但关键决策——修改共享接口、删除"看起来没用的"函数、放宽停止条件——放手就是负债。

6 个问题，每周问一次。但组织级的进化不只靠一张周检查清单。驱动它的飞轮逻辑是：短迭代产生高频复盘，高频复盘让成功经验和失败模式被快速捕获，捕获后的经验通过 Memory 模块（GOAL.md/PROGRESS.md）和 Skills 模块（SKILL.md）资产化，资产化后的知识被 Agent 在每次执行中调用和验证，而每次人工修正又回流到蒸馏管道——人走了，经验留在系统里。传统组织中"老员工离职=经验蒸发"的困境，在这个飞轮中被结构性地消解了。这不是理想主义——敏捷在硬件领域的实践已经证明了同样的逻辑：基于可工作特性持续交付的短迭代，让成功模式被固化、失败模式被复盘，而智能体的记忆机制让这一模式内化为系统的持续进化能力。

1990 年代，软件工程的核心问题是"怎么写"——编程语言、设计模式、重构。2010 年代，核心问题变成了"怎么协作"——git、CI/CD、code review、敏捷。2026 年，核心问题再一次迁移——"怎么让组织从 AI 的工作中学习"。

这个迁移不是断裂。它是继承。Loop Engineering 的声明式配置——用 /loop 15m 替代手写 cron bash 脚本，用 .claude/settings.json 替代手动安装 git hooks——是对 DevOps 二十年声明化运动的结构性延续。Jenkins Pipeline as Code、GitHub Actions YAML、Terraform HCL 的共同方向始终是：让操作者描述"想要什么状态"而不是"执行什么步骤"。Loop Engineering 把同样的范式带入了 AI Agent 的工作流编排——从"写脚本管理 Agent"变成"声明你想要它达到的状态"。

写一次好代码是个人能力。让那次代码背后的判断、排查路径、失败模式被整个组织继承——那是系统能力。

两个人可以搭出完全一样的 loop，却得到截然相反的结果。一个人用它在自己深刻理解的工作上跑得更快；另一个人用它来逃避理解工作本身。Loop 不知道这两者的区别。你知道。

Build the loop. Stay the engineer.

设计循环。保持监督。渐进信任。

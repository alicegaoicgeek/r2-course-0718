团队经验：是进化的 Skill 系统，还是僵尸 md 文档？

Skill 的本质是分类学

论文里有一组被反复引用的数：Skills 数量在 20 个以下时，路由准确率大约还能撑在 90% 以上；过了 30 个就开始明显掉；堆到大约 200 个，准确率只剩 20% 左右，而且又慢、又烧 token。

不是模型突然变笨了。是你把分类器塞爆了。

每个 skill 在会话启动时，并不是整本说明书先塞进上下文。系统先扫的是一串极短的元数据——大致是 name + description，量级大约 250 字节一个。命中了，才去读完整的 SKILL.md；再往深处，才是 references/、scripts/ 那些分支材料。路由发生在最薄的那一层。你越往顶层塞「种」级差异，越是在用同一张窄表，硬分出上百个几乎撞车的标签。

所以现场长这样：库里 skill 越装越多，触发却越来越飘——该上的不上，不该上的抢戏，最后你手动 @ 指定，等于自己做了分类器。

问题从一开始就不在「正文写得够不够厚」。

朋友圈里常见一种狂热：万物都可以蒸馏，万物都可以封装成 skill。有人库里六七十个，离谱的破百。每多一个场景就新开一个文件夹，封面图一个、小红书一个、PPT 配图一个，美其名曰「职责单一」。

这是收藏家路径。收藏家相信数量等于能力：装得越多，Agent 越能干。反事实立刻拆穿它——若数量本身是资产，200 个 skill 该逼近全知，而不是准确率塌到两成。真正塌掉的是选择空间的熵：候选标签互相抢 description 语义，模型在局部视图里做模式匹配，编码空间越脏，解码越容易错。

Claude Code 一侧有个对照直觉：工具是 Agent 的感官，感官太乱，它反而看不清世界；官方体量上工具大约稳住在二十个量级，不是能力不够，是克制。Skill 库若反其道而行，用「种」的粒度铺满顶层路由表，等于把工具集那种熵爆炸，换皮重演一遍。

还有一层更隐蔽的错觉：把 Skill 当成软件组件。程序有类型、可编译、可静态分析；Skill 首先是文本片段——不能自修改成真正的程序语义，也不能靠接口契约做确定性组合。冯·诺依曼意义上的「程序与数据同储、可自修改」在这里不成立：它是注入上下文的说明，不是可链接的计算原语。它和 Agent 之间往往只有自然语言这一层最弱接口：你以为在下控制指令，其实是在对概率模型做一次希望式映射。写出来的那一刻，文本就定死了；API 变、需求变、模型变，文件自己不会醒。

单次会话里它能跑通，所以看起来像资产。跨项目、跨模型、跨周期再看，它常常只是便利贴：贴上去能干活，干完就散；换个项目要重贴；没有编排层，就没有可累积的结构。便利贴化不是因为它「能力弱」，而是一组工程缺陷叠在一起：可拆卸却无结构耦合、临时却无生命周期、接口是自然语言、缺少组织化编排、执行信号不回流。原厂智能体像裸机，skill 像预装应用——能让你在业务场景跑起来，不代表架构原生正确；预装能跑，不等于手机架构对了。

厂商把 Skill 当生态钩子推，是另一条叙事。技术上它未必是最优解，却是当前条件下好用的锁定抓手——行为习惯与工作流沉在某家生态里，切换成本就起来了。那是供给端动机，解释不了你库里的路由灾难。你的问题更土、也更硬：分得对不对，触不触发得准。

一个好的 skill，核心就两个词：分类和触发。

怎么触发、能不能正确触发、触发以后能干什么——这三件事排完，正文厚度才排得上号。Claude 一度更新 Skills 生成器，最要紧的动作之一，就是用反馈去磨触发条件；不是先把说明书写成小说。

分类学从来不鼓吹「分得越细越好」。生物学一层层穿：界、门、纲、目、科、属、种。你把前面全抹掉，只给人一堆「种」名，世界会乱成灾难。Skill 顶层该停在「科」或「属」，不是「种」。

封面图和 PPT 配图之间的差异，不值得各占一个独立的顶层 skill——它们是「图片生成」类别内部的变异。图片生成和服务器管理之间的差异，才大到必须分家。操作上很具体：相关场景合成一个顶层 skill，内部写清主流分支；Agent 触发之后，再根据上下文做二次分析，走内部场景。一个 skill 同时覆盖细分需求与通用生图，验证的就是颗粒度有没有停对层。

判断「值不值得新建」可以压成三条，全过才开新文件：

  • 场景有没有明确边界；
  • 会不会高频复现；
  • 能不能归进已有 skill。

外加奥卡姆：如无必要，勿增实体——用不到的别装。常年把常用集压在大约 30 个以内，不是洁癖，是路由表的卫生学。更上游还有一刀：哪些该写进 CLAUDE.md 这种长期背景，哪些才配当可触发 skill。分类系统先于 skill 列表；没有分类系统，列表只是杂货铺货架。

反事实再钉一次：若「种级拆分」永远正确，合并生图分支应导致场景丢失；实践里合并后由上下文二次路由，反而同时保住细分与通用。失败的不是合并，是把变异抬成顶层物种。

路由不是玄学，是分层加载。

用户意图进来之后，实际发生的是一条很短的决策链：先扫所有 skill 的 name + description（分类/触发），命中后才加载 SKILL.md 全文（按需展开），再在正文或 references/、scripts/ 里做分支决策（内部分类）。三层不是文风偏好，是上下文经济学：Level 1 永远预载，量级大约 250 字节一个；10 篇知识库和 10000 篇知识库，对这一层难度几乎一样——Agent 始终只拿元数据匹配。Level 2 才是完整工作流，篇幅建议压住，别把百科全书塞进每次命中。Level 3 才是重资源，走到分支再读。

这就是渐进式披露：细节在需要时出现，不在开场抢上下文。Skill 的价值常常不在正文有多厚，而在被正确选中；选中本身就是分类问题。44 号原典讲哲学（为什么要分、分到哪一层），决策树文讲结构（命中后怎么写），Anthropic 公开用法讲工程（description 怎么写才命中）——三层合读，才是完整的分类学，而不是「多写两段 prompt」。

因此 description 不是给人看的摘要，是写给模型的触发条件说明。会话启动时，模型扫技能清单；清单上的句子决定「这个请求有没有对应技能」。触发词要写进描述——像 babysit 这类动作词，写进去是为了命中，不是为了商店好看。你改三天正文却不改 description，等于修了说明书目录外的章节，前台检索词仍是错的。

反过来说：若两个 skill 的 description 在模型眼里同义，你在文件系统里分得多干净都没用——Level 1 已经把它们糊成一个模糊类。分类学的第一现场是 description 空间，不是文件夹缩进。

正文结构要跟着任务形状变。路径唯一、步骤固定——线性流程够用：先 A 再 B 再 C，部署类技能常常长这样。但「帮我选一个云平台来部署」不是部署，是选择：不同预算、不同技术栈、不同约束，岔路从第一句就分叉。这时该给的是决策地图，不是写死的一条路。

决策树技能通常长这样：开头 3～4 个明确入口，别让模型猜「我该先问什么」——例如科学问题选择类技能会先分「提新项目创意 / 拆现有项目问题 / 问战略问题」；然后适应性提问，每步只问一个维度，给编号选项，用上一答决定下一问。产品优先级顾问类技能最多大约四个追问：阶段、决策风格、利益相关者、数据量级，逐步收敛到 RICE、ICE、价值/难度等框架，而不是一上来把所有框架说明书拍脸上。分支用 IF-THEN、表格或编号层级写清楚；复杂细节甩到 references/，走到再加载；必须有兜底——都不匹配时走哪条、何时停问、如何回退。

树的终点常常不是替用户做完决定，而是缩小选择范围，让模型带着用户信息走到可辩护的推荐。这和顶层分类学同构：外层减少候选，内层再精细——永远不要在第一层上同时做完所有细分。

几个会把树写烂的坑，值得钉在屏幕上：一个 SKILL.md 里别嵌多棵决策树，一件事做对就够；分支宜控在大约 5～7 条，过密则清晰度与触发一并垮；没有兜底的树，是假装完备的迷宫；可复用 SOP 才是 skill 主场，复杂并行该交给 Subagents，外部连接用 MCP，长期背景用 Projects——别让决策树吞掉全世界。

判断公式可以很短：能用「Step1→2→3」说清的，走线性；需要「如果 X 走 A，如果 Y 走 B」的，走决策树 + 按需加载。复杂技能常见组合是：先用树决定加载哪个子能力，再用线性流程执行。决策树质量决定走对路的概率；按需加载决定选项变多时上下文是否还扛得住。你给模型的，究竟是一张干净利落的决策地图，还是一团浆糊般的条件判断——这句拷问对顶层 taxonomy 同样成立。

写 skill 时还有几条与分类同向的纪律。别陈述模型早就知道的常识，占上下文却无增量；前端审美类技能有价值，正因为它专治「Inter 字体 + 紫色渐变」这类默认思维定式，而不是复述「要写干净代码」。坑点专区往往是含金量最高的块：subscriptions 表只增不改、该取 version 最大而不是 created_at 最新；网关里叫 @request_id、计费里叫 trace_id，其实是同一个值——这些句子才改变行为分布。别把每一步钉死成铁轨，技能要被反复复用，过细的剧本会在现场碎掉。需要配置就设计初始化（例如站会发到哪个频道写入 config.json），而不是假设用户脑内有份隐式 config。文件夹不是装饰：主文件指向 stuck-jobs.md、api.md、模板文件，Claude 在特定状态下自己去读——这是用文件系统做二级分类，而不是用更长段落冒充分类。

衡量也服务分类：有人用 PreToolUse 钩子记 skill 调用日志，找出「超受欢迎」和「触发率低于预期」的项。触发率低，优先怀疑 description 与真实话术不对齐，其次才怀疑正文。分发上，仓库直提交适合小团队；技能一多，每人强制全量安装等于强制膨胀 Level 1 清单——插件市场让成员自选安装，本身就是组织层的路由卫生。

若分类与触发是微观机制，组织方式就是宏观 IP。

Anthropic 把内部技能盘完后，收成大约九类：库与 API 参考、产品验证、数据获取与分析、业务流程与团队自动化、代码脚手架与模板、代码质量与审查、CI/CD 与部署、运维手册、基础设施运维。最好用的技能职责单一，稳稳落在一类；想一招鲜横跨多类的，反而把智能体整懵。这不是宇宙标准答案，却是可用的审计框架：拿它照自己的库，立刻能看见空白带和污染带。

每一类回答的是不同的「何时触发」。库与 API 参考：私有库边界、自伤型操作、内部 CLI 子命令。产品验证：无头浏览器跑通注册到引导、每步断言——对输出质量的拉动，在他们内部被描述为九类里最立竿见影的一档，专门打磨一周被认为值回票价。数据类：漏斗该 JOIN 哪些事件、user_id 在哪张表。自动化类：站会只展示增量、工单字段强制校验。脚手架、审查、部署、runbook、基础设施运维——各自有症状词和工具链。taxonomy 在这里不只是贴标签，还是投资优先级：你把热情全砸在又一个脚手架模板上，可能正好错过验证层这块杠杆。

R2 侧有另一张「科/目」级地图，大约十一类 skill hub：r2-productsmart（需求/竞品/度量等）、r2-specx（规格抽取与 diff）、r2-designsmart（切片、C2RTL、顶层生成等）、r2-verismart（vplan、覆盖率、复用等）、r2-metrics、r2-eda、r2-ip、r2-debug、r2-pr、r2-ams、r2-learn。顶层是领域能力族，下面再挂可组合的具体技能——不是「某次会议纪要生成器」这种种级碎片。不能一个场景搞一套 skill，就像不能一个场景单独开一颗不可复用 IP；知识经验若要能卖、能复用，组织方式必须先像 IP 库，而不是像桌面便利贴墙。

两张地图并读，会看出同一纪律：官方九类是通用工程「目」；R2 十一类是芯片研发「目」。目可以不同，不要把种抬到目这条纪律相同。你的 hub 缺的往往不是第一百零一个文件，而是某一目整层空白，或某一目内部种级拥挤到路由互咬。

这里的 IP 一词，不要理解成法务意义上的著作权登记。芯片语境更贴：GitHub 上能跑的开源核很多，AI 也能在很短周期里吐出能仿真的设计；Arm、SiFive 仍然能卖得很贵，贵在架构、验证、交付、工艺适配被压进可授权模块——能跑和能扛不是同一个词。Skill 同构：资产价值不在那段静态文本本身，而在它是否把「谁在什么情况下怎么做怎么判断」压成可复制的决策单元，并且能被系统编排。

一句话分界：Skill 是决策单元，Harness 是决策架构。

单元可以很多；没有架构，单元只是散落的便利贴。同一个 skill 文件，在 Harness 治理下可以是 IP，在个人随手工具场景里仍是便利贴——IP 属性不是文件内生的，是系统赋予的。赋予至少要三条线同时在场（这里只取组织含义，不展开成进化专论）：

结构约束。 Harness 提供编排协议：协议层定义接口契约，编排层调度组合，运行层管生命周期。场景变、工具升级、模型更新、skill 增删，都该触发「行为是否仍满足验收」的再确认——类似芯片改约束后的 re-verification，重点是偏差度量与修正能力，不是再写一篇更长的散文。 反馈回路。 手工 skill 常陷入信度分配失败：二百行里哪四行真正起作用说不清。被重复触发且经人修正的规则，应写回、升版本、留修正日志——人的修正是归因的 ground truth，把单次成功变成可累积的决策路径。没有写回，你只有会话级便利贴。 组织背书。 生命周期靠流程而不是英雄个人：DRI 维护、标注回库、版本语义，让资深判断从隐性变显性。没有中心化纪律时，自下而上的热情会碎片化，知识停在部落口径，adoption 平台期很快到来。

三线俱在，才有资格把 skill 叫作可治理资产；缺一以上，演示「能跑」可以，自称工业级 IP 是语言通胀。人的最终杠杆，往往不在多写几份 SKILL.md，而在设计和演化 Harness 的判断力：哪些能力提升层、哪些降级、哪些合并、哪些永远不进顶层路由表。Skill 仓库不是护城河；分类纪律和编排判断才是。细胞生命周期短，基因穿越周期——单元会换，架构判断留下。

还有一层和分类直接相关的组织选择：中台对人对不动，对 Agent 反而合适——大数据小数据、大 IP 小 CBB、脚本与 skill hub，作为可动态调用的组件池；前端智能体按需取用、不好用再换。前提仍是池子有干净的类目与契约，否则中台只是更大的杂货铺。

进化、自训练、SkillOpt 那一套，是另一道题：讨论的是单元如何在时间里变强。本题只钉住更先验的一层——还没资格谈进化之前，你的分类器是不是已经崩了。分类错误的进化，只是把噪声训得更稳。

把上述机制收成一张可执行的审计顺序，避免又落回「再写一百个」：

  • 先画目，再数种。 用九类或十一类（或你域内等价的科/目表）给现有 skill 贴标签。贴不上的，不是「独特创新」，多半是跨类杂糅或根本不该是 skill。某一目全空，优先补类，而不是在拥挤目里继续拆种。
  • 只读 Level 1。 把所有 name + description 导出成一张表，不看正文。做两件事：近义合并候选（description 撞车的）；触发词补全（用户真实会说的动词/对象不在句里的）。这一步专门修路由表，禁止顺手去润色 Level 2。
  • 再剖 Level 2/3。 命中稳定的 skill，才打开正文：线性还是决策树？分支是否超过约 5～7？有无兜底？重细节是否已下沉 references/？坑点是不是来自真实失败而非想象？
  • 最后才问 IP。 这个 skill 有没有 DRI、有没有修正写回、有没有在编排里被调用而不是单聊粘贴——三线没有，就老实叫便利贴，把预期降下来，反而好治理。

顺序不能反：先进化正文、先堆数量、先谈「工业级资产」，都会在脏路由上空转。审计的产出应是一张合并/退役/重写 description 的短名单，不是又一份「待新建 skill」愿望清单。

便利贴化有一组结构性原因，和「你不够勤奋」无关。

静态文本对概率执行：指令定死，基座、工具、上下文在动，错位是结构的。自然语言接口：不可严格验证、不可硬约束、不可稳定复现。多 skill 同时入场时缺少结构化耦合，互相干扰难以预期——你以为在组合能力，其实在叠加不可测的上下文相互作用。经验压成规则常常是有损压缩，因果链与信度被抹平，你不知道哪四行在干活、哪一百九十六行在添乱；手工写满二百行仍可能大部分是噪声。Token 预算是硬约束：每条 skill 都是固定税，挤掉示例、工具输出和推理空间——减 token 往往优先于加技能。技能库不是越大越好，没有选择性的容量扩张只会制造噪音。

组织层还有对称的坑：生产者与消费者不在同一反馈环里，写的人领一次「能跑」，用的人承担漂移；methodology 需要组织背书才能扩散，否则热情碎片化，知识停在部落口径。n 个 skill 在集成视角下近似 n(n-1)/2 量级的隐式接口压力——你不是在建库，是在裸奔接口。高质量 skill 稀缺、精准匹配困难、粒度管理复杂——这三句老生常谈，背后都是同一个分类学失败：分不清、触不准、没人收。

实操上最常见的五种败相：

  • 种级爆炸：顶层 skill 数跟着场景数线性涨，description 互相咬。
  • 摘要式 description：写得像应用商店文案，触发词缺失，模型扫清单时盲。
  • 正文当路由器：把本该 Level 3 的分支全塞进 Level 2，上下文未中先胖。
  • 跨类杂糅：一个 skill 既要审查又要部署又要写周报，类边界被揉碎。
  • 有清单无 taxonomy：只有文件名列表，没有科属框架，新人只能复制粘贴迷信。

预警信号很简单：你开始靠手打 @skill 续命；同样意图有时中 A 有时中 B；加 skill 后旧场景反而抖；没有人说得清「这类请求该进哪一类」。对照工具设计那条教训也成立——格式约束永远替代不了清晰的行为边界：description 语义混叠，就像把「提问」塞进「规划」工具，模型不是笨，是表征脏了。

边界也要讲清。分类学解决的是选择与组织，不自动给你领域正确性——坑点写错，路由对了也会错得干净。模型升级后，旧引导可能从有用变有害，不存在买断一套 skill 一年不改的用法；技能的引导效果绑在当前模型的概率分布上，参数空间一迁，偏置方向不再可预期。那是生命周期问题，仍要回到 Harness 与责任人，而不是再买一百个新便利贴。个人周末项目，便利贴足够美；把演示库直接称作企业数据资产，是语言通胀。

把整条链收束成可检查的定义：

Skill 是决策单元——把「谁在什么情况下怎么做怎么判断」压成可复制模块的最小知识原子；它本身不是 IP。IP 来自分类是否停在合适颗粒度、触发是否在 Level 1 就分得清，以及 Harness 是否赋予结构约束、反馈回路与组织背书。三者缺位，Skill 即便利贴。

如果只带走三句话：

  • Skill 的核心是分类和触发，不是正文厚度。
  • 顶层停在科/属，种级变异放进内部分支与 Level 3。
  • 单张 Skill.md 是便利贴；IP 在分类颗粒度、触发路由和 Harness 组织方式。

Monday Morning

明天打开你的 skills 目录，只做三件事，两小时内能做完：

第一步：数顶层。

数清楚有多少个可被路由的 skill（不是文件夹美学，是会话启动会进清单的那些）。若已经明显超过大约 30，标红，不先写新的。

第二步：合并一次种级分裂。

找出 2～3 个 description 高度相近、场景同属一类的 skill（典型：同介质不同尺寸的生成、同流程不同出口的单据）。合成一个顶层 skill：对外一个 name + 一段写清触发边界的 description；对内用分支或 references/ 保留差异。用三条新建判据验收：边界、频率、是否可归入——合完后应更能归入，而不是更碎。

第三步：重写三行 description。

挑最常用却总「偶发不中」的一个 skill，按「写给模型的触发条件」重写：动词、对象、场景边界、反例（何时不要用）。改完用同一句用户原话连续测几次，只看是否还需要手动 @。

做完这三步，你还没有「进化系统」，但你已经在修分类器。分类器不稳，后面一切优化都是在给错误标签做加固。

先分对，再谈装多少；先触发准，再谈正文美；先有组织方式，再叫 IP。
